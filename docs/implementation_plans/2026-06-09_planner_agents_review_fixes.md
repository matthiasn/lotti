## Bucket 2 — Control-loop fixes (one commit)

### 2a. Forced-tool-choice recovery never terminates after success (verified ✓, HIGH)

Chain (all confirmed in code):

- `DayAgentWorkflow._forceCaptureParseIfMissing` / `_forceDraftDayPlanIfMissing`
  (`lib/features/daily_os_next/agents/workflow/day_agent_workflow.dart`) call
  `conversationRepository.sendMessage` with `toolChoice` pinned to the mandated
  tool, a filtered single-tool list, and the **same** `DayAgentStrategy`.
- `ConversationRepository.sendMessage`
  (`lib/features/ai/conversation/conversation_repository.dart`) passes
  `toolChoice` to **every** inference call of its multi-turn loop, and
  continues whenever the strategy's `processToolCalls` returns
  `continueConversation` and `getContinuationPrompt` returns non-null.
- `DayAgentStrategy.getContinuationPrompt`
  (`lib/features/daily_os_next/agents/workflow/day_agent_strategy.dart`)
  **always** returns a non-null string, and `processToolCalls` always returns
  `continueConversation`.
- Net effect: once the forced retry triggers, a `tool_choice`-compliant
  provider must emit the same tool call every turn until
  `manager.canContinue()` fails at the turn cap (default 10,
  `agent_config.dart`). Each extra turn re-runs `persistDraftPlan` (fresh plan
  write + `onPersistedStateChanged`) or `persistParsedItems` (soft-deletes and
  re-creates all parsed items with new UUIDs —
  `day_agent_capture_service.dart`).
- Contrast the proven pattern: `TaskAgentStrategy.getContinuationPrompt`
  (`task_agent_strategy.dart`) returns **null once the report is submitted**,
  terminating its forced `update_report` retry after one call.

**Fix:** mirror `TaskAgentStrategy`. The strategy already tracks
`didPersistCaptureParse` / `didPersistDraftDayPlan`. Note the subtlety: in the
*normal* (non-forced) flow the model may legitimately continue after
`draft_day_plan` (record observations, `set_next_wake`), so an unconditional
"null once persisted" would truncate normal wakes. Recommended shape: a
`forcedRecovery` mode flag on `DayAgentStrategy` (set by the two `_force*`
helpers, or a thin wrapper strategy) under which `getContinuationPrompt`
returns null (or `processToolCalls` returns `ConversationAction.complete`)
as soon as the mandated output has been persisted.

**Tests:** extend `day_agent_workflow_test.dart` / `day_agent_strategy_test.dart`
with a fake conversation loop that honors `toolChoice` (the current tests stub
`sendMessage` entirely, which is why this bug survived — `day_agent_workflow_test.dart:3667+`).
Assert: forced retry executes the mandated tool **exactly once**; parsed items
are not re-created with new UUIDs on retry.

### 2b. Refine wakes have no enforced output; provider failures swallowed (HIGH)

- Only capture wakes (`parse_capture_to_items`) and drafting wakes
  (`draft_day_plan`) have the persisted-output check + forced retry + throw
  (`day_agent_workflow.dart`, `_requiresCaptureParse` / `_requiresDraftDayPlan`
  blocks in `execute`). Refine wakes — whose system prompt mandates
  `propose_plan_diff` — have nothing: a wake that ends in prose reports
  success.
- Worse, `ConversationRepository.sendMessage` catches any mid-turn error
  (429/5xx/timeout/stream abort), logs to an event stream nobody in the wake
  path consumes, and returns normally — so even total inference failure looks
  like success, and the workflow resets the failure counter.

**Fix (two parts):**
1. Track `didPersistPlanDiff` in `DayAgentStrategy` (success of
   `propose_plan_diff`), add `_requiresPlanDiff` for `wakeContext.isRefineWake`,
   a `_forceProposePlanDiffIfMissing` mirror, and a
   `_MissingPlanDiffException`. **Caveat to verify first:** check the refine
   prompt/UX for a legitimate "no change needed" outcome. If the model can
   correctly conclude no diff is warranted, forcing one fabricates changes —
   in that case enforce "diff or explicit `record_observations` no-op note"
   rather than blind forcing, or have the forced call allow an empty `changes`
   list if the service accepts it. Decide from the code, not this document.
2. Surface terminal turn errors from `sendMessage` without breaking other
   callers (task agent etc.): an opt-in `rethrowOnTerminalError: true`
   parameter (or an error field on the returned usage/result consumed by the
   day workflow) so the wake is recorded as failed and the failure counter is
   not reset. Keep blast radius zero for existing callers.

**Tests:** refine wake ending in prose → wake fails (or forces) instead of
success; provider error mid-refine → wake recorded failed, failure count
increments.

### 2c. `set_next_wake` handler exceptions abort the remaining tool batch (MEDIUM, same files)

Unexpected exceptions in the `set_next_wake` handler escape
`DayAgentStrategy.processToolCalls` and silently abort the remaining tool
calls in the batch. Wrap per-tool dispatch so one tool's unexpected exception
becomes a model-visible error response for that call id (the pattern already
used for unknown tools) instead of killing the batch. Verify the exact
escape path in `day_agent_strategy.dart` first.

## Bucket 3 — Plan state guards (one commit)

### 3a. `persistDraftPlan` silently reverts a committed plan (verified ✓, HIGH)

`DayAgentPlanService.persistDraftPlan`
(`lib/features/daily_os_next/agents/service/day_agent_plan_service.dart`)
validates categories, dayId/planDate match, capture, capacity — but never
checks the existing plan's status, and unconditionally writes
`status: DayPlanStatus.draft()` over the deterministic plan id
(`dayAgentPlanEntityId(dayId)`). A drafting wake against an already-committed
day silently flips it back to draft and replaces all blocks. `commitDay` /
`uncommitDay` in the same file show the guard pattern (status checks before
mutation).

**Fix:** load the existing plan (`planForDay`-style read, not just
`draftPlanForDay`) and throw a `DayAgentCaptureException` when its status is
`DayPlanStatusCommitted` ("day <dayId> is committed; refine or uncommit
first"). The seeded directives already tell the model the service rejects
this — make that true. Check `deletePlanForDay`/redraft interplay: a
soft-deleted committed plan must not sneak past the guard via the upsert path.

**Tests:** drafting over committed plan → rejected, plan unchanged; drafting
over a draft plan → still works; tool path returns model-visible failure.

### 3b. Stale-diff fence (`baselinePlanId`) is structurally inert (HIGH)

`proposePlanDiff` documents `baselinePlanId` as a stale-diff guard, but the
plan id is the deterministic constant `day_agent_plan:<dayId>`
(`day_agent_slots.dart`), reused by every `persistDraftPlan` for the day —
so `baselinePlanId != plan.id` can never fire. Redrafts also never retract
pending change sets, so a stale diff (possibly proposed on another device —
multi-device sync widens the window) applies against a restructured plan.

**Fix (content-based, no schema change preferred):** at accept time, verify
each pending item's `from` snapshot (start/end/blockId existence) against the
live plan block before applying; fail mismatched items with a clear,
model-visible/user-visible message while applying still-valid ones. If a
schema change turns out cheaper than expected, stamping the change set with
the plan's `updatedAt` at propose time and rejecting accept when the live
plan is newer is the alternative — but check `ChangeSetEntity` freezed
regen cost first (`make build_runner`). Additionally consider retracting
pending change sets in `persistDraftPlan` (a redraft supersedes pending
diffs by construction).

**Tests:** propose diff → redraft plan (block moved/removed) → accept: the
stale items fail individually, valid items apply, no silent corruption.
Glados property test candidate: for arbitrary (plan, diff, plan-mutation)
sequences, accepted items' `from` snapshots always matched the live plan at
apply time.

## Bucket 4 — Durability (one commit)

### 4a. Capture-parse wakes are silently destroyed (HIGH)

`submitCapture` (`day_agent_capture_service.dart`) enqueues the parse wake
with `supersede: false`, but that only stops *this* wake from removing others.
Three loss paths: (1) any later `supersede: true` wake in the same
(agentId, workspaceKey) — drafting (`enqueueDraftingWake`), refine
(`enqueueRefineWake` in `day_agent_service.dart`), or scheduled pre-warm —
calls `queue.removeByAgent` (`wake_orchestrator.dart`) and deletes the queued
parse job; (2) app kill between `submitCapture` and drain completion (queue is
in-memory); (3) failed wake — no retry. Result: a capture that never parses,
no re-parse affordance, stranded user speech. Related medium finding: there is
no durable retry either ("Capture parsing has no durable retry").

**Fix:** make parse-pending durable and recoverable:
- On restore (`DayAgentService.restoreSubscriptions` or equivalent startup
  path), find recent captures (bound the lookback, e.g. 7 days) that have no
  parsed items and no terminal parse failure marker, and re-enqueue their
  parse wakes. Check what queryable signal distinguishes "parsed" — parsed
  items reference their capture; if ambiguous, add a parse-state field to
  `CaptureEntity` only if freezed regen is acceptable, else derive.
- Exclude capture-parse jobs from same-workspace superseding: in the wake
  queue/orchestrator, jobs whose trigger tokens include
  `capture_submitted:` should survive `removeByAgent` superseding (the
  foundation layer is shared with task agents — gate the exclusion carefully
  and test the task-agent path is unaffected).

**Tests:** superseding drafting wake does not delete a queued parse wake;
restart with an unparsed capture re-enqueues exactly one parse wake (no
duplicates when one is already queued); parsed captures are not re-enqueued.

### 4b. Scheduled pre-warm wakes double-fire across devices (HIGH)

`ScheduledWakeManager._processDueRecords`
(`lib/features/agents/wake/scheduled_wake_manager.dart`) enqueues first, then
flips the synced record to consumed via LWW — the comment says this only
prevents resurrection. `start()` immediately checks, so two devices opened in
the morning both fire the due record before the consumed flip syncs
(ADR 0018 predicted exactly this and prescribed a lease + source-derived
idempotency key; neither exists). Conversely, fire-then-consume is
at-most-once on a single device: a crash between flip and LLM run drops the
pre-warm. `run_key_factory.dart` derives manual run keys from a per-device
local timestamp, so the two devices' runs don't even collide on run key.

**Fix:** derive the run key for record-fired wakes deterministically from the
scheduled-wake entity id + its `scheduledAt` (stable across devices), and
suppress execution when a wake run with that key already exists in synced
state (check what `AgentRepository` exposes for run lookups by key — the
ADR 0018 designated-primary lease is the bigger alternative; prefer the
idempotent run key as the minimal sound fix). Consider consume-then-fire vs
fire-then-consume ordering for the single-device crash case and document the
chosen at-least/at-most-once semantics in the wake README.

**Tests:** two managers (simulated devices) processing the same due record
produce one effective run (second suppressed by run-key collision); crash
simulation (flip without run) → behavior matches the documented semantics.

### 4c. Decision 5 regressed: capture planning-day derived from `capturedAt` (HIGH)

`submitCapture` has no `dayId` parameter and stamps
`dayId = dayAgentIdForDate(capturedAt)`. Consequently both capture UIs
falsify `capturedAt` to route plan-ahead captures:
`day_planning_modal.dart` (`_capturedAtForDay`) and `capture_page.dart`
(`capturedAtForSelectedDate`) — planning tomorrow tonight persists a synced
`CaptureEntity` with a **future** `capturedAt`, corrupting the episodic
record (memory log `sourceAt`, migration recency cutoff, prompt capture
block all treat it as real time). The model-side `submit_capture` tool also
has no `dayId`, so a capture submitted during a wake for tomorrow's workspace
routes to today. The refine path already proves the correct form:
`enqueueRefineWake` sets `dayId` explicitly while keeping `capturedAt = now`.

**Fix:** add explicit `dayId` to `DayAgentCaptureService.submitCapture`, the
`DayAgentInterface.submitCapture` seam, and the `submit_capture` tool schema
(validate the tool's `dayId` against the wake workspace in the workflow's
existing day guard — it inspects `args['dayId']`). Derive from `capturedAt`
only as a legacy fallback. Then revert `_capturedAtForDay` /
`capturedAtForSelectedDate` so `capturedAt` records real capture time again.

**Tests:** capture for tomorrow keeps `capturedAt = now` and
`dayId = tomorrow`; tool path rejects `dayId` mismatching the wake workspace;
fallback still derives from `capturedAt` when `dayId` absent.

## Bucket 5 — Compaction growth bound (one commit, the largest)

### Checkpoints accumulate quadratically; every wake reloads all of them (HIGH)

`agent_log_compactor.dart`: each fold persists a summary whose
`coveredSources` is the monotone union of everything ever folded
(`{...?view.active?.coveredSources, ...}`), so checkpoint k carries O(k)
entries → total persisted+synced bytes O(N²). Old checkpoints are never
tombstoned; `loadSummaries` loads ALL summary messages and payloads on every
`_projectActiveView`, which runs (at least) twice per wake. For a multi-year
planner there is no growth bound — the ADR's own flagged consequence with no
code behind it.

**Fix (scoped to stay safe):**
1. Tombstone (deletedAt) checkpoints strictly covered by a newer valid
   checkpoint at fold time. **Convergence caution (ADR 0018):** deletion must
   be safe under concurrent multi-device folds — verify a tombstoned
   checkpoint can't be the other device's active head; prefer tombstoning
   only checkpoints superseded by a checkpoint that is itself causally
   stable, and keep the most recent k (e.g. 2) regardless.
2. Cache the `_projectActiveView` result per wake so `maybeCompact` and
   prompt assembly don't double-load.
3. **Do not** change the `coveredSources` payload format in this pass: the
   compaction fold is deterministic and machine-checked for byte-stable
   prefixes (see `test/` invariant tests around the projection/compaction —
   run the full `test/features/agents/projection/` suite). Delta-encoding
   `coveredSources` changes persisted bytes and risks the prefix-cache
   invariant; flag it as follow-up instead.

**Tests:** fold N times → older fully-covered checkpoints tombstoned, active
view identical before/after tombstoning (byte-stable rendered output);
projection suite green; per-wake summary loads bounded (assert via the
existing instrumentation if present, else count repository reads in a fake).

### [HIGH] User-verdict plan tools (accept_diff, commit_day, uncommit_day, revert_diff) are model-callable with no structural gate, and accept_diff bypasses the day-workspace guard — **fixed in bucket 1 (62447999e)**

- **Dimension:** adr-conformance  
- **File:** `lib/features/daily_os_next/agents/tools/day_agent_tools.dart`

**Description:** ADR 0006/0022 frame the ChangeSet as a deferred-confirmation gate: the agent proposes, the user confirms. In code the gate is advisory only — the planner model is handed fully-wired accept_diff/revert_diff/commit_day/uncommit_day tools and can apply its own proposed plan diff or flip a day's commit state in a single wake. Because these tools take changeSetId/dayId-free arguments resolved server-side, the wake's day-workspace rejection (Decision 4) never sees them. A single instruction-following lapse (common under long prompts) silently mutates the live plan and records a 'confirmed' decision the user never made, with no audit field to distinguish it.

**Evidence:** day_agent_tools.dart:466-543 defines accept_diff/revert_diff/commit_day/uncommit_day as LLM tool definitions; day_agent_tool_names.dart:85-93 puts them in planTools, so day_agent_workflow.dart:1029-1040 (_isToolEnabled) hands them to the model whenever planService is configured, and day_agent_plan_service.dart:74-89 executes them unconditionally. The ONLY restraint is prompt prose (day_agent_workflow.dart:918-924: "Do not call accept_diff or revert_diff autonomously — those are the user's verdicts"). The UI never goes through this tool path — real_day_agent.dart:360/394/421 call planService.acceptPlanDiff/revertPlanDiff/commitDay directly — so the tool wiring is exclusively model-reachable. propose_plan_diff's tool result returns the changeSetId (day_agent_plan_tool_dispatcher.dart:64-65), so a refine wake can propose a diff and immediately self-accept it. The wake's day guard at day_agent_workflow.dart:511-521 only inspects args['dayId']; accept_diff/revert_diff carry only changeSetId, and _resolvePlanDiff derives the target day from changeSet.taskId (day_agent_plan_service.dart:597-599), so the workspace check is fully bypassed. The resulting ChangeDecisionEntity (day_agent_plan_service.dart:655-670) records no actor, making a model-issued accept indistinguishable from a user confirmation. ADR 0022 Decision 11 and the ADR-0006 ChangeSet gate require these verdicts to be structurally user-only; ADR 0022 Decision 4 requires day mismatches to be rejected. Notably this same codebase does not trust prompt compliance elsewhere — it forces tool choice via retries (day_agent_workflow.dart:1062-1147) precisely because models deviate.

**Suggested fix:** Remove acceptDiff, revertDiff, commitDay, and uncommitDay from the model-visible tool set (exclude them in _buildToolDefinitions / _isToolEnabled — the UI already calls the service methods directly), or hard-reject them in _executeToolHandler. Additionally record an actor (user vs agent) on ChangeDecisionEntity, and validate in _resolvePlanDiff that the change set's plan day matches the wake workspace when invoked from a wake.

### [HIGH] Decision 5 regressed: capture planning-day is derived from capturedAt, forcing the UI to falsify capture timestamps and leaving the agent tool path wrong-day-routed — **bucket 4c**

- **Dimension:** adr-conformance  
- **File:** `lib/features/daily_os_next/agents/service/day_agent_capture_service.dart`

**Description:** The implementation re-conflates the two concepts ADR 0022 Decision 5 explicitly separates. Every plan-ahead capture (the core evening-planning ritual) stores a falsified, possibly future capturedAt into the durable synced log, corrupting the planner's episodic record of when things were actually said; and the model-side submit_capture tool routes captures (and their parse wakes) to the wrong day workspace whenever the wake targets a non-today day, with no guard able to catch it.

**Evidence:** ADR 0022 Decision 5 (docs/adr/0022-long-lived-daily-os-planner.md:87-89): "Captured time and planning target are not the same concept. CaptureEntity must carry the planning dayId directly...". The entity does carry dayId, but submitCapture has no dayId parameter and stamps dayId = dayAgentIdForDate(capturedAt) (day_agent_capture_service.dart:135-161) — planning target is a pure function of captured time, the exact conflation the decision forbids. Consequently both capture surfaces falsify capturedAt to route the day: day_planning_modal.dart:123-129 (_capturedAtForDay shifts now onto the planned day) used at :193, and capture_page.dart:887 + 911-934 (capturedAtForSelectedDate, with the comment "capturedAt routes the capture to the day-agent for that day"). Planning tomorrow tonight therefore persists a synced CaptureEntity whose capturedAt is a future timestamp. Downstream consumers treat capturedAt as real capture time: the memory log's source position (day_capture_events.dart:28-40, sourceAt: capture.capturedAt), the migration recency cutoff (day_agent_service.dart:303), and the prompt's capture block (day_agent_workflow.dart:1309-1315). The agent-side submit_capture tool (day_agent_tools.dart:108-131) likewise has no dayId, so a capture submitted by the model during a wake for tomorrow's workspace is stamped with today's dayId and its parse wake is enqueued into today's workspace (day_agent_capture_service.dart:176-188) — invisible to the day guard at day_agent_workflow.dart:511-521 because no dayId argument exists. The refine path proves the entity supports the correct form: enqueueRefineWake sets dayId explicitly while keeping capturedAt = now (day_agent_service.dart:434-446).

**Suggested fix:** Add an explicit dayId parameter to DayAgentCaptureService.submitCapture, the DayAgentInterface.submitCapture seam, and the submit_capture tool schema (validated against the wake workspace in _executeToolHandler); derive from capturedAt only as a legacy fallback. Then revert _capturedAtForDay/capturedAtForSelectedDate so capturedAt records real capture time again.

### [HIGH] Scheduled pre-warm wakes double-fire across devices: ADR 0018's idempotency key and lease are not implemented anywhere — **bucket 4b**

- **Dimension:** concurrency-lifecycle  
- **File:** `lib/features/agents/wake/scheduled_wake_manager.dart`

**Description:** ADR 0022 moved day pre-warms into synced ScheduledWakeEntity records, making every device's hourly-plus-startup ScheduledWakeManager an independent firer of the same record. ADR 0018 explicitly predicted this ("if two devices observe the same trigger and both run a behavior, side effects duplicate") and prescribed a lease plus a source-derived idempotency key; neither exists in code. The realistic trigger needs no partition: phone and laptop both launched within Matrix sync latency of each other after the wake comes due. Conversely, the fire-then-consume ordering makes the wake at-most-once on a single device: a crash between the consumed flip and the LLM run silently drops the morning pre-warm.

**Evidence:** scheduled_wake_manager.dart:130-160 — _processDueRecords enqueues the wake first, then flips the synced record to consumed via LWW; the comment explicitly states this only prevents resurrection. scheduled_wake_manager.dart:42-43 — start() immediately runs _checkAndEnqueue, so two devices opened in the morning both find the due 07:00 record pending before the consumed flip can sync. run_key_factory.dart:46-56 — the manual run key is SHA(agentId|reason|workspaceKey|local timestamp): per-device wall clock, so two devices firing the same record produce different run keys and the wake_run_log UNIQUE dedup (device-local table, agent_repository.dart:928-938 reads agent_entities; wake runs are inserted via repository.insertWakeRun, never synced) cannot collapse them. ADR 0018 rule 9 (docs/adr/0018:111-121) requires side effects to carry an idempotency key built from "the scheduled-wake *entity* id … never a per-run local UUID"; rg for lease/fencing/idempotency finds no implementation in lib/features/agents or lib/features/daily_os_next — the only single-flight is the in-process WakeRunner lock (wake_runner.dart:30-37). The merge-layer override in agent_concurrent_resolver.dart ("no double-fire") only converges the *record*, not the execution. Consequence: both devices run the full planner LLM wake; observations/thoughts are appended with uuid-v4 ids (day_agent_persistence.dart:99-134) and converge by set-union (ADR 0018 rule 1), so near-duplicate entries permanently pollute the planner's cross-day memory, plus duplicate token spend and two competing plan writes racing LWW. The inverse hole: enqueue-then-consume ordering (scheduled_wake_manager.dart:135-147) means a kill after the consumed upsert but before the in-memory job executes loses the pre-warm forever — startup marks orphaned 'running' runs abandoned with no replay (agent_providers.dart:366-375, agent_repository.dart:1728-1740). scheduled_wake_manager_test.dart has no two-device/two-manager scenario.

**Suggested fix:** Derive the run key for record-fired wakes from the scheduled-wake entity id + scheduledAt (stable across devices) and persist wake completion keyed by it in synced state, so the second device can suppress an already-executed fire; alternatively implement the ADR 0018 designated-primary lease for (planner, dayId). Flip the record to consumed only after the wake-run completes (or add a 'claimed' state with replay-on-abandon) so a crash re-fires instead of losing the wake.

### [HIGH] persistDraftPlan silently reverts a committed day plan to draft — no status guard, violating ADR 0018 rule 10 — **bucket 3a**

- **Dimension:** concurrency-lifecycle  
- **File:** `lib/features/daily_os_next/agents/service/day_agent_plan_service.dart`

**Description:** The committed day plan — the strongest user-facing commitment in the feature — has no write-path protection against late, duplicate, or remote drafting wakes. Because the plan lives at one deterministic id and merges by last-write-wins, any draft_day_plan execution that lands after commitDay silently destroys both the committed status and the user's committed block list, with no history to recover from and no reconcile surface. Everything that prevents this today is a natural-language instruction in the system prompt plus UI sequencing assumptions.

**Evidence:** day_agent_plan_service.dart:802-840 — persistDraftPlan reads the existing plan only to preserve createdAt (line 802, 820) and then unconditionally overwrites the deterministic register `day_agent_plan:<dayId>` with status DayPlanStatus.draft (line 812) and a wholesale-replaced block list. Contrast commitDay/uncommitDay (lines 407-458, 524-580), which carefully gate on current status. Three concrete triggers: (1) zombie wake — wake_drain_engine.dart:250-256 documents that an aborted/timed-out executor "runs to completion in the background" and "its mutations land via the normal DB path"; a drafting wake that hits the 2-minute cap (wake_orchestrator.dart:188) is marked aborted, the user re-drafts and commits, then the zombie's draft_day_plan lands minutes later with a newer updatedAt and clobbers the committed plan; (2) cross-device — the plan is a mutable LWW register (agent_concurrent_resolver.dart:34-40 'whole-version winner'), so a pre-warm wake on device A that chooses to call draft_day_plan (the tool is always available; only a prompt instruction restrains it, day_agent_workflow.dart:887-908) overwrites the day the user committed on device B, picked silently by updatedAt LWW; (3) a stale queued drafting wake draining after the user commits. ADR 0018 rule 10: divergent user-facing commitments must "surface as a ChangeSet … never silently overwritten" — commit status is exactly such a commitment.

**Suggested fix:** In persistDraftPlan, load the existing plan and refuse (tool error) when its status is committed — directing the model to the propose_plan_diff path — or convert the incoming draft into a pending plan-diff ChangeSet against the committed plan. - **File:** `lib/features/daily_os_next/agents/workflow/day_agent_workflow.dart`

**Description:** Once the read-flip engages (the normal steady state), a critical observation is just another tail line that the next LLM fold may compress away — its criticality marker never even reaches the summarizer. The one mechanism the foundation layer already has against exactly this drift mode exists 30 lines away in the task agent and was not carried over to the planner, the agent with the longest horizon and therefore the most folds. Grep confirms zero `critical`/`significance` handling anywhere under lib/features/daily_os_next/agents/ outside the tool schema enum.

**Evidence:** The day agent's `record_observations` schema accepts `priority: 'critical'` (day_agent_tools.dart:27-30) and persists it into the payload (day_agent_persistence.dart:114-118). But unlike the task agent — which has a dedicated verbatim critical section in every prompt (task_agent_workflow.dart:1082-1124) — the day-agent prompt has no critical handling at all: observations reach the prompt only via the compacted log, whose render (`renderCompactedSourceLine`, compaction_summary.dart:166-187) and summarizer fold input emit only `content['text']`, silently dropping the priority field; the `recentObservations` fallback (last 20, text only) is included only `if (compactedLog == null)` (day_agent_context_builder.dart:54-63). ADR 0022 Decision 10 (:131-132) points at this mitigation ('Always-on critical items may still be injected verbatim, cf. ADR 0014 critical-observation injection'), and the research doc's §4.1 names the precise failure: 'a rare but critical instruction from day one… tends to vanish by the third pass.'

**Suggested fix:** Port the task agent's critical section: query observation payloads with `priority == critical` (optionally unresolved-only) and inject them verbatim into the stable prompt prefix regardless of compaction state; at minimum, include the priority tag in `renderCompactedSourceLine`'s observation rendering so folds can preserve it.

### [MEDIUM] DST-broken day window (`localDay + 24h`) in plan parser and diff validation — valid late-evening blocks rejected on 25h days, next-day spill accepted on 23h days — **unscheduled**

- **Dimension:** plan-logic  
- **File:** `lib/features/daily_os_next/agents/service/day_agent_plan_parser.dart`

**Description:** The deterministic in-day validation that every drafted block and every accepted diff must pass uses a day-window computation that is wrong twice a year, producing both false rejections (failed drafting wakes on the 25h day) and false acceptances (cross-midnight contamination on the 23h day), while another file in the same PR already contains the documented correct idiom.

**Evidence:** Four sites compute the day window as `dayStart.add(const Duration(days: 1))`: day_agent_plan_parser.dart:38-44 (block bounds), 104-110 (energy bands), day_agent_plan_diff.dart:165-172 (_assertWithinDay) and 226-235 (_validateApplicableBatch). Dart's DateTime.add is instant arithmetic, so on the DST fall-back day (25h local day, e.g. EU 2026-10-25) dayEnd lands at 23:00 the same day — a legitimate 23:00–23:45 wind-down block throws 'blocks must stay within the planDate day'. Because a drafting wake hard-fails when draft_day_plan never persists (_MissingDraftDayPlanException, day_agent_workflow.dart:398-415), one rejected evening block plus an unlucky forced retry kills the whole drafting wake. On the spring-forward day (23h) dayEnd is 01:00 of the NEXT local day, so blocks/energy bands placed at 00:00–01:00 of the following day pass validation and are stored on the wrong day's plan — breaking the dayId=local-day invariant ADR 0022 Decision 2 builds on. The authors demonstrably know the correct pattern: day_agent_context_builder.dart:170-177 uses `DateTime(start.year, start.month, start.day + 1)` with an explicit comment 'so the window stays at local midnight across DST transitions, where a day may be 23 or 25 hours' — the fix simply wasn't applied to the plan pipeline. No DST test exists in day_agent_plan_service_test.dart (grep for 'DST' is empty).

**Suggested fix:** Replace all four `dayStart.add(const Duration(days: 1))` sites with a shared helper `DateTime nextLocalMidnight(DateTime dayStart) => DateTime(dayStart.year, dayStart.month, dayStart.day + 1)` (matching the context-builder idiom), and add tests pinning a 23h and a 25h day using a fixed timezone.

### [MEDIUM] Model-supplied block ids are stored without uniqueness validation; duplicate ids break diff identity (drop removes all, move/summary disagree on target) — **unscheduled**

- **Dimension:** plan-logic  
- **File:** `lib/features/daily_os_next/agents/service/day_agent_plan_parser.dart`

**Description:** Block id is the identity key for the entire diff/accept machinery, yet the only producer of ids the system does not control (the LLM) is allowed to violate uniqueness silently, after which the three consumers of that key each resolve duplicates differently — confirmed edits can target a different block than the one summarized, and drops delete more than the user approved.

**Evidence:** _parsePlannedBlock accepts the LLM's `id` verbatim with only a fresh-uuid fallback (day_agent_plan_parser.dart:73-76), and persistDraftPlan (plan_service.dart:772-794) performs no intra-plan uniqueness check before persisting — realistic with this prompt, since drafting wakes are told to re-emit the baseline block list (ids included) and an LLM splitting one baseline block into two commonly reuses the source id. Downstream code then disagrees about what an id denotes: proposePlanDiff's `blockById` map keeps the LAST duplicate (plan_service.dart:322-324, feeds validation and _formatChangeSummary fallbacks), _applyItem's move_block edits only the FIRST match via indexWhere (day_agent_plan_diff.dart:353-360), drop_block's removeWhere deletes ALL duplicates (393-401), and _validateApplicableBatch's simulatedIds set collapses duplicates so a second drop of the same id is rejected even though two physical blocks existed (220-225, 293-301). End-to-end failure: model re-drafts splitting 'Deep work' into 09:00 and 15:00 blocks that share id block_abc; a later refine proposes dropping the 15:00 one; the user confirms a summary describing one block and both blocks vanish from the plan. No test covers duplicate ids (grep 'duplicate|same id' over the service and its test file is empty).

**Suggested fix:** In persistDraftPlan, validate ids across the parsed block list and reject duplicates with a tool error (the model retries with corrected ids), or deterministically re-mint the second occurrence; add a regression test that a duplicated id is rejected.

### [MEDIUM] Diff accept path skips the task-reference validation the draft path enforces, and its 'order-aware' simulation ignores moves — phantom taskIds and inverted intervals can be persisted — **unscheduled**

- **Dimension:** plan-logic  
- **File:** `lib/features/daily_os_next/agents/service/day_agent_plan_diff.dart`

**Description:** The accept path is the second writer of plan blocks and is explicitly designed to re-run 'the propose-time invariants' (diff:215-218 doc), but it re-runs only some of them: task-reference integrity is dropped entirely relative to the draft path, and the sequential simulation its own docstring promises covers drops but not moves, allowing an accepted batch to persist a block that violates the end>start invariant every other entry point enforces.

**Evidence:** Two gaps in the same accept path. (1) taskId: _parsePlannedBlock strictly rejects any taskId outside decidedTaskIds/allowedExistingTaskIds (day_agent_plan_parser.dart:61-72, with a comment insisting empty sets are 'not a license' for arbitrary ids), but the refine path passes `to.taskId` straight through (_optionalBlockSnapshot, diff:149-157 → toArgs diff:478) and neither _parseDiffChange nor _validateApplicableBatch (267-336: cases cover only blockId/category/timestamps) ever checks it; _applyItem then writes it onto the block (371-373 for move, 387 for add) and _resolvePlanDiff recomputes pinnedTasks from it (plan_service.dart:693), so an accepted diff can pin a hallucinated or category-forbidden taskId into the plan — invisible to the user because humanSummary (diff:174-198) never renders taskId. (2) moves: the batch validator's doc claims order-awareness so 'one item's effect … is visible to later items' (diff:201-214), but only drops mutate the simulation (simulatedIds.remove, 301); blocksById is never updated, so for a second move of the same block `live` (276-280) is the pre-batch block. Failing scenario: block 09:00–12:00; item0 move toEnd=10:00 validates and applies (09:00–10:00); item1 move toStart=11:00 validates against the stale live end 12:00, then applies against the already-shortened block, persisting an 11:00–10:00 inverted interval whose negative Duration.inMinutes also corrupts scheduledMinutes (_scheduledMinutesFor, parser:145-149).

**Suggested fix:** In _validateApplicableBatch: (a) validate any taskId on add_block/move_block items against live tasks and the resolving agent's allowedCategoryIds (mirror _allowedExistingTaskIds), and surface taskId in humanSummary; (b) simulate moves by updating a working copy of blocksById so effective start/end fall back to the post-prior-item block, and assert end>start on the final effective interval.

### [MEDIUM] Refine wakes have no mandatory-output enforcement — a refine that ends in plain text reports success while the user's requested change silently vanishes — **bucket 2b**

- **Dimension:** plan-logic  
- **File:** `lib/features/daily_os_next/agents/workflow/day_agent_workflow.dart`

**Description:** Of the three wake modes with a contractually mandatory tool output, two are deterministically enforced and the third — the only one carrying a direct, explicit user instruction — relies purely on the model's compliance, making refine the least reliable path in the pipeline despite being the highest-intent one.

**Evidence:** The workflow enforces its two other mandatory wake outputs with a forced-tool-choice retry plus hard failure: capture wakes must persist parse_capture_to_items (day_agent_workflow.dart:371-396, _forceCaptureParseIfMissing 1062-1106) and drafting wakes must persist draft_day_plan (398-415, _forceDraftDayPlanIfMissing 1108-1147). Refine wakes get neither: DayAgentStrategy tracks only _didPersistCaptureParse and _didPersistDraftDayPlan (day_agent_strategy.dart:69-71, 80-83) — there is no didPersistProposePlanDiff — and _requiresCaptureParse explicitly excludes refine wakes (workflow:1048-1060), so a refine capture is exempt from the parse mandate too. Yet the system prompt declares propose_plan_diff the refine wake's priority output (workflow:910-917), and the product flow depends on it: enqueueRefineWake persists the user's transcript as a capture and fires the wake (day_agent_service.dart:398-467), and the UI surfaces results solely via pendingPlanDiffsForDay (plan_service.dart:125-167). Failing scenario: user dictates 'push my deep-work block to after lunch'; the model answers with prose and never calls propose_plan_diff; the wake completes with WakeResult(success: true) (workflow:473-474), the milestone is appended, no pending diff exists, and the UI shows nothing actionable — the user's instruction is lost with no error, no retry, and nothing distinguishable from 'agent decided no change was needed'.

**Suggested fix:** Mirror the existing pattern: track didPersistProposePlanDiff in DayAgentStrategy (set when propose_plan_diff succeeds), and in execute() add a _requiresPlanDiff(wakeContext) branch (refine wake + planService + non-empty transcript capture) with a _forceProposePlanDiffIfMissing forced-tool-choice retry, failing the wake (so the UI can surface the failure) when the diff still wasn't persisted.
