# Lotti AI Settings — Improvement Plan (v2)

## Design direction

The redesign commits to six principles, derived from convergent panel feedback (round 1) and sharpened by the five round-1 reviews:

1. **One accent, one kit.** The page is brand-teal. The search bar, filter chips, and selected states currently leak Material purple (`colorScheme.primary` on the search icon/border) and live off the token system. Every chrome element renders on design-system tokens (`background.level02` surface, `radii.l`, `interactive.enabled` teal). **Sanctioned secondary hues are explicit:** the `DRAFT`/`Active` badge blue and the on-card capability-tag blue are intentional, token-backed secondary accents and stay. What is banned is *Material-primary purple* and any second *chrome* accent. The audit covers four surfaces, not two: (a) search field, (b) selected filter-chip state, (c) the per-provider **leading icon tiles** (the tinted OpenAI/Gemini/Ollama brand tiles — these are a sanctioned per-brand tint, not a rogue accent, but they must be confirmed token-backed and must not introduce a *chrome* accent), and (d) the on-card capability tags.
2. **The dependency chain is walkable in both directions, not just nameable.** Provider→model→profile is the core mental model. Today it's described in text across three flat lists the user must cross-reference by hand. Counts and slot model-names become navigation, **bidirectionally** (provider count → its models; profile slot → its model). Empty states point up the chain. A first-run guided path sequences the whole chain as one flow.
3. **One vocabulary for capabilities — and the fix is scoped correctly.** The model **cards already render the full task-oriented capability list**: `modelCapabilityLabels` (`ai_provider_visual.dart:96`) returns a `List<String>`, and Gemini 3 Flash already shows two tags ("Image recognition" + "Transcription"). **The ONLY mismatch is the filter chips**, which render off a separate `Modality` enum (Text/Vision/Audio/Reasoning). The fix is to reconcile the *filter taxonomy* with the task-oriented card/slot vocabulary — **not** to change how cards render tags. We adopt the task-oriented set (Thinking / Image recognition / Transcription / Image generation) for the filter chips so chips, card tags, and profile slots share one vocabulary.
4. **Honest, complete state with a legible floor.** Profiles silently omit unset slots, so "deliberately minimal" looks identical to "broken/incomplete." We surface unset slots and a completeness signal. **Every newly introduced muted string** (unset "Not set" slots, generated fallback descriptions, "N of 4 configured", quiet Ready/Incomplete status) renders at **at least `mediumEmphasis`** — never `lowEmphasis` — because these land on dark cards where the existing secondary grey is already borderline. Keep the model count visible even on broken providers.
5. **Touch targets, occlusion, and text scaling are non-negotiable.** The FAB overlaps live card content on phone; the per-card overflow menu is a 32px target. Both are WCAG 2.5.5 blockers, fixed globally. Additionally, the three tabs must survive **large text scaling (`textScaler` 1.3–2.0)** without truncating single-line slot rows or re-occluding cards under the FAB.
6. **Decide layout, don't leave it to accident.** Odd-count two-column behavior is specified for **all three desktop tabs**, not just Models.

**Conflict resolutions (stated up front):**
- **Data density (IA expert: "add provider name to every model row, baseline Text chip") vs simplicity (everyone else).** Call: *partial.* Suppress the mono `providerModelId` only when it's a trivial slug of the title; do **not** add a per-row provider-name string (the tinted icon plus provider-filtered navigation carries the link), and do **not** add an always-on "Text" baseline chip. Instead drop the "Text" filter so the filter set matches what cards can show.
- **Card-body tap: provider→detail vs model/profile→edit (Interaction expert demands a commit, not a deferral).** Call: **all three tabs route card-body tap → detail/overview; edit is one tap deeper.** Why: same gesture must mean the same thing across three sibling tabs (mode-error elimination). Provider already has a detail page (PR-4 path); we add lightweight model and profile detail/overview targets so the three agree. This is now **P1**, not a deferred P2. (See §5.)
- **"N models" count tap vs card-body tap on the provider card (Interaction expert: two nested tap targets is itself cumbersome).** Call: the count meta is a **visibly distinct ≥48px target** with its own affordance (chevron + underline treatment), so it cannot be mis-tapped into provider-detail. (See §1, AC#10.)
- **Profile-slot deep-link priority (Onboarding expert: P2 leaves the chain one-directional).** Call: **promote slot→model deep-link to P1**, peer of the provider count jump. The chain must be walkable both ways; Profiles is where the abstraction becomes concrete for a newcomer.
- **"Set active" affordance on profile cards (UX expert) vs card simplicity.** Call: ship it — P1.
- **`thinkingHighEnd`: 4 or 5 slots (Onboarding expert: the "N of 4" denominator is undefined until resolved).** Call: **4 slots.** `resolved_profile.dart:85` already resolves `thinkingHighEndModelId ?? thinkingModelId` — the high-end id is an *optional override of the Thinking slot*, not a fifth skill. We render four slots, the count says "N of 4", and we document `thinkingHighEnd` as a Thinking-slot override surfaced inside the Thinking slot's editor, not as its own card row. Count, rendered slots, and resolver now agree. (See §3 and §6.)

## Design-system component reuse (audit 2026-06-23)

The original plan was token-strong but under-specified **component** reuse. A DS-component audit found the chrome rebuilds widgets the design system already ships. Mandate: prefer the DS component over raw Material wherever one exists.

- **Search →** retire `LottiSearchBar`, adopt **`DesignSystemSearch`** app-wide (see §4).
- **Filter / capability chips →** replace Material `FilterChip` / `ActionChip` (`ai_settings_filter_chips.dart`) with **`DesignSystemChip`** (already used in this feature's `inference_profile_slot_field.dart`).
- **Card "⋯" overflow menu →** replace `PopupMenuButton` (`ai_card_action_menu.dart`) with **`DesignSystemContextMenu`** (handles token resolution + sizing; also fixes the ≥48px target).
- **"Fix →" affordance →** **`DesignSystemButton`** (small text/secondary) instead of a bare `InkWell` + `Text`.
- **Edit-form fields →** the forms use the AI-specific `AiTextField`; evaluate migrating to **`DesignSystemTextInput`/`DesignSystemTextarea`** when the forms are tackled (P1) — decision deferred to form-time after checking behavioral parity (validation, API-key visibility toggle, connection verifier).
- **Already good DS citizens (keep):** `DesignSystemFloatingActionButton` (FAB), `DesignSystemBadge` (status / DRAFT / Active).

Note on the two color systems: Material `colorScheme.primary` is **purple** (FlexColorScheme, the active theme) while DS `tokens.colors.interactive.enabled` is **teal `0xFF1F7963`**. DS teal is the documented migration destination; this rework moves chrome onto DS components/tokens rather than the legacy Material colorScheme.

## Explicitly NOT changing

- **The three-tab structure and left-to-right order** (Providers → Models → Profiles). It correctly encodes the dependency chain; the fixes make it *legible*, not different.
- **How model cards render capability tags.** Cards already render the full task-oriented `List<String>`; we touch only the *filter* taxonomy. (Corrected from v1.)
- **Status vocabulary and the three differentiated provider states** (Connected / Offline / Invalid key). The strongest affordance in the feature — keep.
- **The `DRAFT` / `Active` badge system** and its sanctioned secondary blue. Tokenized and not color-only; reuse as-is.
- **The mono `providerModelId` concept.** We only suppress trivial-slug echoes; the disambiguator stays where it adds value.
- **The per-provider brand-tinted leading icon tiles** as a concept. We confirm they're token-backed; we don't flatten the brand tint.
- **The debounced live connection verifier, "Save as draft," Cmd+S, and the tiered FTUE provider picker.** Genuinely good flows; the only change is not stranding users in the power-user list and reusing the FTUE scaffold for the guided path.
- **The shared `AiCardActionMenuButton` entry point.** We enlarge its hit area and per-surface its tooltip; we don't replace it.
- **Dangling-reference warning rendering.** Honest and accessible; keep.

## Acceptance criteria (panel ≥9)

1. **Single accent, four-surface audit.** No Material-primary (purple) renders in chrome on the three tabs: search icon, search border, and selected filter-chip state are teal `interactive.enabled`; the search field uses `background.level02` + `radii.l`. The leading icon tiles are confirmed token-backed per-brand tints (no chrome accent); the `DRAFT`/`Active` badge blue and capability-tag blue are confirmed sanctioned token-backed secondary hues. *(Resolves Visual #1 + the leading-icon-tile and sanctioned-blue mustFixes.)*
2. **One capability vocabulary, scoped to filters.** Filter chips use the task-oriented set (Thinking / Image recognition / Transcription / Image generation); no "Text" filter; an Image-generation filter exists. Card tag rendering is unchanged. A user filtering "Image recognition" finds cards already tagged "Image recognition." *(Resolves the all-five IA failure with the IA expert's correction.)*
3. **On-card tags are one kit.** Capability pills on model cards render on the same token set and sanctioned blue as the badges, and multi-tag rows wrap (verified at phone width and `textScaler` 2.0) without clipping. *(Resolves Visual mustFix #2.)*
4. **Chain is walkable both directions.** Tapping "N models" on a provider switches to Models with the **existing provider filter** set to that provider (no new view); tapping a profile slot's model name opens that model's detail/overview. Both are P0/P1, peers. *(Resolves the top Onboarding/IA priority + the IA "reuse existing filter" mustFix + the Onboarding slot-deep-link-promotion mustFix.)*
5. **One tap meaning across tabs.** Card-body tap → detail/overview on all three tabs; edit is one tap deeper. No tab routes a card-body tap straight into the editor. *(Resolves the Interaction card-tap-consistency mustFix.)*
6. **Count target is unmistakable.** The provider "N models" target is a visibly distinct ≥48px affordance (chevron + underline), separate from the card-body tap, so it cannot be mis-tapped into provider-detail. *(Resolves the Interaction count-vs-body disambiguation mustFix.)*
7. **Tap budget owned.** End-to-end happy path is specified and measured: **API key in hand → working profile in ≤ N taps**, with the FTUE provider-picker, provider-scoped model add (`preselectedProviderId`), and profile assembly counted; the guided path must not exceed that budget. *(Target N stated in the build ticket; resolves the Interaction "owns the core journey" bar.)*
8. **No occlusion, real touch targets, at scale.** On phone, no FAB overlaps any card content (verified on full Providers/Models lists) and every per-card overflow menu is ≥48px — **and both hold at `textScaler` 1.3–2.0**. *(Resolves both A11y critical/high items + the text-scaling mustFix for the FAB reservation.)*
9. **Filter chips never clip silently.** On the narrowest supported phone width and at `textScaler` 2.0, all four capability filters are reachable (wrapped to a second line). *(Resolves the A11y high + 3 other mentions.)*
10. **Legibility floor for new muted text.** Every newly introduced muted string — unset "Not set" slots, generated fallback descriptions, "N of 4 skills configured", quiet non-active "Ready/Incomplete" status — renders at **≥`mediumEmphasis`**, never `lowEmphasis`. *(Resolves the A11y muted-text mustFix.)*
11. **Text-scaling robustness.** The three tabs are exercised at `textScaler` 1.3–2.0: single-line slot rows ("Image recognition → Gemini 3 Flash") do not truncate (wrap or ellipsize-with-tooltip, decided in the ticket) and no card re-occludes under the FAB. *(Resolves the A11y text-scaling acceptance mustFix.)*
