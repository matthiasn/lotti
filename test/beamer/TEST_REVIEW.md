## Test quality improvements

- [ ] **[HIGH] `test/beamer/not_logged_in_toast_test.dart` — manual GetIt wiring should use
  `setUpTestGetIt`.**
  Lines 139–155, 240–248, 372–381, 455–466 contain inline
  `getIt.isRegistered/unregister/registerSingleton` boilerplate — exactly the pattern
  `AGENTS.md` forbids. Tests should call `setUpTestGetIt(additionalSetup: () { ... })`
  and `tearDownTestGetIt()` (imported from `test/widget_test_utils.dart`) instead of the
  repeated if-isRegistered-unregister-register triples. The `_registerNavService` and
  `_registerTimeService` helpers at lines 138–155 are a partial step but bypass the
  centralized lifecycle entirely.

- [ ] **[HIGH] `test/beamer/not_logged_in_toast_test.dart` — `pumpAndSettle` misuse (10 calls).**
  Every test in this file calls `pumpAndSettle()` immediately after `pumpWidget` and again after
  emitting a stream event. The toast appears after a
  `WidgetsBinding.instance.addPostFrameCallback` — a single `tester.pump()` is sufficient to
  flush the post-frame callback. `pumpAndSettle` adds up to 10 × 10 s timeout exposure per run
  and can cause hangs if any animation in the subtree fails to settle. Replace with targeted
  `tester.pump()` calls.

- [ ] **[HIGH] `test/beamer/my_beamer_app_test.dart` — constructor-smoke and constant-value
  tests have zero signal.**
  Several tests in the `'MyBeamerApp theming'` group (lines ~37–202) assert on pure Dart
  constant expressions or on objects that were constructed in the same test body:
  - Lines 134–147: `'GestureDetector onTap calls unfocus on primary focus'` defines a local
    closure, calls it, and asserts the variable it set is `isTrue`. The production code is not
    involved at all.
  - Lines 179–186: `'default tooltip visibility is true when stream has no value'` evaluates
    `const bool? streamValue = null; const enableTooltips = streamValue ?? true` — both are
    pure constant expressions with no relationship to provider behavior.
  - Lines 188–202: `'tooltip visibility uses stream value when available'` likewise tests
    `const bool` literal comparisons.
  - Lines 205–213: `'currentPath from NavService is used for initial route'` checks that a
    literal string `'/settings'` is non-empty and starts with `'/'`.
  These tests should be deleted; they provide false coverage confidence.

- [ ] **[MED] `test/beamer/locations/` — five files define inline `MockBuildContext extends

## Generative (Glados) testing opportunities

- [ ] **[MED] `_dailyOsNextRouteFrom(Uri)` in `lib/beamer/locations/calendar_location.dart`
  (line 70) — parse/reject round-trip.**
  This pure function takes a `Uri` and returns either a typed route record or `null`. Its
  contract has several algebraic invariants:
  - Any `Uri` with `< 3` path segments must return `null`.
  - Any `Uri` whose second segment is not a valid `DailyOsNextRouteTarget.name` must return
    `null`.
  - Any `Uri` whose third segment is not a valid ISO date string must return `null`.
  - A valid triple `(calendar, <target.name>, <isoDate>)` must always produce a non-null result
    with the correct `target` and a `date` that round-trips back through
    `parseDailyOsNextRouteDate`.
  A Glados property test driving random URI segment combinations (using a custom `_AnyUriSegments`
  generator) would exercise far more boundary conditions than the current five hand-written path
  cases in `calendar_location_test.dart`. Tag with `tags: 'glados'`.

- [ ] **[MED] `calculateClampedIndex(...)` in `lib/beamer/beamer_app.dart` (line 83) —
  clamp invariants.**
  This pure function has clear algebraic properties:
  - The result is always `>= 0`.
  - The result is always `<= itemCount - 1` for the computed `itemCount`.
  - `rawIndex == 0` with any flag combination must always return `0`.
  A Glados test with `Glados3<int, bool, bool>` (rawIndex × two flag booleans, then varying the
  remaining flags) would replace the five hand-rolled unit tests at lines 447–505 and cover the
  full 2⁴ × ℤ input space. Tag with `tags: 'glados'`.

## Test execution speed opportunities

- [ ] **[HIGH] `test/beamer/not_logged_in_toast_test.dart` — 10 `pumpAndSettle` calls with
  10 s default timeout each.**
  Each `pumpAndSettle()` at lines 308, 317, 353, 427, 435, 510, 514, 536, 557, 560 carries up
  to 10 s timeout. In a healthy run they complete quickly, but if any animation in the subtree
  stalls (e.g. a snack-bar slide that never settles), a single test can tie up the CI shard for
  100 s. The post-frame toast path is fully synchronous once the stream event and
  `WidgetsBinding.addPostFrameCallback` flush — replace each `pumpAndSettle()` with one or two
  `tester.pump()` calls. Estimated saving on the happy path: negligible per-run, but risk
  elimination for stall scenarios is significant.

- [ ] **[MED] `test/beamer/my_beamer_app_test.dart` — `pump(Duration(seconds: 1))` and
  `pump(Duration(seconds: 2))` at lines 239 and 246.**
  The `'starts agent initialization on app startup'` test advances virtual time by 3 s to verify
  that an in-progress async provider does not complete. The same assertion could be made by
  flushing microtasks once (`tester.pump()`) to confirm the loading shell is still shown, without
  advancing a full second of virtual time. The 3 s advancement itself is not costly but signals
  that the test was written without understanding of Riverpod's async provider scheduling. Rewrite
  as `await tester.pump()` (microtask flush only) both times.

- [ ] **[MED] `test/beamer/beamer_app_test.dart` — per-test `_registerAppScreenGetIt` +
  `tearDownTestGetIt` called 39 times.**
  Every widget test calls the full `setUpTestGetIt` chain, which spins up the complete GetIt
  container. The seven `'AppScreen bottom navigation style'` tests (lines 598–620) and three
  `'AppScreen mobile nav item taps'` tests (lines 1110–1147) are structurally identical apart
  from the tab index. Extracting these into a single helper that loops over the permutation table
  (already done for the for-loop style tests, but the surrounding `_registerAppScreenGetIt` is
  still repeated per iteration) would reduce 10 full container setups to 1.

- [ ] **[LOW] `test/beamer/beamer_app_test.dart` — each desktop layout test restores
  `tester.view` manually instead of using `_useViewport`.**
  Seven desktop-layout tests (lines ~739–1043) each inline:
  ```dart
  tester.view
    ..physicalSize = const Size(1280, 800)
    ..devicePixelRatio = 1.0;
  addTearDown(tester.view.resetPhysicalSize);
  addTearDown(tester.view.resetDevicePixelRatio);
  ```
  The `_useViewport` helper (defined at line 75) does exactly this. Replace the 7 × 4-line
  blocks with `_useViewport(tester, _desktopViewportSize)` to cut ~28 lines of boilerplate and
  ensure teardown is always registered. No speed impact, but reduces copy-paste drift risk.

- **Biggest single opportunity**: Split `test/beamer/beamer_app_test.dart` (2155 lines) into
  three files sharing a helper module — this alone halves the largest test file in the scope and
  makes the groups individually navigable.
