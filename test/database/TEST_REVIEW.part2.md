## Generative (Glados) testing opportunities

The existing Glados coverage is strong and well-targeted. Below are genuine gaps where non-trivial pure logic exists and property tests would add value.

- [x] **[MED] `SyncDatabase.getMissingEntriesWithLimits` — `maxPerHost` post-filter logic (lines 1936–1968)** — the per-host capping is pure Dart: a `Map<String, List<...>>` accumulator that emits at most `maxPerHost` rows per host key, then applies `offset`/`limit` to the result. A Glados test exercising arbitrary (rows: List, maxPerHost: int?, offset: int, limit: int) combinations would verify that: (a) no host exceeds `maxPerHost` in the result, (b) the total result is ≤ `limit`, (c) all returned rows satisfy the `status IN (1,2)` predicate, and (d) `offset` skips exactly the right prefix. This pure post-processing belongs in a unit test rather than an integration test.

- [x] **[MED] `SyncDatabase._retireInPages` (lines 2090–2152)** — the pagination loop contains the same "loop terminates when last batch < pageSize" pattern as `pruneSentOutboxItemsChunked`, which already has a Glados property test (`ExploreConfig(numRuns: 80)` at line 2057). Adding a Glados property test for `retireExhaustedRequestedEntries` and `retireAgedOutRequestedEntries` that varies `(rows, pageSize, maxRequestCount, grace)` would mirror the prune-scenario test and guard against off-by-one regressions in the inner `_retireInPages` loop.

- [x] **[LOW] `SyncDatabase.batchIncrementRequestCounts` (line 1550)** — the method takes a `List<({String hostId, int counter})>` and issues a bulk UPDATE. A Glados test with arbitrary (rows in DB, update targets, some targets absent) would verify: (a) `request_count` increments by exactly 1 for every matched row, (b) rows not in the target list are untouched, (c) absent (hostId, counter) pairs are silently ignored. This is pure DB-state logic with no I/O dependency other than an in-memory DB.
  **RESOLVED:** Added a `glados`-tagged property (`ExploreConfig(numRuns: 120)`) in `test/database/sync_db_backfill_test.dart` (`batchIncrementRequestCounts property` group). It seeds a de-duplicated set of `sync_sequence_log` rows over hosts `host-0..host-2` × counters `1..5` with arbitrary initial `request_count` and a status drawn from {`missing`, `requested`, `received`, `backfilled`}, then calls the method with a de-duplicated mix of present and absent `(hostId, counter)` targets (target host range extends to `host-3` and target counter range to `6` so some targets address rows that were never inserted). The group opens ONE in-memory `SyncDatabase` in `setUp` and `DELETE FROM sync_sequence_log` / `sync_sequence_watermarks` at the start of each run. **Oracle:** a row is incremented iff it is targeted AND its status is `missing` or `requested` (mirroring the SQL guard `WHERE host_id = ? AND counter = ? AND status IN (missing, requested)`); a matched row gets `request_count + 1`, `status = requested`, and a non-null `last_requested_at`; every other row (non-targeted, or targeted-but-ineligible) keeps its original `request_count`, status, and null `last_requested_at`; absent targets insert nothing (the full table key-set must equal the seeded key-set) and never throw.

- [x] **[LOW] `SyncDatabase.getCountersForHostInRange` (line 1708)** — pure range-filtering SQL. A Glados test varying (counters in DB, rangeStart, rangeEnd) would verify the returned set is always the correct mathematical intersection.
  **RESOLVED:** Added a `glados`-tagged property (`ExploreConfig(numRuns: 120)`) in `test/database/sync_db_sequence_test.dart` (`getCountersForHostInRange property` group). It seeds arbitrary counters (`0..11`) for `host-under-test` plus decoy counters in the same value space for `other-host`, then queries with arbitrary `(rangeStart, rangeEnd)` in `0..11` (which may be inverted). The group opens ONE in-memory `SyncDatabase` in `setUp` and truncates `sync_sequence_log` / `sync_sequence_watermarks` at the start of each run. **Oracle (range inclusivity confirmed from the SQL):** the method's WHERE is `counter >= startCounter AND counter <= endCounter` — **inclusive on both ends** — with an explicit early `return <int>{}` when `endCounter < startCounter`. So the expected result is `{}` when `rangeEnd < rangeStart`, otherwise `{c ∈ hostCounters : rangeStart <= c <= rangeEnd}`; the decoy host's counters are asserted never to appear in the result.

- The following are explicitly **NOT** good Glados candidates (real DB I/O, no pure logic to generalize):
  - Migration tests — each test opens a real file-backed DB with a specific baseline schema; property testing can't improve these.
  - Coalescing tests — correctness depends on microtask scheduling; Glados can't meaningfully vary the concurrency model.
  - `labels_performance_test.dart` — timing benchmark; not a property.

## Test execution speed opportunities

- [x] **[HIGH] `sync_db_test.dart` — 27 independent `setUp/tearDown` cycles each opening a `SyncDatabase(inMemoryDatabase: true)`** — Drift's in-memory DB must run all 24 migration steps on each `open`. With 27 independent groups, that is 27 schema-creation passes. Groups whose tests are entirely read/write-independent (e.g., `HostActivity Tests` at line 2927, `getBackfillStats Tests` at line 3004, `hasActionableEntries Tests` at line 3200, `getMissingEntriesWithLimits Tests` at line 3378, `Batch Operations Tests` at line 3606, `getRequestedEntries Tests` at line 4121) could share one `SyncDatabase` opened in `setUpAll`/`tearDownAll` — each test's `setUp` would truncate only the tables it needs. Estimated reduction: ~20 schema-creation passes → potentially several seconds off the per-file run.
  - **RESOLVED:** (assessed — keep per-group isolation) duplicate of the quality-section entry above; same disposition: per-group fresh DB is the isolation boundary, a shared `setUpAll` DB trades guaranteed isolation for marginal speed and risks single-isolate-bundle contamination. Kept deliberately.

- [x] **[HIGH] Glados tests each open a new `SyncDatabase` per run** — the five Glados tests (`numRuns` = 40, 80, 80, 100, 180) collectively open `480 + safety` in-memory databases, each running the full migration ladder. This is the single largest speed contributor in the file. The `outboxClaimScenario` (line 1529, 40 runs) and `pruneScenario` (line 2057, 80 runs) tests open a DB inside a `try/finally` block, seed data, execute the method, and close. For these tests the DB could be opened once in `setUpAll` (or via a shared fixture) and truncated between runs using `DELETE FROM outbox` — drastically reducing schema-init overhead. The `sequenceLifecycleScenario` (180 runs, line 6983) is the most expensive: each run opens, seeds up to 14 rows, calls one method, asserts, and closes.
  - **RESOLVED:** (assessed — keep per-run isolation) these are the crown-jewel **non-deterministic** sync properties (outbox claim, prune, sequence lifecycle). A per-run fresh DB guarantees each generated input starts from a clean schema; sharing a DB with `DELETE FROM` between runs would only be correct if *every* table each property touches is truncated every run — and because Glados explores a different input each seed, a missed truncation manifests as a rare, seed-dependent failure that is nearly impossible to reproduce or trust. Trading the strongest isolation guarantee for speed on the safety-critical CRDT/sequence properties is not worth it; the per-run open is kept deliberately. (The `numRuns` were already trimmed to the ≤180 band, which is the safe lever for this file's Glados cost.)

- [x] **[HIGH] `sync_db_migration_test.dart` — every test opens a file-backed `SyncDatabase` (real disk I/O)** — each test creates a temp `.db` file, seeds it with raw `sqlite3`, disposes, then opens it with Drift to trigger migration. Fourteen tests × (file create + seed + Drift open + migration) is inherently slow. This cannot be eliminated (migration tests need file-backed DBs), but seeding via shared `_seedVN` helpers (rather than repeating 50+ lines of DDL per test) reduces the test-body overhead and makes failures easier to diagnose.
  **RESOLVED (the seeding-helper half):** The file-backed I/O itself is unavoidable and untouched (temp `.db` create → raw seed → dispose → Drift open → migration kept exactly as-is for every test). The actionable part — replacing the 50+-line repeated DDL blocks with shared seed helpers — is done; see the two RESOLVED notes above. Test bodies now seed via `_createOutboxVN` / `_createSyncSequenceLogVN` / `_createHostActivity` / `_createInboundEventQueueVN` / `_createQueueMarkers`, shrinking each test body and localising failures.
