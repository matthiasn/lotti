## Test execution speed opportunities

- [ ] **[HIGH]** `test/features/agents/database/agent_repository_test.dart` — The shared `setUp` / `tearDown` pair at lines 2050–2057 opens (`AgentDatabase(inMemoryDatabase: true, background: false)`) and closes a Drift in-memory database for **every one of the 199 tests** in the file. Drift in-memory DB construction is inexpensive but not free; the per-test open/close cycle adds up across the large test suite. Groups that only read data and never mutate shared state (e.g. the Glados property tests at lines 3079, 3440, 4147, 4525, 5411, 6077, 6286) could share a single `setUpAll`-scoped database instance, reducing DB spin-up cost by roughly 80 %  of those tests. The easiest first step is to audit which `group()` blocks do not depend on a clean DB for isolation, and move those to a class-level `setUpAll`/`tearDownAll` pattern.

- [ ] **[HIGH]** `test/features/agents/database/agent_repository_change_set_test.dart`, `agent_repository_evolution_test.dart`, `agent_repository_soul_test.dart` — Each satellite test file independently opens its own `AgentDatabase` in `setUp` and closes it in `tearDown` (lines 392–398, 541–547, 529–536 respectively). The duplicated DB lifecycle (noted in the DRY section above) also doubles the DB initialization cost across three files. If the `_TestBench` shared-fixture refactor recommended in the quality section is adopted, initializing once per suite (or at least once per `group`) via `setUpAll` would halve the spin-up count.

- [ ] **[MED]** `test/features/agents/database/agent_repository_test.dart:3079,3440,4147,4525,5411,6077` — Seven Glados property tests use `glados.ExploreConfig(numRuns: 80)` each; they run against a live in-memory `AgentDatabase`. Because each Glados iteration performs a full async DB round-trip (insert + query), these tests are significantly slower than pure in-memory Glados properties. Consider extracting the pure query-logic into a function that takes a list of entities (rather than a live `AgentRepository`) so the majority of iterations can run without DB I/O. Only one representative DB-integration test per property would then be needed.

- [ ] **[LOW]** `test/features/agents/database/agent_db_conversions_test.dart:178,317` — Two Glados properties (`numRuns: 120` and `numRuns: 150`) open and work with a shared `AgentDatabase` fixture. These are serialization round-trip tests (`toEntityCompanion` + `fromEntityRow`) that do not need a real database at all — they only need the companion and row types, which can be constructed in memory without Drift's I/O layer. Replacing the database call with direct constructor/factory calls would remove the async overhead from every Glados iteration and speed up these two properties considerably.

## Summary

- **Oversized impl files:** 1 (`agent_repository.dart`, 1 768 lines — ~77% over the 1 000-line
  limit, with five clear extraction seams).
- **Oversized test files:** 5 (`agent_repository_test.dart` at 7 412 lines is critical;
  `agent_database_test.dart` at 1 563, `agent_db_conversions_test.dart` at 1 449,
  `agent_repository_evolution_test.dart` at 1 504, `agent_repository_change_set_test.dart`
  at 1 358).
- **Weak / infrastructure tests:** 4 actionable items (inline `registerFallbackValue` already
  in central file; generator preamble dwarfing test logic in main test file; duplicated
  setUp boilerplate across three satellite test files; assertion depth thin in a few
  roundtrip groups).
- **Glados candidates:** 4 genuine opportunities (`_effectiveLedgerStatus` state-machine,
  `AgentDbConversions` entity round-trip property, `_sqliteInClauseChunks` iterator
  invariants, `_extractVectorClock` parser).
- **Biggest single opportunity:** Splitting `agent_repository_test.dart` (7 412 lines) and
  `agent_repository.dart` (1 768 lines) along their existing logical seams would dramatically
  improve navigability, reduce CI re-run cost on targeted changes, and bring both files into
  compliance with the repository's size guidelines. The satellite test files (`change_set`,
  `evolution`, `soul`) already prove the split approach works — it just needs to be applied
  consistently to the remaining groups in the monolithic file.
