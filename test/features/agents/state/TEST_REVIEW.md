## Generative (Glados) testing opportunities

- [x] **[HIGH]** `aggregateByModel` in `lib/features/agents/state/agent_query_providers.dart:359` — Pure function with non-trivial fold logic: accumulates token counts per `modelId`, handles `null` optional fields (`inputTokens`, `outputTokens`, `thoughtsTokens`, `cachedInputTokens`), and sorts descending by `totalTokens`. Key invariants: (1) `sum(result.map(r => r.totalTokens)) == sum(records.map(totalTokens))` (totals preserved), (2) result is sorted descending, (3) wakeCount equals records count per model. No direct tests exist; only tested indirectly via provider tests. Ideal Glados candidate with `WakeTokenUsageEntity` generator.

- [x] **[HIGH]** `_deduplicateChangeSets` in `lib/features/agents/state/change_set_providers.dart:37` — Pure function. Invariants amenable to Glados: (1) output length ≤ input length, (2) fully-resolved sets (no pending items) are never collapsed with each other, (3) for any given fingerprint key, the newest `createdAt` wins, (4) idempotent (running twice produces the same result). No direct tests exist.

- [ ] **[MED]** `_buildDailyUsage` in `lib/features/agents/state/token_stats_providers.dart:243` — Pure function building N-day buckets. Invariants: (1) result always has exactly `chartDays` entries, (2) last entry is always `isToday == true`, (3) sum of all `totalTokens` equals sum of input records' tokens, (4) `tokensByTimeOfDay` for `isToday` always equals `totalTokens`. Tested only via `dailyTokenUsageProvider`. Glados with a generator over `(chartDays ∈ [1,30], List<WakeTokenUsageEntity>)`.

- [ ] **[MED]** `_keepLatestRunningTimerUpdate` in `lib/features/agents/state/unified_suggestion_providers.dart:129` — Pure dedup function. Key invariants: (1) non-timer-update items are always passed through unchanged, (2) for each timer ID, at most one update survives, (3) the surviving update has the newest `changeSet.createdAt` (or highest `itemIndex` on ties), (4) output length ≤ input length. The existing Glados test in `unified_suggestion_providers_test.dart` tests the provider as a whole, not this helper in isolation.

- [ ] **[LOW]** `_buildDailyWakeCounts` in `lib/features/agents/state/ritual_review_providers.dart:216` — Pure date-bucketing helper. Invariant: output always spans exactly `(today - chartStart).inDays + 1` days; all dates fall within `[chartStart, today]`. Small but extractable from provider tests.

## Test execution speed opportunities

- [x] **[HIGH]** `test/features/agents/state/agent_providers_test.dart:182,225,235` — Three `await Future<void>.delayed(Duration.zero)` calls violate the fake-time policy (already flagged in the quality section above). Each call yields to the real event loop, adding non-deterministic delay to those tests. Replacing them with `await pumpEventQueue()` (already used elsewhere in the file) removes the real-timer yield and makes the tests deterministic and faster under `fakeAsync`.

- [ ] **[MED]** `test/features/agents/state/agent_pending_wake_providers_test.dart:772,846,1495` — Three Glados properties (`numRuns: 180`, `numRuns: 160`, `numRuns: 180`). The outer `setUp` at the top of the file re-creates the full `ProviderContainer` on every test and every Glados iteration. `ProviderContainer` construction with multiple overrides is non-trivial. Identify which Glados iterations require isolated container state vs. those that only read derived provider values; the latter can share a single container initialized in `setUpAll`.

- [ ] **[MED]** `test/features/agents/state/unified_suggestion_providers_test.dart:448` — Glados property at `numRuns: 180`. The Glados scenario class at the top of the file (~200 lines) is re-evaluated per iteration but is stateless; the mock wiring inside `setUp` is rebuilt for each of the 180 runs. Extracting a shared container/mock construction to `setUpAll` (with `reset(mock)` between iterations) would reduce the per-run overhead.

- [ ] **[LOW]** `test/features/agents/state/agent_providers_test.dart` — Multiple inner `setUp` blocks at lines 91, 1289, 1382 each call `getIt.reset()` + re-register singletons. `getIt.reset()` is a global operation that rebuilds the service locator; when these groups run in the batched `very_good test` suite, each reset is a potential source of inter-group overhead. Consolidating GetIt usage under `setUpTestGetIt()`/`tearDownTestGetIt()` (as recommended in the quality section) would also reduce per-test reset cost.
