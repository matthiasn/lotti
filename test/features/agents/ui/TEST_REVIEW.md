## Test execution speed opportunities

- [ ] **[HIGH]** `test/features/agents/ui/agent_template_detail_page_test.dart` — 57 calls
  to `pumpAndSettle` across 49 tests. Each navigates a `NestedScrollView` +
  `TabBarView` layout. The 12 calls to `scrollUntilVisible` each call
  `pumpAndSettle` internally. Tests that navigate to the Stats tab (e.g.
  `'rollback calls rollbackToVersion'`, `'shows error when deleting template with
  instances'`) call `pumpAndSettle` 4–5 times per test body. **Impact**: each
  `pumpAndSettle` has a 100 ms default idle timeout; 57 × ~80 ms ≈ 4.5 s of
  additional latency per run of this file.
  Mitigation: for tests that only need one state change, replace the 3rd and 4th
  `pumpAndSettle` with `tester.pump(const Duration(milliseconds: 100))` after the
  async action completes.

- [ ] **[HIGH]** `test/features/agents/ui/agent_settings_page_test.dart` — 57 calls to
  `pumpAndSettle` for 34 tests in a file that wraps `AgentSettingsPage`, which
  embeds `TokenStatsTab`, `AgentInstancesList`, and `SidebarWakeQueue`. Each
  full-page pump-and-settle resolves multiple async providers. **Impact**: up to
  5–6 s per run. Consider caching the built widget in a `setUpAll`/`setUp` block
  using a shared `ProviderContainer` for tests that don't mutate state between
  assertions.

- [ ] **[HIGH]** `test/features/agents/ui/agent_soul_detail_page_test.dart` — 74 calls to
  `pumpAndSettle` across 38 tests. Many edit-mode tests navigate tabs and scroll
  to find UI elements, contributing 2–4 `pumpAndSettle` calls each. Replacing
  intermediate navigation `pumpAndSettle` with `tester.pump(Duration(seconds: 1))`
  after tab taps would reduce idle time. **Impact**: ~5–6 s per run.

- [ ] **[MED]** `test/features/agents/ui/sidebar_wake_queue_test.dart` — 33 calls to
  `pumpAndSettle`. Most tests build a fresh `makeTestableWidgetWithScaffold` with
  a `DesignSystemTheme.dark()` theme. Theme build is constant — if a `setUpAll`
  pre-built scaffold were possible (it's not with per-test provider state), it
  would save ~200 ms. Simpler win: the tests at lines 338–390 that check
  cancellation/snooze call `pumpAndSettle` twice after each button tap; one
  `pump(milliseconds: 300)` would suffice after the await-chain resolves.
  **Impact**: ~2 s per run.

- [ ] **[MED]** `test/features/agents/ui/agent_activity_log_test.dart` — 9 calls to
  `pumpAndSettle` in 39 tests. The `buildSubject` factory reconstructs a full
  `makeTestableWidgetWithScaffold` per test including `GptMarkdown` rendering
  setup. The file is large (1095 lines) but most pumps are necessary given the
  async provider resolution. Low priority.

- [ ] **[MED]** `test/features/agents/ui/agent_template_detail_page_test.dart:1181` —
  manual pump loop at lines 1187–1190 (`for (var i = 0; i < 10; i++) { await
  tester.pump(Duration(milliseconds: 100)); }`) to drive tab animation. This
  pattern is fragile and slow (10 × 100 ms = 1 s). Replace with
  `tester.pump(const Duration(milliseconds: 400))` (a single pump past the
  animation duration) or `tester.pumpAndSettle(const Duration(seconds: 2))` if
  animation never terminates; investigate whether the loading completer can be
  completed before the pump to allow `pumpAndSettle`.

- [ ] **[LOW]** `test/features/agents/ui/token_stats_tab_test.dart:752` and `:803` —
  both `tester.view.physicalSize = const Size(1200, 2400)` + `addTearDown(tester.view.reset)`
  blocks add overhead. Since the underlying widget does not respond to screen
  dimensions for the test assertions (only bar height matters, not absolute pixel
  size), a `SizedBox(width: 800, height: 1200)` wrapper around `TokenStatsTab`
  would be lighter and more deterministic.

## Summary

- **3 oversized impl files** (`token_stats_tab.dart` 1212 lines, `agent_template_detail_page.dart`
  831, `agent_soul_detail_page.dart` 771) each exceed the 1000/~500-line guidance.
- **3 oversized test files** (`agent_template_detail_page_test.dart` 2391,
  `agent_soul_detail_page_test.dart` 1602, `agent_settings_page_test.dart` 1313) — the
  template detail test at 2391 lines is the most severe.
- **Weak tests**: 2 significant quality issues (38-copy-paste `getAgentsForTemplate`
  stub; 2 raw `ProviderContainer`+`MaterialApp` bypasses); otherwise test assertions
  are mostly meaningful and behavior-driven.
- **0 Glados candidates** — the UI subdirectory is expected to have minimal generative
  candidates; `report_content_parser_test.dart` and `agent_date_format_test.dart`
  already have Glados coverage.
- **3 coverage gaps**: `agent_badge_widgets.dart` (no test), `agent_palette.dart`
  (no test, function gap), `_formatDuration` minutes-only branch not asserted, and
  `AiSummaryCard` root shell not tested at the root level.
- **5 speed wins**: Template detail (57 `pumpAndSettle`s, ~4.5 s), settings page
  (57, ~5 s), soul detail (74, ~5–6 s) are the top targets; the 10-pump loop at
  template detail line 1187 is also an easy fix.
- **Biggest opportunity**: Extract `_stubGetAgentsEmpty(mockTemplateService)` helper
  and `buildEditSubjectWithContainer` in `agent_template_detail_page_test.dart`
  to collapse ~140 lines of copy-paste mock setup and 2 raw `MaterialApp` bypasses,
  reducing the test file by ~350 lines and making the intent of each test clearer.
