- [ ] **[MED]** `lib/…/task_agent_strategy.dart`, `_parseToolArguments` (lines 757–829): the JSON-recovery parser is a pure function with well-defined algebraic invariants (idempotence: parsing an already-valid object returns the same map; round-trip: `jsonEncode(map)` always parses back cleanly; recovery: a map wrapped in markdown fences always round-trips; balanced-brace invariant: the first balanced JSON object in any string is always returned). Currently covered by 7 static example tests. **Add a Glados property** for the fence-recovery and balanced-brace paths using `any.string` narrowed to ASCII printable characters (excluding control chars), verifying that for any valid JSON object `o`, `_parseToolArguments('```json\n${jsonEncode(o)}\n```')` returns `o`. - [ ] **[MED]** `lib/…/change_proposal_filter.dart`, `checkTaskMetadataRedundancy` for the `update_task_estimate` tool: unlike the string-keyed tools (priority/status/title/language), the estimate path compares `int` equality without normalization. - [ ] **[LOW]** `lib/…/change_set_builder.dart`, `_deduplicateItems` (line 608–621): a pure, static function with a clear algebraic invariant — if every item in `proposed` appears in `existing`, `_deduplicateItems` returns an empty list; if none do, it returns `proposed` unchanged; the set union property holds for `rejectedFingerprints`. ## Test execution speed opportunities

- [x] **[HIGH]** `task_agent_workflow_test.dart` — **GetIt is re-initialized on every test** via `setUp(() async { ... await setUpTestGetIt(...) })` at line 140 for a file with 135 tests. GetIt setup involves registering `PersistenceLogic` and `TimeService` singletons. Because these registrations are identical for every test, moving the initialization to a **`setUpAll`/`tearDownAll` pair** (with individual tests only resetting mock return values) would cut the 135× repeated registration overhead significantly. Estimated saving: ~50–100 ms on a warm-cache run; potentially 0.5–2 s on a cold run.

- [ ] **[HIGH]** `task_agent_workflow_test.dart` — the `build → notification fire-and-forget` path (lines 2386, 2480) calls `await pumpEventQueue()` to drain the unawaited embedding pipeline. `pumpEventQueue()` is not a fake-time primitive — it calls `Future.delayed(Duration.zero)` internally multiple times and is expensive relative to `flushMicrotasks`. The fire-and-forget embedding unawaited future could instead be collected into a `Completer` in the test variant and awaited directly, eliminating the open-time drain. **[LOW]** impact if only 2 occurrences, but worth noting for any future additions of fire-and-forget paths.

- [ ] **[MED]** `task_agent_workflow_test.dart` `group('tool handler dispatch with real Task')` (lines 4342–5107): the nested `setUp` at line 4342 calls `stubFullExecutePath` unconditionally before every one of the ~40 tests in the group, setting up ~10 `when(...)` stubs each time. A `setUpAll` for the stubs that don't change between tests (provider/model configs) and a lighter per-test `setUp` for the task-entity stub would cut the per-test mock setup cost. Estimated saving: 30–60 `when(...)` calls saved per test (20–40 tests), ~1–2 s total.

- [ ] **[MED]** `change_set_builder_test.dart`, `build → notification fire-and-forget` group (lines 1756–1972): each test calls `await setUpTestGetIt(...)` in `setUp` and `tearDownTestGetIt()` in `tearDown`, restarting the full GetIt container for each test. Since all tests in this group need the same `NotificationRepository` + `JournalDb` registrations, **use `setUpAll`/`tearDownAll`** with shared mock instances that are reset (via `reset(mock)`) between tests, avoiding repeated GetIt initialization.

- [ ] **[LOW]** `change_set_builder_test.dart` line 3017: `numRuns: 220` on the `'follow-up task placeholder stability under padding/casing'` Glados property exceeds the repo-stated ceiling of 200 (see `test/README.md`). **Reduce to 180** — the deterministic-placeholder invariant is simple and doesn't require that many runs to surface shrinkage failures.

- [ ] **[LOW]** `task_agent_workflow_test.dart` group `'_buildUserMessage context'` lines 2663–3580: the `executeAndCaptureMessage` local helper (line 2666) stubs ~12 `when(...)` calls on every invocation, but many tests within the group call it with the same defaults and only vary a single parameter (e.g. `lastReport`, `observations`, `triggerTokens`). Extracting the common stubs into a local `setUp` and letting `executeAndCaptureMessage` apply only the per-test overrides would reduce mock-setup CPU inside the helper.

## Summary

- **Oversized implementation files:** 3 (`task_agent_workflow.dart` at 2134 lines, `task_agent_strategy.dart` at 1203 lines, `change_set_builder.dart` at 1121 lines)
- **Oversized test files:** 4 (`task_agent_workflow_test.dart` at 6057 lines, `task_agent_strategy_test.dart` at 4092 lines, `change_set_builder_test.dart` at 4040 lines, `task_tool_dispatcher_test.dart` at 1179 lines)
- **Weak / thin-assertion tests:** ~15 tests in `tool handler dispatch with real Task` that assert only `result.success == isTrue` without verifying LLM tool-response content or handler-level side effects; several error-path tests in `task_tool_dispatcher_test.dart` that assert only `result.success == isFalse`
- **Glados candidates:** 3 genuine opportunities (`_parseToolArguments` round-trip + fence-recovery property; `checkTaskMetadataRedundancy` for the estimate-int branch; `_deduplicateItems` algebraic invariant)
- **Speed wins:** 5 opportunities (GetIt re-init 135× in workflow setUp → move to setUpAll; `pumpEventQueue` in fire-and-forget tests; over-stubbed `tool handler dispatch` setUp; GetIt restart per test in notification group; `numRuns: 220` → 180)
- **Biggest opportunity:** Split `task_agent_workflow_test.dart` (6057 lines) into 6 focused files aligned with the existing group structure — this single change would reduce per-test context load, make CI shard assignment more balanced, and enable the GetIt `setUpAll` consolidation that yields the largest speed win.
