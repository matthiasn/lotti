## File size / split opportunities

- [ ] **[HIGH]** `test/…/template_evolution_workflow_test.dart` (5527 lines, 104 tests) is the largest test file in the entire Part-2 scope — almost 4× the recommended ≤1400-line soft target. The file covers at least six distinct functional domains:
  - `startSession` / `sendMessage` (lines ~167–500)
  - `approveProposal` / `rejectProposal` (lines ~617–1216)
  - `approveSoulProposal` / `rejectSoulProposal` (lines ~1219–1700)
  - `abandonSession` / `getActiveSessionForTemplate` (lines ~1701–1996)
  - `approveProposal` idempotency (lines ~1998–2550)
  - `startSoulSession` / `completeSoulSession` (lines ~3466–5527)
  Each of these could become its own `*_test.dart` file that mirrors a natural seam. Mirror-splitting is explicitly required by the one-test-file-per-source-file rule only when source files are split, but the sheer size makes navigation and maintenance painful. Suggested split: `template_evolution_workflow_session_test.dart` (start/send/abandon) + `template_evolution_workflow_proposal_test.dart` (approve/reject) + `template_evolution_workflow_soul_session_test.dart` (soul start/complete).

- [x] **[HIGH]** `lib/…/template_evolution_workflow.dart` (1437 lines) is the largest implementation file in scope and exceeds the 1000-line guidance. Clear split seam: `ActiveEvolutionSession` + `TemplateEvolutionWorkflow`'s template-session methods (`startSession`, `sendMessage`, `approveProposal`, `rejectProposal`, `abandonSession`) vs. the standalone soul-session methods (`startSoulSession`, `completeSoulSession`, `approveSoulProposal`, `rejectSoulProposal`). Those could move to a `SoulEvolutionWorkflow` class without touching the template side. **RESOLVED (adapted):** done as a `part`-file extension `soul_evolution_workflow.dart` (457 lines) instead of a separate class — the soul methods share the workflow's private session map, and `MockTemplateEvolutionWorkflow` is stubbed across the evolution chat-state suites, so the class keeps thin delegators for the four soul-session publics; `template_evolution_workflow.dart` is down to 1 010 lines.

- [ ] **[HIGH]** `lib/…/project_agent_workflow.dart` (1137 lines) exceeds the 1000-line threshold. Natural seam: the `execute` orchestration (lines 96–553) vs. the pure prompt-building helpers `_buildSystemPrompt` / `_appendSoulPersonality` / `_buildUserMessage` / `_writeProjectContext` (lines 651–918) vs. the data-loading helpers `_buildLinkedTasksContext` / `_resolveObservationPayloads` (lines 955–1076). Extracting the prompt-builder group into a `ProjectAgentContextBuilder` (mirroring the pattern of `EvolutionContextBuilder`) would cut `project_agent_workflow.dart` below 700 lines and open its pure-function group to Glados testing.

The test contains deeply nested `setUp` blocks (line 376 has a second-level `setUp` that re-stubs 8 mocks on top of a 60-line outer `setUp`). ## Generative (Glados) testing opportunities

The files below contain non-trivial pure logic with structured inputs that are currently tested only with handcrafted concrete examples. These are genuine Glados candidates.

- [x] **[HIGH]** `lib/…/project_tool_dispatcher.dart` — `_parseProjectStatus(String, {reason, now})` (line 332) and `_isSameSemanticStatus(ProjectStatus, ProjectStatus)` (line 379) together form a round-trip property: for any input string that maps to a non-null status, `_isSameSemanticStatus(result, result)` must return `true`, and `_isSameSemanticStatus(result, differentVariant)` must return `false`. Currently the test file at lines 636–668 covers 5 alias strings as individual concrete tests. A Glados property over `any.choose(allAliases)` combined with the semantic-equality law would cover the full mapping. Similarly `_parseTaskPriority` (line 319) has a normalise-then-match pattern amenable to a round-trip property: `_parseTaskPriority(priority.canonicalString)?.name == priority.name` for all valid priorities.

- [ ] **[MED]** `lib/…/evolution_context_builder.dart` — `_writeSeedChangelog` (line 325) filters `seedDirectiveChangelog` by `kind` and date. The date boundary logic — a version created on day X should include changelog entries on day X but exclude entries on day X−1 — is pure and currently only exercised by a couple of static tests. A Glados property over `(AgentTemplateKind, DateTime)` pairs combined with a synthetic changelog list would thoroughly cover the `!e.dateTime.isBefore(versionDate)` boundary.

- [ ] **[MED]** `lib/…/evolution_strategy.dart` — `_readStringArg(Map<String, dynamic>, String)` (line 384) tolerates non-String JSON values by returning `''`. A Glados property `_readStringArg({'key': nonString}, 'key') == ''` for arbitrary non-String values (int, bool, List, Map, null) would ensure the defensive fallback is exhaustive.

- [ ] **[MED]** `lib/…/project_agent_strategy.dart` — `_parseToolArguments(String)` (line 345) has two code paths: direct JSON and markdown-wrapped JSON. The markdown path is completely untested. A Glados property over arbitrary JSON-serialisable maps would verify `_parseToolArguments(jsonEncode(m)) == m` (direct path) and `_parseToolArguments('```json\n${jsonEncode(m)}\n```') == m` (markdown path). This function is public-ish (no underscore restriction in tests) since it is exercised via `processToolCalls`.

- [ ] **[LOW]** `lib/…/soul_evolution_context_builder.dart` — `_writeFeedback` feedback ordering (line 299): for any list of `ClassifiedFeedbackItem`, negative sentiment items must appear before positive before neutral in the output for each template group. A Glados property over generated feedback lists would be stronger than the single static test currently present.

## Test execution speed opportunities

- [x] **[HIGH]** `test/…/template_evolution_workflow_test.dart` — The file runs 104 `test()` cases sequentially as part of the standard (non-Glados) CI lane. All tests are pure unit tests that call `await workflow.startSession(...)` or `await workflow.approveProposal(...)` which are synchronous in the test double (no real I/O). However, many tests recreate a `ConversationManager` with `..initialize()..addAssistantMessage(...)` followed by `await strategy.processToolCalls(...)` as inline setup — sometimes 4–5 of these per test. Extracting these into shared helpers (see Test Quality section) would not reduce correctness but would reduce the per-test setup footprint and make CI shard balancing easier. **RESOLVED (dominant shapes extracted):** `_stubCreateVersion` (12 sites) and `_strategyWithProposal` (8 sites) now carry the duplicated setup; the remaining inline ConversationManager setups are bespoke per-test variants (different ids/tool-call shapes) where a forced shared helper would obscure intent.

- [ ] **[MED]** `test/…/project_agent_workflow_test.dart` — the outer `setUp` (lines 168–243) calls `setUpTestGetIt` with `additionalSetup` on every single test. `setUpTestGetIt` registers singletons into a shared `GetIt` instance, which can be slow when running under `very_good test` with its thread-level batching. The `PersistenceLogic` registration at line 184 is only needed by the `create_task` path inside `ProjectToolDispatcher` and should not be a global requirement for all 55 `ProjectAgentWorkflow` tests. Moving it to a nested `setUp` in only the groups that exercise the dispatcher path would reduce `setUpTestGetIt` overhead for the majority of tests.

- [ ] **[MED]** `test/…/project_agent_strategy_test.dart` (783 lines, 31 tests) — each test in the `update_project_report` group repeats `verify(mockManager.addToolResponse(...)).called(1)` as a standalone assertion. Since `mockManager` is a `MockConversationManager` (Mocktail), every `verify` call scans the full interaction log for the mock instance. Consolidating multiple `verify` calls per test into fewer, broader assertions (or using `captureMockManagerCallAt(index)`) would reduce the number of interaction-log scans.

- [ ] **[LOW]** `test/…/evolution_strategy_test.dart` — the `all EvolutionNoteKind values are accepted` test (line 202) loops over `EvolutionNoteKind.values` and calls `processToolCalls` + `addAssistantMessage` once per kind (currently 4 iterations). Each iteration constructs a `ChatCompletionMessageToolCall` and a new `Map`. This is fine now, but if `EvolutionNoteKind` grows, the loop will slow linearly. A Glados property `Glados(any.choose(EvolutionNoteKind.values)).test(...)` would run faster and shrink failures automatically.

- [ ] **[LOW]** `test/…/improver_agent_workflow_test.dart` — `stubHappyPath` (line 83) is called in 10+ tests, each of which re-stubs 6 `when(...)` expressions even for tests that override only one of them (e.g., the `feedbackCount: 2` test at line 176 stubs the full path and then relies on count to change behavior). Extracting the common stubs into `setUp` and letting individual tests override only the one they need to change would reduce total stub-call overhead.

## Summary

- **2 oversized implementation files**: `template_evolution_workflow.dart` (1437 lines) and `project_agent_workflow.dart` (1137 lines) both exceed the 1000-line target; each has a clear split seam.
- **2 oversized test files**: `template_evolution_workflow_test.dart` (5527 lines, 104 tests) and `project_agent_workflow_test.dart` (3013 lines, 55 tests) are the most critical; both need decomposition.
- **5 weak test spots**: `_createVersionIdempotent` recovery path, markdown-JSON path in `_parseToolArguments`, `getContinuationPrompt` non-null branch, `_buildMetaRitualSystemPrompt`/`_buildRitualSystemPrompt` `highPriorityProtocol` inclusion, and the `_resolveObservationPayloads` exception path are all untested or undertested behavior branches.
- **4 genuine Glados candidates**: `_parseProjectStatus` + `_isSameSemanticStatus` round-trips, `_writeSeedChangelog` date-boundary, `_readStringArg` non-String-type tolerance, and `_parseToolArguments` markdown-JSON path — all pure, non-I/O, with algebraic invariants that Glados would cover better than hand-rolled examples.
- **3 speed wins**: `template_evolution_workflow_test.dart`'s duplicated inline setup (HIGH), `project_agent_workflow_test.dart`'s per-test `setUpTestGetIt` over-registration (MED), `project_agent_strategy_test.dart`'s per-test `verify` scan accumulation (MED).
- **Biggest single opportunity**: Split `template_evolution_workflow_test.dart` (5527 lines) into three focused files — this alone would make the test suite navigable, reduce CI shard imbalance, and expose the duplicate mock setup that currently hides within an enormous file.
