## Generative (Glados) testing opportunities

- [ ] **[HIGH]** `lib/features/ai/repository/unified_ai_inference_repository.dart` — The JSON-object extraction loop in `processToolCalls` (lines 888–904) is a hand-rolled brace-depth parser that extracts JSON objects from a potentially concatenated argument string. This is pure string logic with clear algebraic invariants: any valid JSON produced by the AI must be extractable; any sequence of N well-formed JSON objects concatenated must yield N parsed results. A Glados property test over generated lists of non-nested JSON objects would explore edge cases (empty string, single object, whitespace between objects) far better than the single fixture in the existing tests.

- [ ] **[HIGH]** `lib/features/ai/repository/mistral_inference_repository.dart` — `_convertMessages()` (line 103) converts OpenAI-style `ChatCompletionMessage` objects to plain maps. This is a pure transformation with a straightforward round-trip invariant: role and content survive conversion. No existing Glados coverage; a generator over `(role, contentVariant)` pairs with a property `converted['role'] == msg.role.name && converted['content'] == extractedContent` would catch regressions.

- [ ] **[HIGH]** `lib/features/ai/repository/dashscope_inference_repository.dart` — The DashScope SSE parser (the `buildSseResponse` helper in the test already hints at its structure) is a bespoke framing format (`id:N\nevent:result\n:HTTP_STATUS/200\ndata:{...}\n\n`). A Glados property test for `buildSseResponse` round-trips (N events encoded then decoded must yield N content items in order) would be a genuine generative candidate.

- [ ] **[MED]** `lib/features/ai/repository/gemini_utils.dart` — `buildMultiTurnRequestBody` converts a list of `ChatCompletionMessage` objects (system, user, assistant, tool) into Gemini's `contents` format. The existing Glados test on `stripLeadingFraming` is good but doesn't cover multi-turn message conversion. A property: "every assistant message with tool calls must produce a `functionCall` part in the Gemini body" is testable with generated message sequences.

- [ ] **[MED]** `lib/features/ai/repository/gemini_utils.dart` — `buildStreamGenerateContentUri` and `buildGenerateContentUri`: the existing static examples in `gemini_utils_test.dart` cover specific cases well, but a Glados property over `(scheme, host, port, model, apiKey)` tuples asserting that scheme/host/port are always preserved and the path always follows the `/v1beta/models/<model>:<endpoint>` pattern would catch corner cases (empty scheme, no port, `models/` prefix already present, trailing slash).

- [ ] **[MED]** `lib/features/ai/repository/ollama_embedding_repository.dart` — `embed()` extracts a `Float32List` from a JSON response. A Glados property over generated `(dims, values)` pairs asserting `result.length == dims` and `result[i] ≈ values[i]` would be stronger than the current two fixture-based tests.

- [ ] **[LOW]** `lib/features/ai/repository/gemini_stream_parser.dart` — `maxBufferSize` trim-and-align logic: currently tested with a single static fixture. A Glados property over `(junkLength, objectContent, maxBufferSize)` triples asserting the object is still parsed even when the junk prefix exceeds the cap would cover the trim+align invariant.

Extracting `setUp` into a shared `_TestBench` would allow the temp directory to be created once in `setUpAll` and reused across tests (with per-test subdirectories if needed), cutting ~92 filesystem mkdir/rmdir round trips per suite run. **Estimated impact: ~200–400 ms per full suite run on CI.**

- [ ] **[HIGH]** `cloud_inference_repository_test.dart` — 10 separate `setUp` blocks each instantiate `ProviderContainer` + `CloudInferenceRepository` + `MockHttpClient`. `ProviderContainer` construction is not free (riverpod graph traversal). - [ ] **[HIGH]** `unified_ai_inference_repository_test.dart` — The mock reset in `tearDown` (lines 202–230) does multiple `getIt.isRegistered` + `getIt.unregister` calls per test in addition to `container.dispose()`. ## Summary

- **2 oversized impl files** over 1 000 lines (`unified_ai_inference_repository.dart` at 1 382, `gemini_inference_repository.dart` at 1 115); `ollama_inference_repository.dart` at 852 is approaching the limit.
- **4 oversized test files** (`unified_ai_inference_repository_test.dart` 7 306, `cloud_inference_repository_test.dart` 4 404, `gemini_inference_repository_test.dart` 3 288, `ollama_inference_repository_test.dart` 3 063) — all prime candidates for mirror-splitting alongside their impl files.
- **6+ weak test-quality issues**: inline `getIt` manipulation violating the centralized-mocks guideline in 3 files; `getIt.reset()` contamination risk; 7+ inline mock/fake class definitions that duplicate centralized mocks; no `_TestBench` in two of the three largest test files; SSE helper code copy-pasted verbatim between mistral and voxtral test files.
- **5 Glados candidates**: the `processToolCalls` brace-depth JSON extractor, Mistral `_convertMessages`, DashScope SSE framing, Gemini `buildMultiTurnRequestBody` message conversion, and `GeminiUtils.buildStreamGenerateContentUri` with varied host/model inputs — all pure functions not yet property-tested.
- **5+ coverage gaps**: `_filterAnthropicPings` error-swallowing path untested; `kMaxStreamingChars` safety-cap branch in Gemini repo unreached; silent error swallowers in `_handlePostProcessing` for image and audio; Mistral tool-response message conversion corner cases; automatic-rerun skip-when-language-set-and-response-nonempty branch.
- **3 speed wins**: real `createTempSync` per test in unified repo setup (~92 OS calls), 10 `ProviderContainer` rebuilds per test run in cloud repo (~88 constructions), and scattered inline GetIt manipulation in unified repo (~15–20 extra GetIt calls per test); collectively estimated at **600–1 200 ms of avoidable overhead per full CI run**.
- **Biggest opportunity**: Split `unified_ai_inference_repository.dart` (1 382 lines → 3–4 files) and its test (7 306 lines → 3–4 files), simultaneously replacing the ad-hoc GetIt manipulation with `setUpTestGetIt()` and extracting a `_TestBench`. This single change addresses all five axes at once for the most complex file in scope.
