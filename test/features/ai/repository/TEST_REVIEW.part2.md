## File size / split opportunities

- [x] **[HIGH]** `test/features/ai/repository/ai_input_repository_test.dart` is 2238 lines.  The file already has three sibling split files (`_language_test`, `_suppressed_ids_test`), which themselves violate the one-test-file-per-source rule (`AGENTS.md` §"One test file per source file").  The correct fix is to consolidate all four files back into a single `ai_input_repository_test.dart` and find size by extracting a `_TestBench` helper class.  Natural seams: (1) `generate()` + status mapping, (2) `buildTaskDetailsJson()` + label/suppression, (3) `buildProjectContextJsonForTask`, (4) `buildRelatedProjectTasksJson`, (5) `buildRelatedTaskDetailsJson`, (6) linked-task context, (7) provider wiring.

- [x] **[HIGH]** `test/features/ai/repository/ai_config_repository_test.dart` is 1262 lines and has two satellite files (`_cascade_delete_test.dart` at 491 lines and `_integration_test.dart` at 168 lines), both of which violate the one-test-file rule.  The main file should absorb the cascade-delete and integration groups; internal `setUp`/`tearDown` helpers per group keep it DRY without exceeding 800 lines.

- [ ] **[HIGH]** `test/features/ai/repository/mistral_transcription_repository_test.dart` is 1077 lines.  The `isMistralTranscriptionModel` group (7 boolean tests at ~10 lines each) + `transcribeAudio` group (20+ tests) need a `_stubOkStreaming()` factory function to eliminate the 15-20 lines of `MockClient.streaming` boilerplate repeated in every request-shape test.  The file itself should remain a single file but the boilerplate extraction alone would cut it to ~650 lines.

- [ ] **[MED]** `test/features/ai/repository/whisper_inference_repository_test.dart` is 815 lines.  The `MockHttpClient.post` stub is repeated verbatim 10+ times.  Extracting a `_stubHttpPost({int statusCode, Map<String, dynamic> body})` helper would cut ~300 lines.

- [ ] **[MED]** `test/features/ai/repository/ai_linked_task_context_test.dart` is 988 lines.  The first 300 lines test the pure model struct (serialization / deserialization) and can stay.  The second 688 lines test `AiInputRepository` integration behaviour; those belong in `ai_input_repository_test.dart` (see HIGH above), not a separate file.

- [ ] **[LOW]** `lib/features/ai/repository/ai_input_repository.dart` is 633 lines.  It is borderline but not over the 1000-line threshold.  The private helpers `_buildLinkedTaskContextsBatched`, `_compareRelatedProjectTasks`, and `_buildLabelTuplesFromCache` are already well-contained; no split is required now.

## Generative (Glados) testing opportunities

- [x] **[HIGH]** `AiLinkedTaskContext` JSON round-trip (`test/features/ai/repository/ai_linked_task_context_test.dart`).  The `toJson()` / `fromJson()` pair is a pure, non-I/O function with structured input (all fields can be generated).  A `Glados` property `encode → decode → encode` must produce the same output.  Fields: string `id/title/status/priority/estimate/timeSpent`, optional string `languageCode/latestSummary`, `DateTime statusSince/createdAt`, list-of-maps `labels`.  All types map to existing generators (`any.string`, `any.dateTime`, etc.).

- [ ] **[HIGH]** `TaskSummaryResolver._summaryFromReport` (`lib/features/ai/repository/task_summary_resolver.dart:137`).  The pure function picks `tldr.trim()` when non-empty else `content.trim()`, returning `null` for both-empty.  A `Glados` property can verify: for any non-whitespace `tldr`, the result equals `tldr.trim()`; for whitespace `tldr` and non-whitespace `content`, the result equals `content.trim()`; for both whitespace, the result is null.  This covers the three branches without mocking.

- [ ] **[MED]** `VectorSearchRepository._recordBestDistance` (`lib/features/ai/repository/vector_search_repository.dart:325`).  This is a pure static function.  A `Glados` property can verify: calling it twice with `d1 < d2` and then `d2 < d1` always stores the minimum; calling it N times with any sequence of distances always stores the global minimum.  The existing tests only check the two-hit case via integration.

- [ ] **[MED]** `MistralTranscriptionRepository.isMistralTranscriptionModel` (`lib/features/ai/repository/mistral_transcription_repository.dart`).  A `Glados` property can verify the prefix check: for any string not starting with `voxtral-`, the function returns `false`; for strings starting with `voxtral-`, it returns `true`.  The 6 hand-written example tests would shrink to 2 examples + 1 property.

- [ ] **[MED]** `context_bias` deduplication and truncation logic inside `MistralTranscriptionRepository.transcribeAudio` is pure set-and-slice logic applied to an input list before the HTTP request is sent.  A `Glados` property can verify: the output list never exceeds 100 items; it contains no duplicates; all original unique items up to the 100-item cap are present.

- [ ] **[LOW]** `_compareRelatedProjectTasks` in `AiInputRepository` (`lib/features/ai/repository/ai_input_repository.dart:573`).  The three-tier comparator (`updatedAt` → `dateFrom` → `createdAt` → `id`) should be an irreflexive, transitive, total order on any pair of `Task` objects.  A `Glados2` property testing transitivity (`if compare(a,b) < 0 && compare(b,c) < 0 then compare(a,c) < 0`) would be valuable.

## Test execution speed opportunities

- [x] **[HIGH]** `test/features/ai/repository/whisper_inference_repository_test.dart:354–398` — "creates unique IDs for each transcription" uses two `Future.delayed` calls (`Duration(milliseconds: 1)` and `Duration(milliseconds: 2)`) plus a mock answer that itself delays by 1 ms.  This is real I/O time on the CI runner.  The fix is to inject a clock or counter into `WhisperInferenceRepository` so unique IDs are generated deterministically rather than by wall-clock epoch.  **Estimated CI impact: +3–5 ms per run × number of shards.**

- [x] **[HIGH]** `test/features/ai/repository/ai_config_repository_test.dart:887–940` and `943–990` — two stream-error-path tests each have `await Future<void>.delayed(Duration.zero)` × 2 calls (four yields total) to propagate async errors through the broadcast controller chain.  Under `very_good test` on the CI runner these are real-time yields.  Replacing with `fakeAsync` + `flushMicrotasks()` eliminates the real-time dependency.  **Estimated CI impact: +8–20 ms per run.**

- [ ] **[MED]** `test/features/ai/repository/ai_config_repository_test.dart` — the "with in-memory database" group (line 306) opens a real SQLite in-memory database for every test in `setUp`.  There are ~15 tests in this group; each creates a fresh `AiConfigDb(inMemoryDatabase: true)` and tears it down.  Sharing one `AiConfigDb` instance across tests that do not interfere (read-only or independent table rows) would eliminate ~14 DB open/close cycles.  **Estimated CI impact: −50–150 ms per run.**

- [ ] **[MED]** `test/features/ai/repository/ai_config_repository_integration_test.dart` — spins up another `AiConfigDb(inMemoryDatabase: true)` for a single integration test.  This test's assertions (save + retrieve + watch) are already exercised in the main file's in-memory group.  Consider removing the integration file (after consolidating per the split-file finding) and relying on the main file's in-memory tests.

- [ ] **[MED]** `test/features/ai/repository/mistral_transcription_repository_test.dart` — every test creates its own `MockClient` / `MockClient.streaming` lambda inline.  The lambda creates a new `Stream.value(...)` on each call.  Extracting a file-level `_stubStreamingOk()` helper that returns a pre-built `MockClient.streaming` eliminates repeated object creation.  **Estimated CI impact: −10–30 ms per run due to fewer closure allocations.**

- [ ] **[LOW]** `test/features/ai/repository/ai_linked_task_context_test.dart` — the `AiInputRepository` integration tests in this file each call `container.read(aiInputRepositoryProvider)` in `setUp`, which constructs a fresh provider on every test.  Using a `setUpAll` with a single container (resetting only mock state) would halve provider initialization overhead.  **Estimated CI impact: −5–15 ms per run.**

## Summary

- **Oversized impl files**: 1 (`ai_input_repository.dart` at 633 lines — borderline, monitor)
- **Oversized test files**: 3 (`ai_input_repository_test.dart` 2238, `ai_config_repository_test.dart` 1262, `mistral_transcription_repository_test.dart` 1077)
- **One-test-file rule violations (split files)**: 5 files (`_language_test`, `_suppressed_ids_test`, `_cascade_delete_test`, `_integration_test`, and the misplaced `ai_linked_task_context_test.dart` groups)
- **Weak / quality tests**: 7 findings — duplicate `expect` calls, `Future.delayed` violations (× 2 files), inline mock class duplication (× 3 files), inline `MockAiConfigDb` (× 2 files), weak exception-field assertions, and untested `buildLinkedTasksJson` + `includeLogEntries: false` branches
- **Glados candidates**: 6 — `AiLinkedTaskContext` round-trip **(strongest)**, `_summaryFromReport`, `_recordBestDistance`, `isMistralTranscriptionModel` prefix check, `context_bias` dedup+truncation, `_compareRelatedProjectTasks` comparator transitivity
- **Speed wins**: 5 — real `Future.delayed` in whisper test, real `Future.delayed` in config-repo stream tests, per-test SQLite open in in-memory group (×15), `MockClient.streaming` inline closure churn, per-test provider construction in linked-task context tests
- **Biggest opportunity**: Consolidate the five split test files back to two canonical files (`ai_input_repository_test.dart`, `ai_config_repository_test.dart`), extract a `_TestBench` class per file, move `MockTaskProgressRepository` and `MockAiConfigDb` to `test/mocks/mocks.dart`, and eliminate the four `Future.delayed` fake-time violations — this removes the most structural debt and fixes the two most impactful speed regressions in a single coordinated refactor.
