## Test quality improvements

- [ ] **[MED]** `test/features/ai/service/embedding_content_extractor_test.dart` line 450‚Äì453 ("is 20" test): this test asserts `expect(kMinEmbeddingTextLength, 20)` ‚Äî a pure constant smoke test with no behavioral value. - [ ] **[MED]** `test/features/ai/service/embedding_content_extractor_test.dart` lines 284‚Äì346: the `entityType` tests assert only that the method returns the correct string constant, e.g. `expect(EmbeddingContentExtractor.entityType(entry), kEntityTypeJournalText)`. These are essentially delegation tests that provide value only because `entityType` is used in a `switch` ‚Äî there is no logic to misorder. The real behavioral gap is: what is the return value for each entity type and does it correspond to what the embedding store uses as a type discriminator? The current tests adequately cover the switch arms and are acceptable, but the `Checklist`/`JournalImage` "returns null" tests are redundant with the unsupported-type tests already in `extractText` (same entity, same conclusion). Consider consolidating entity-type tests into a single parameterized loop.

- [ ] **[LOW]** `test/features/ai/service/text_chunker_test.dart` lines 239‚Äì282: the "consecutive chunks have overlapping content" test uses a heuristic set-intersection to check overlap (`currentEndSet.intersection(nextStartSet)`). The Glados marker-based tests below (lines 483+) are far more rigorous for this invariant. The static overlap test can be simplified to: after chunking, verify that the first word of `chunk[i+1]` appears somewhere in `chunk[i]`.

- [ ] **[LOW]** `test/features/ai/service/embedding_processor_test.dart`: the stub helpers `_stubNoExistingHash`, `_stubNoCategoryId`, `_stubReplaceEntityEmbeddings`, `_stubEmbed` at lines 363‚Äì398 are defined at file level and called in `setUp`. ## Generative (Glados) testing opportunities

Existing Glados coverage is strong and well-targeted. The following are genuine gaps in pure non-trivial logic:

- [ ] **[MED]** `lib/features/ai/service/text_chunker.dart` ‚Äî `estimateTokens` (lines 77‚Äì97): the CJK path (rune-count), the long-whitespace-free-token path (character-count / 4), and the normal word path are all covered by static tests, but there is no Glados property verifying that the three paths are mutually exclusive and collectively exhaustive (i.e., for any non-empty input exactly one path fires and the result is > 0). A small generator producing ASCII words, CJK characters, and long non-CJK tokens mixed in various combinations would close this gap. All three constituent cases are pure with no I/O.

- [ ] **[LOW]** `lib/features/ai/service/embedding_content_extractor.dart` ‚Äî `extractTaskText` (lines 93‚Äì113): a property could verify: (a) result is `null` iff the total assembled string length < 20; (b) if both title and labels are non-empty the output always contains both; (c) bodyText appears after labels in the output. This is a simple pure formatter with composable invariants ‚Äî a textbook Glados candidate. Current tests cover named examples but not the threshold edge case under random inputs (e.g., a title of exactly 15 chars + labels whose combined text pushes it above 20).

The following areas do NOT qualify as Glados candidates:
- `EmbeddingProcessor.processEntity`: not pure (accesses DB, Ollama HTTP, embedding store). The Glados test in `embedding_processor_test.dart` correctly models the I/O-free logic of the decision tree; this is the right level of abstraction.
- `EmbeddingService._onBatch` / `_processNext`: orchestration logic with side effects; existing fakeAsync tests are the correct form.
- `TextChunker.chunk` as a whole: the two existing Glados tests (`chunkText` with 160 runs and `longToken` with 120 runs) provide thorough coverage of the main chunking invariants. No additional Glados tests are needed.

## Coverage / missing-behavior gaps

- [x] **[HIGH]** `lib/features/ai/service/embedding_service.dart` ‚Äî `start()` idempotency: the docstring says "Idempotent ‚Äî calling while already started is a no-op." There is no test that calls `start()` twice and verifies that the second call does not create a second subscription (and that only one listener is registered on `localUpdateStream`). A double-start followed by a notification should still produce exactly one embedding, not two. **RESOLVED:** new test 'start is idempotent: a second start adds no extra subscription' ‚Äî double `start()` followed by one notification yields exactly one `embed` call.

- [x] **[HIGH]** `lib/features/ai/service/embedding_service.dart` ‚Äî `stop()` awaits in-flight work: the code at lines 65‚Äì76 sets `_stopped = true`, cancels the subscription, clears pending IDs, and then awaits `_inFlightProcessing`. There is no test that verifies that a `stop()` called while an embedding is in flight (i.e., `_isProcessing = true`) actually waits for that embedding to complete before returning. The comment at line 966 in the test file acknowledges a `fakeAsync` limitation here. If a real-time test using `tester.runAsync` is acceptable for this one case, it would be valuable. **RESOLVED:** new real-async test 'stop waits for the in-flight embedding before returning' ‚Äî a gated `embed` Completer holds the embedding in flight, `stop()` is verified not-yet-returned after `pumpEventQueue()`, and only completes once the gate releases.

- [ ] **[MED]** `lib/features/ai/service/embedding_service.dart` ‚Äî notification filtering: `_onBatch` is tested for `notifyIrrelevantType` (image notification) and `notifyInvalidUuid`, but the case where a batch contains a relevant type token AND an invalid UUID (mixed batch) is tested only in the Glados scenario. - [ ] **[MED]** `lib/features/ai/service/embedding_processor.dart` ‚Äî category-shard move when hash unchanged but category changed: the path at lines 60‚Äì76 (`existingHash == hash && categoryChanged ‚Üí moveEntityToShard`) is exercised only through the Glados test. - [ ] **[MED]** `lib/features/ai/service/embedding_content_extractor.dart` ‚Äî `_firstTranscript` returns the **first** transcript (index 0), not the most recent. This differs from `SkillInferenceRunner._resolveEntryContent` which uses `reduce` to find the **latest** by `created`. Both are called by different code paths for different use cases (embedding vs. inference), but the divergence is undocumented and untested with multiple-transcript audio entries. A test with two transcripts verifying that `extractText` returns the first (index 0) one would document this semantic difference explicitly and guard against accidental alignment with the inference helper.

- [ ] **[LOW]** `lib/features/ai/service/text_chunker.dart` ‚Äî `chunk` with a single CJK sentence longer than `kChunkTargetTokens`: the Glados `longToken` test exercises long ASCII tokens; the CJK code path inside `_splitOnWordBoundaries` (lines 147‚Äì156) is exercised by the static "handles CJK text" test but only for a short sentence that fits in a single chunk. - [x] **[HIGH]** `test/features/ai/service/embedding_service_test.dart` ‚Äî all non-Glados tests use `fakeAsync` correctly with `sendAndProcess` (line 447) which elapse 150 ms of virtual time and flush microtasks. No real timers or real delays are present. The `fakeAsync` approach is correct and fast. No action needed here. **RESOLVED (no action needed):** the item itself confirms the fakeAsync usage is correct.

- [ ] **[MED]** `test/features/ai/service/embedding_service_test.dart` ‚Äî the Glados test at line 811 (`numRuns: 180`) iterates an entire `EmbeddingService` lifecycle (start/notify/stop) per run. Each iteration creates a new `MockEmbeddingStore`, `MockOllamaEmbeddingRepository`, `MockJournalDb`, `MockAiConfigRepository`, and `UpdateNotifications`, all inside the body of the Glados lambda (lines 817‚Äì1032). This is intentionally local to ensure isolation between Glados runs, but 180 √ó full service construction + fakeAsync execution is expensive. Extracting the per-scenario mock setup into a factory that reuses pre-built mocks (resetting stubs via `reset(mock)` between iterations) could halve construction overhead. Estimated impact: 10‚Äì20% speedup on the Glados CI lane for this test.

- [ ] **[MED]** `test/features/ai/service/embedding_processor_test.dart` ‚Äî the Glados test at line 664 (`numRuns: 140`, entity-processing scenarios) and line 1228 (`numRuns: 160`, agent-report scenarios) each construct fresh `MockJournalDb`, `MockEmbeddingStore`, and `MockOllamaEmbeddingRepository` per run inside the lambda. Same observation as above ‚Äî stub reset is cheaper than fresh allocation. Estimated impact: 5‚Äì10% per test.

- [ ] **[LOW]** `test/features/ai/service/text_chunker_test.dart` lines 300‚Äì313: the "handles single very long sentence" static test iterates through 500 words checking each against a regex in every chunk (500 √ó chunks, ~3 chunks = ~1500 regex applications). Using a `Set<String>` from all chunk tokens and asserting each marker is in the set is O(500) vs. O(500 √ó chunks). Minor improvement.
