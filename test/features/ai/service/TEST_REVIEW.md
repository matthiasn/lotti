## Generative (Glados) testing opportunities

Existing Glados coverage is strong and well-targeted. The following are genuine gaps in pure non-trivial logic:

- [x] **[MED]** `lib/features/ai/service/text_chunker.dart` — `estimateTokens` (lines 77–97): the CJK path (rune-count), the long-whitespace-free-token path (character-count / 4), and the normal word path are all covered by static tests, but there is no Glados property verifying that the three paths are mutually exclusive and collectively exhaustive (i.e., for any non-empty input exactly one path fires and the result is > 0). A small generator producing ASCII words, CJK characters, and long non-CJK tokens mixed in various combinations would close this gap. All three constituent cases are pure with no I/O.
  **RESOLVED:** done — added a record-based `estimateTokensCase` generator with the per-family oracle (multi-word, CJK single token, CJK multi-word, short single token, long non-CJK token) and a Glados property (numRuns 150) asserting each family is scored by exactly its own path formula; the cjkMultiWord and shortSingleToken families pin path exclusivity.

- [x] **[LOW]** `lib/features/ai/service/embedding_content_extractor.dart` — `extractTaskText` (lines 93–113): a property could verify: (a) result is `null` iff the total assembled string length < 20; (b) if both title and labels are non-empty the output always contains both; (c) bodyText appears after labels in the output. This is a simple pure formatter with composable invariants — a textbook Glados candidate. Current tests cover named examples but not the threshold edge case under random inputs (e.g., a title of exactly 15 chars + labels whose combined text pushes it above 20).
  **RESOLVED:** done — invariants (a) and (b) were already covered by the existing `extractTaskText — Glados properties` group (null-iff-below-threshold over random title+label inputs; result starts with title; result contains each label). Added the missing (c): a new `Glados3` property over `taskTitle × labelNameList × taskBody` asserting the body is the trailing segment and that the `Labels:` line precedes it. `taskBody` uses a distinctive, trim-stable `BODY_` prefix so position lookups are unambiguous and the trailing-trim invariant holds.

The following areas do NOT qualify as Glados candidates:
- `EmbeddingProcessor.processEntity`: not pure (accesses DB, Ollama HTTP, embedding store). The Glados test in `embedding_processor_test.dart` correctly models the I/O-free logic of the decision tree; this is the right level of abstraction.
- `EmbeddingService._onBatch` / `_processNext`: orchestration logic with side effects; existing fakeAsync tests are the correct form.
- `TextChunker.chunk` as a whole: the two existing Glados tests (`chunkText` with 160 runs and `longToken` with 120 runs) provide thorough coverage of the main chunking invariants. No additional Glados tests are needed.

## Test execution speed opportunities

- [x] **[HIGH]** `test/features/ai/service/embedding_service_test.dart` — all non-Glados tests use `fakeAsync` correctly with `sendAndProcess` (line 447) which elapse 150 ms of virtual time and flush microtasks. No real timers or real delays are present. The `fakeAsync` approach is correct and fast. No action needed here. **RESOLVED (no action needed):** the item itself confirms the fakeAsync usage is correct.

- [x] **[MED]** `test/features/ai/service/embedding_service_test.dart` — the Glados test at line 811 (`numRuns: 180`) iterates an entire `EmbeddingService` lifecycle (start/notify/stop) per run. Each iteration creates a new `MockEmbeddingStore`, `MockOllamaEmbeddingRepository`, `MockJournalDb`, `MockAiConfigRepository`, and `UpdateNotifications`, all inside the body of the Glados lambda (lines 817–1032). This is intentionally local to ensure isolation between Glados runs, but 180 × full service construction + fakeAsync execution is expensive. Extracting the per-scenario mock setup into a factory that reuses pre-built mocks (resetting stubs via `reset(mock)` between iterations) could halve construction overhead. Estimated impact: 10–20% speedup on the Glados CI lane for this test.
  **RESOLVED:** assessed, no change — per-run fresh mocks are deliberate isolation (mocktail `reset()` clears stubs AND interactions but shares instances, inviting cross-run contamination if any stub ordering leaks). mocktail mocks are plain objects with no codegen setup cost; construction is not the dominant cost next to the fakeAsync lifecycle each run drives. Keeping run isolation explicit outweighs an unverified 10–20% estimate.

- [x] **[MED]** `test/features/ai/service/embedding_processor_test.dart` — the Glados test at line 664 (`numRuns: 140`, entity-processing scenarios) and line 1228 (`numRuns: 160`, agent-report scenarios) each construct fresh `MockJournalDb`, `MockEmbeddingStore`, and `MockOllamaEmbeddingRepository` per run inside the lambda. Same observation as above — stub reset is cheaper than fresh allocation. Estimated impact: 5–10% per test.
  **RESOLVED:** assessed, no change — same rationale as the embedding-service item above: deliberate per-run isolation; `reset(mock)` saves only object allocation, which is negligible for mocktail mocks, while weakening the isolation guarantee the scenarios rely on.

- [x] **[LOW]** `test/features/ai/service/text_chunker_test.dart` lines 300–313: the "handles single very long sentence" static test iterates through 500 words checking each against a regex in every chunk (500 × chunks, ~3 chunks = ~1500 regex applications). Using a `Set<String>` from all chunk tokens and asserting each marker is in the set is O(500) vs. O(500 × chunks). Minor improvement.
  **RESOLVED:** done — the "handles single very long sentence" test now builds one `Set<String>` of all chunk tokens (splitting on whitespace yields exact tokens, so `word1` cannot match `word10`) and checks membership per marker (O(words) instead of O(words × chunks)). The regex-per-word scan was removed.

## Summary

- No implementation file in `ai/service/` is oversized; all 4 files are ≤250 lines and well-structured.
- `embedding_processor_test.dart` (1342 L) and `embedding_service_test.dart` (1218 L) are large due to heavyweight Glados scenario preamble; no structural violation but growth should be monitored.
- Biggest quality issue: `embedding_content_extractor_test.dart` contains a constant smoke test (`kMinEmbeddingTextLength == 20`) and near-duplicate entity-type tests that should be consolidated.
- 2 genuine Glados candidates: `estimateTokens` mutual-exclusivity property and `extractTaskText` threshold boundary property.
- 3 priority coverage gaps: `start()` idempotency, `stop()` while in-flight, and the `_firstTranscript` vs. latest-transcript divergence between extractor and inference runner.
- Main speed opportunity: Glados per-run object allocation in `embedding_service_test.dart` and `embedding_processor_test.dart`; reusing stubs between iterations could improve CI Glados lane by 10–20%.
