## Test execution speed opportunities

- [x] **[HIGH]** `test/features/ai/services/skill_inference_runner_test.dart` — real filesystem I/O: the `setUp` at line 537 calls `Directory.systemTemp.createTemp(...)` and most `runTranscription`/`runImageAnalysis` tests write actual bytes to disk (`File(...).writeAsBytes([...])`). This is legitimate because the runner reads audio/image bytes from disk, but ~20 happy-path and override-fallback tests each write files individually. Extracting a `setUp`-level audio file and image file (created once per `group`) and reusing paths across tests within a `group` would reduce disk I/O from ~20 `writeAsBytes` calls to 2 per group. Estimated savings: 50–200 ms per test run on slow CI disks. **RESOLVED (partially assessed):** the ten verbatim create-dir+write blocks now share `createStubAudioFile()`/`createStubImageFile()` helpers. The per-test temp directory itself stays (it is recreated in setUp by design for isolation), so the per-group single-write optimization does not apply without weakening that isolation.

- [x] **[MED]** `test/features/ai/services/skill_inference_runner_test.dart` — the two Glados tests (`promptStreamScenario` numRuns=120 and `promptGenerationScenario` numRuns=160) each create a full `ProviderContainer` + `SkillInferenceRunner` per iteration. The `_GeneratedSkillRunnerBench.create()` helper at line 179 is efficient, but the per-run container allocation (×120 and ×160) means 280 containers are created across both tests. If the bench were reusable across runs within one Glados test (resetting mock call counts between iterations), this would reduce allocation overhead. Impact: estimated 5–15% speed gain on the Glados CI lane.
  **RESOLVED (assessed, no change):** the per-iteration container IS the isolation mechanism — setUp runs once per Glados test, not per iteration, so reusing a bench across iterations would share mock call counts and provider state between generated cases (the in-suite-contamination class of bug). 280 lightweight containers cost far less than one contaminated shrink cycle.

- [x] **[LOW]** `test/features/ai/services/skill_inference_runner_test.dart` — the `stubLoggingEvent()` / `stubLoggingException()` helpers at lines 480–498 are defined at top-level and also duplicated inside `_GeneratedSkillRunnerBench` (lines 234–252). The bench helpers call out to local variables, but the top-level helpers are reconstructed fresh on every test. Moving both stubs into the bench or a shared helper would make the duplication visible and invite consolidation.
  **RESOLVED:** extracted two file-level helpers `_stubLoggingExceptionFor(MockDomainLogger)` / `_stubLoggingEventFor(MockDomainLogger)` that hold the single `when(...)` shape; both the `main()` closures (over `mockLoggingService`) and the bench methods (over its own `loggingService`) now delegate to them, so the stub definition exists once.

- [x] **[LOW]** `test/features/ai/services/auto_checklist_service_test.dart` — `getIt.reset()` is called in `setUp` (line 37), which is a full registry teardown and rebuild on every test. `setUpTestGetIt()` from `widget_test_utils.dart` wraps this in a safer, more selective approach and avoids the overhead of rebuilding all singletons. Addressing this also fixes the quality issue above.
  **RESOLVED (already fixed):** the inline `getIt.reset()` no longer exists — the [HIGH] fix at line 43 replaced `setUp` with `setUpTestGetIt(additionalSetup:)` and `tearDown(tearDownTestGetIt)`. Verified: the only `getIt`-related calls in the file are `setUpTestGetIt` (line 36) and `tearDownTestGetIt` (line 54).

## Summary

- `skill_inference_runner.dart` (931 L) and its test (4036 L) are the dominant size violations; both should be split along the four `run*` method boundaries.
- `auto_checklist_service_test.dart` violates three infrastructure rules: inline `getIt` setup, duplicate fallback registration, and a near-duplicate test body.
- The `skillInferenceRunnerProvider factory` tests at the end of the runner test file contain only `isA<>` smoke assertions and need behavior-level assertions to comply with the "no constructor smoke tests" rule.
- 2 genuine Glados candidates found: `_resolveEntryContent` (multi-case pure dispatch with `reduce`) and `_formatSpeechDictionaryText` (pure string escaping).
- 3 coverage gaps identified: missing `runImageGeneration` content-resolution paths, missing `runTranscription` provider-wrong-type fallback, and missing direct `shouldAutoCreate`-returns-false-with-checklistIds test.
- 2 speed wins: reuse of disk files across tests in the runner test groups (reduces real I/O), and per-group Glados bench reuse.
