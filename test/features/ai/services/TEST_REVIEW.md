Even without splitting the impl, this test file has distinct `group` blocks (`runTranscription`, `runImageAnalysis`, `runPromptGeneration`, `runImageGeneration`, provider factory, entity guards) that map cleanly to 3–4 files of ~1000 lines each. ## Generative (Glados) testing opportunities

- [ ] **[MED]** `lib/features/ai/services/skill_inference_runner.dart` — `_resolveEntryContent` (lines 792–827): a pure static method with a well-defined input type (`JournalEntity`) and three distinct cases (`JournalAudio` with/without transcript, `JournalEntry` with/without text, other). A Glados property test could verify: (a) output is never null; (b) for `JournalAudio` with a non-empty `entryText`, the result equals `entryText.plainText.trim()`; (c) for multiple transcripts the latest (by `created`) wins over an older one. This logic is non-trivial (it involves `reduce` over `created` dates) and pure — a genuine Glados candidate.

- [ ] **[MED]** `lib/features/ai/services/skill_inference_runner.dart` — `_formatSpeechDictionaryText` (lines 830–843): pure function, deterministic output for any `List<String>`. A property could assert that the output always starts with `'IMPORTANT'` when terms is non-empty and is always empty-string when terms is empty, and that round-tripping a term through JSON escaping produces a valid JSON string.

- [ ] **[LOW]** `lib/features/ai/services/profile_automation_service.dart` — `_fallbackCandidateRank` (lines 286–308): pure function mapping `InferenceProviderType` → integer rank. A Glados property could verify that (a) MLX Audio ranks best on macOS, (b) MLX Audio demotes to rank 100 on non-macOS, (c) the comparator is consistent (no ties that break transitivity). The manual test at line 634 covers the comparator ordering but exercises only a fixed set of providers; Glados would verify all enum values.

No other files in this scope contain sufficiently non-trivial pure logic that is not already covered by existing Glados tests.

## Test execution speed opportunities

- [x] **[HIGH]** `test/features/ai/services/skill_inference_runner_test.dart` — real filesystem I/O: the `setUp` at line 537 calls `Directory.systemTemp.createTemp(...)` and most `runTranscription`/`runImageAnalysis` tests write actual bytes to disk (`File(...).writeAsBytes([...])`). This is legitimate because the runner reads audio/image bytes from disk, but ~20 happy-path and override-fallback tests each write files individually. Extracting a `setUp`-level audio file and image file (created once per `group`) and reusing paths across tests within a `group` would reduce disk I/O from ~20 `writeAsBytes` calls to 2 per group. Estimated savings: 50–200 ms per test run on slow CI disks. **RESOLVED (partially assessed):** the ten verbatim create-dir+write blocks now share `createStubAudioFile()`/`createStubImageFile()` helpers. The per-test temp directory itself stays (it is recreated in setUp by design for isolation), so the per-group single-write optimization does not apply without weakening that isolation.

- [ ] **[MED]** `test/features/ai/services/skill_inference_runner_test.dart` — the two Glados tests (`promptStreamScenario` numRuns=120 and `promptGenerationScenario` numRuns=160) each create a full `ProviderContainer` + `SkillInferenceRunner` per iteration. The `_GeneratedSkillRunnerBench.create()` helper at line 179 is efficient, but the per-run container allocation (×120 and ×160) means 280 containers are created across both tests. If the bench were reusable across runs within one Glados test (resetting mock call counts between iterations), this would reduce allocation overhead. Impact: estimated 5–15% speed gain on the Glados CI lane.

- [ ] **[LOW]** `test/features/ai/services/skill_inference_runner_test.dart` — the `stubLoggingEvent()` / `stubLoggingException()` helpers at lines 480–498 are defined at top-level and also duplicated inside `_GeneratedSkillRunnerBench` (lines 234–252). The bench helpers call out to local variables, but the top-level helpers are reconstructed fresh on every test. Moving both stubs into the bench or a shared helper would make the duplication visible and invite consolidation.

- [ ] **[LOW]** `test/features/ai/services/auto_checklist_service_test.dart` — `getIt.reset()` is called in `setUp` (line 37), which is a full registry teardown and rebuild on every test. `setUpTestGetIt()` from `widget_test_utils.dart` wraps this in a safer, more selective approach and avoids the overhead of rebuilding all singletons. Addressing this also fixes the quality issue above.

## Summary

- `skill_inference_runner.dart` (931 L) and its test (4036 L) are the dominant size violations; both should be split along the four `run*` method boundaries.
- `auto_checklist_service_test.dart` violates three infrastructure rules: inline `getIt` setup, duplicate fallback registration, and a near-duplicate test body.
- The `skillInferenceRunnerProvider factory` tests at the end of the runner test file contain only `isA<>` smoke assertions and need behavior-level assertions to comply with the "no constructor smoke tests" rule.
- 2 genuine Glados candidates found: `_resolveEntryContent` (multi-case pure dispatch with `reduce`) and `_formatSpeechDictionaryText` (pure string escaping).
- 3 coverage gaps identified: missing `runImageGeneration` content-resolution paths, missing `runTranscription` provider-wrong-type fallback, and missing direct `shouldAutoCreate`-returns-false-with-checklistIds test.
- 2 speed wins: reuse of disk files across tests in the runner test groups (reduces real I/O), and per-group Glados bench reuse.
