## Test execution speed opportunities

- [x] **[HIGH] `unified_ai_progress_view_test.dart` — 7 `tester.runAsync` blocks** (lines 413, 457, 516, 561, 615, 1209, 1281): `runAsync` forces the test framework to use real OS task-scheduling. Each block costs ~50-150 ms of wall time versus the near-zero cost of `tester.pump()`. Total impact estimate: 5–15 s extra per full CI shard. The post-frame-callback + stream-subscription pattern used in these blocks can be driven deterministically with sequential `pump()` calls once the `ProviderContainer` state is set up synchronously before `pumpWidget`.

- [ ] **[MED] `unified_ai_progress_view_test.dart` line 1993 — `pump(const Duration(milliseconds: 350))`**: advances the clock by 350 ms for a SnackBar animation. The progress-view `_installModel` success toast is based on a `DesignSystemToast`; replacing this delay with `tester.pump(const Duration(milliseconds: 1))` after the toast is mounted (which satisfies the "entered the tree" check) is sufficient.

- [ ] **[MED] `unified_ai_popup_menu_test.dart` line 1155 — `pump(const Duration(seconds: 1))`**: advances the clock by a full second to let modal-close/open animations play out. Using `tester.pump(const Duration(milliseconds: 16))` × N or `tester.pump(const Duration(milliseconds: 400))` to cover a single animation frame cycle is sufficient and faster.

- [ ] **[MED] `unified_ai_popup_menu_test.dart` — 44 `pumpAndSettle()` calls** and `unified_ai_progress_view_test.dart` — 58 `pumpAndSettle()` calls: where the settled state is reached within one or two frames (e.g. provider overrides resolve synchronously), a single `pump()` is cheaper and avoids the 10 s default timeout risk.

- [ ] **[LOW] `widgets/ai_error_display_test.dart` — `tester.view.physicalSize` resize in every test body** (8 occurrences): extracting to a `setUp` / `_pumpWide` helper not only fixes DRY but also removes 8 redundant addTearDown registrations per test run, reducing teardown overhead.
