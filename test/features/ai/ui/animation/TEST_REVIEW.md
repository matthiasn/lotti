## File size / split opportunities

- [ ] **[HIGH] `lib/features/ai/ui/animation/ai_state_shader_animation.dart` (738 lines)** — this is the largest animation impl file. It contains: two enum types + label extensions (`AiVoiceShaderRoute`, `AiThinkingShaderRoute`), `AiStateShaderAssets`, `AiStateShaderProgramCache`, and two complete StatefulWidget+Painter pairs (`AiVoiceInputShader`/`AiVoiceInputShaderPainter`/`AiVoiceInputFallbackPainter` and `AiThinkingLineShader`/`AiThinkingLineShaderPainter`/`AiThinkingLineFallbackPainter`). Seam: split the two shader families into `ai_voice_input_shader.dart` and `ai_thinking_line_shader.dart`; keep `AiStateShaderAssets` and `AiStateShaderProgramCache` in a shared `ai_shader_program_cache.dart`.

- [ ] **[HIGH] `test/features/ai/ui/animation/ai_running_animation_test.dart` (774 lines)** — oversized for a 361-line impl file. Contains `AiRunningAnimation` group (lines 17–41), `AiRunningAnimationWrapper` group (42–281), and `AiRunningAnimationWrapperCard` group (283–774). Per the 1-test-file-per-source rule, `ai_animation_modal_interaction_test.dart` (414 lines) should be merged here rather than kept separate, OR the full set consolidated around sub-widget seams if the impl file is split.

- [ ] **[MED] Dual test files for one impl file** — `ai_running_animation_test.dart` and `ai_animation_modal_interaction_test.dart` both test `ai_running_animation.dart`. Merge `ai_animation_modal_interaction_test.dart` into `ai_running_animation_test.dart`; the "modal interaction" tests only exercise the same `AiRunningAnimationWrapper` widget already covered in the main file (interactive tap, non-interactive, missing active inference). After merge the combined file would be ~1188 lines — still above 1000 — suggesting the impl itself should be split first.

- [ ] **[LOW] `lib/features/ai/ui/animation/ai_running_animation.dart` (361 lines)** — acceptable size. No split needed unless `AiRunningAnimationWrapper` vs `AiRunningAnimationWrapperCard` are separated.

## Test quality improvements

- [ ] **[HIGH] `ai_animation_modal_interaction_test.dart` — inline mock `MockAiConfigDb extends Mock implements AiConfigDb`** (line 21): if a `MockAiConfigDb` exists in `test/mocks/mocks.dart` it must be removed from here; if it does not yet exist, add it there.

- [ ] **[HIGH] `ai_animation_modal_interaction_test.dart` — manual GetIt `registerSingleton` / `getIt.reset`** (lines 39–43, 63): violates the `setUpTestGetIt()` / `tearDownTestGetIt()` rule. The registrations of `DomainLogger`, `AiConfigRepository`, and `AiConfigDb` should go through the standard test helpers.

- [ ] **[MED] `ai_running_animation_test.dart` — manual `container.dispose()` instead of `addTearDown`** (lines 105, 133, 184, etc.): the containers are disposed in the test body after the assertions, but if an assertion throws the dispose is skipped. Use `addTearDown(container.dispose)` immediately after creation (the pattern used in some but not all test groups in this file — see e.g. the `showExisting` tests in `unified_ai_progress_view_test.dart` for the correct pattern).

- [ ] **[MED] `ai_animation_modal_interaction_test.dart` lines 92–152 — test "tapping animation opens progress modal when inference is active"**: the assertion only checks that `gesture.onTap isNotNull` — it does not actually tap and verify what happens. The intent is clear from the comment ("Full modal display testing would require more complex setup") but it leaves the tap handler's effect untested. The test body should be evolved to verify the tap opens a modal (or fire and verify a specific provider action).

- [ ] **[MED] `ai_running_animation_test.dart` lines 188–236 — test "should handle tap when interactive with active inference"**: taps the `GestureDetector` and then asserts only `find.byType(AiRunningAnimation)` — which proves nothing changed. If the tap is supposed to open a modal or call a callback, that outcome must be asserted.

- [ ] **[LOW] `ai_state_shader_animation_test.dart` — `test('paint fallback painters...', ...)` (line 440)** not wrapped in `testWidgets`: this test creates painters and calls `_paintWith` but does not require `WidgetTester`. That is fine for a pure-logic test — just confirm there is no hidden async/UI dependency (there is not based on static analysis).

- [ ] **[LOW] `ai_running_animation_test.dart` `AiRunningAnimationWrapperCard` group — test "should render with isInteractive parameter"** (line 352): asserts `wrapper.isInteractive, isTrue` — this only proves the parameter was passed, not that the interactive behaviour (GestureDetector, tap callback) is actually wired. This assertion has marginal value; the real behaviour test is `should render animation wrapped in GestureDetector` which is present in `AiRunningAnimationWrapper` group but absent from `AiRunningAnimationWrapperCard`.

## Generative (Glados) testing opportunities

- [ ] **[HIGH] `AiVoiceInputShaderPainter.shouldRepaint` and `AiThinkingLineShaderPainter.shouldRepaint`** — both painters implement `shouldRepaint` as a field-equality check (return `oldDelegate.X != X || ...`). This is pure boolean logic over ~8-10 named parameters. A Glados property test could verify the invariant: for any two painter instances where all fields are equal, `shouldRepaint` must return `false`; if exactly one field differs, it must return `true`. The current tests cover only 3–4 hand-picked change points. Generator shape: `_GeneratedPainterFields` combining all numeric/color/enum parameters via `combine9`.

- [ ] **[MED] `AiStateShaderProgramCache` load-and-reuse identity** — the cache memoises by returning the same `Future` instance. A property: calling `loadVoiceInput()` N times must always return `identical` objects. The current test only covers N=2. Glados with `any.positiveIntOrZero.map((n) => n % 8 + 2)` (2–9 calls) would cover more. This is simple but pure logic.

- [ ] **[LOW] `AiVoiceShaderRoute` and `AiThinkingShaderRoute` label extensions** — both expose a `label` getter that returns a non-null, non-empty string for each enum variant. A trivial property: `for all route in AiVoiceShaderRoute.values, route.label.isNotEmpty` — but this is so simple it is better covered by a standard unit test loop than by Glados.

## Coverage / missing-behavior gaps

- [ ] **[HIGH] `AiRunningAnimationWrapper` interactive tap → modal open**: the two existing "interactive tap" tests (`ai_running_animation_test.dart` lines 188–236 and `ai_animation_modal_interaction_test.dart` lines 91–152) do not verify that tapping actually opens a progress modal. The `_handleTap` method reads `activeInferenceControllerProvider`, builds a `WoltModalSheetPage`, and calls `WoltModalSheet.show`. None of these outcomes is asserted.

- [ ] **[HIGH] `ai_state_shader_animation_test.dart` — `AiVoiceInputShader` and `AiThinkingLineShader` `didUpdateWidget` code path**: both widgets reload the shader program only when `widget.programLoader` changes. The test "reloads programs and stops animation when time override is set" (lines 141, 270) verifies `loadCount == 2` but only because the loader reference changes. The case where `programLoader` does NOT change on widget update (so the cache is reused) is not tested.

- [ ] **[MED] `AiRunningAnimationWrapperCard` — no test for the non-interactive (no GestureDetector) path when `isInteractive: false`**: the `AiRunningAnimationWrapper` group has explicit tests for `isInteractive: true` and `isInteractive: false`, but `AiRunningAnimationWrapperCard` only tests the `isInteractive: true` case (line 352) and does not have a corresponding `isInteractive: false` test.

- [ ] **[MED] `AiVoiceInputShader` semantics label**: the widget accepts a `semanticsLabel` parameter that wraps output in a `Semantics` widget. No test verifies that the label reaches the widget tree with the expected text.

- [ ] **[LOW] `ai_state_shader_animation_test.dart` — shader route-specific paint output**: tests only verify `shouldRepaint` logic and the painter type (shader vs. fallback). No test verifies that different `AiVoiceShaderRoute` values produce different visual output (e.g. by checking `shouldRepaint` when only the route changes — currently covered in the `shouldRepaint` test but worth explicitly naming).

## Test execution speed opportunities

- [ ] **[MED] `ai_running_animation_test.dart` — each ProviderContainer-based test calls `tester.pump()` after `pumpWidget`**: fine per se, but 12 tests × container-create + pump + dispose adds up. Shared `setUpAll` / `setUpTestGetIt()` would avoid repeated GetIt registration in the modal interaction file (currently 4 setUp registrations + 4 tearDown resets).

- [ ] **[MED] `ai_state_shader_animation_test.dart` — two `ui.FragmentProgram.fromAsset` calls per `shouldRepaint` test** (lines 321, 373): real GPU asset loads. These happen in `testWidgets` (so inside the test runner) but use the actual file system. Since `timeOverride` is passed to freeze animation, these tests do not loop; cost is acceptable but one shared program fixture via `setUpAll` + `late ui.FragmentProgram` would avoid N×asset-load overhead.

- [ ] **[LOW] `ai_animation_modal_interaction_test.dart` lines 40–45 — full `getIt` singleton registration in `setUp`**: two registrations (`DomainLogger`, `AiConfigRepository`, `AiConfigDb`) run before every test. `setUpTestGetIt()` centralises this and only registers what is needed.
