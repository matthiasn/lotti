## Generative (Glados) testing opportunities

- [x] **[HIGH]** `_maskApiKey` in `lib/features/ai/ui/settings/provider/ai_provider_detail_page.dart:555` — a pure `String → String` function with three structural branches (empty → `''`, length ≤ 4 → all dots, length > 4 → `•••• last4`). A Glados property can verify: (1) non-empty output always ends with the last 4 chars of the trimmed input; (2) leading/trailing whitespace is stripped; (3) result length is always ≤ input.trimmed.length. Only a handful of example-based tests currently cover this indirectly through widget assertions.

- [ ] **[MED]** `_findConfiguredKnownModel` in `lib/features/ai/ui/settings/services/provider_prompt_setup_service.dart:222` — pure function with a three-clause predicate (providerModelId match, exact-provider match, cross-provider usability). A Glados property over arbitrary lists of `AiConfigModel` and `AiConfigInferenceProvider` can verify: if a model matches providerModelId AND is in the same providerId, it is always returned; if it matches providerModelId but its provider is deleted or a different type, it is never returned. The function is fully deterministic and has no I/O.

- [ ] **[MED]** `ConnectionCheckState` sealed hierarchy dispatch in `lib/features/ai/ui/settings/services/connection_verifier_service.dart` — the URL normalisation path (empty base → default, whitespace trimming, scheme validation) and API-key normalisation (`trim()` + empty guard) are pure string-to-enum-arm functions. A Glados property over arbitrary URL strings can verify: (1) any URL that has no `http`/`https` scheme always routes to `invalidBaseUrl`; (2) a key of pure whitespace always produces `idle` for cloud providers. The existing tests cover these with 3–4 hand-picked fixtures; property tests would close the remaining edge cases.

- [ ] **[LOW]** `AiFtueResult.totalModels` in `provider_prompt_setup_service.dart:64` — `modelsCreated + modelsVerified`. Trivial but can carry a property: `totalModels == modelsCreated + modelsVerified` for any non-negative pair of integers. Covered only by three example fixtures in the test file.

## Test execution speed opportunities

- [x] **[HIGH]** `test/features/ai/ui/settings/widgets/provider_filter_chips_row_test.dart` — 22 `pumpAndSettle` calls (lines 153, 176, 196, 214, 241, 281, 305, 329, 371, 397, and more). Each `pumpAndSettle` has a 10-second default timeout; for a stateless chip widget there are no running animations to wait for. Replace with `tester.pump()` or `tester.pump(const Duration(milliseconds: 50))`. **Estimated saving: 22 × potential 10 s hang risk per flaky run.**

- [x] **[HIGH]** `test/features/ai/ui/settings/widgets/ai_settings_tab_bar_test.dart` — 9 `pumpAndSettle` calls in a stateless tab-bar widget test, most gating only localization load. Replace with `await tester.pump()`. The test file also rebuilds the full `MaterialApp + DefaultTabController + Builder + Scaffold` six times separately; extracting a shared `createWidget` (already partially exists) and a `setUpAll` that registers fonts once would reduce per-test overhead.

- [x] **[HIGH]** `test/features/ai/ui/settings/widgets/ai_settings_floating_action_button_test.dart` — 9 `pumpAndSettle` for a pure stateless `DesignSystemFloatingActionButton` wrapper. No animations. Replace with `tester.pump()`. The test also pumps a full new widget tree for each of 3 tabs in the "distinct labels" test (lines 105–121); a single pump with a loop over tabs would be faster.

- [ ] **[MED]** `test/features/ai/ui/settings/widgets/mlx_audio_model_download_dialog_test.dart` — tests in the `MlxAudioModelDownloadDialog` group use `await Future<void>.value(); await tester.pump();` pairs (lines ~356, ~392, ~424, ~463) as a micro-flush idiom. ## Summary

- **5 oversized impl files** (≥500 lines): `ai_provider_detail_page.dart` (842), `ai_provider_setup_preview_modal.dart` (874), `ai_settings_cards.dart` (847), `provider_prompt_setup_service.dart` (690), `ai_config_delete_service.dart` (457).
- **2 oversized test files** (≥1000 lines): `provider_prompt_setup_service_test.dart` (2033), `ai_config_delete_service_test.dart` (1117); plus `ai_provider_detail_page_test.dart` (1049) and `ai_settings_config_sliver_test.dart` (948) approaching the limit.
- **6 weak/DRY test quality findings**: 6 copy-paste per-provider widget helpers in `provider_prompt_setup_service_test.dart`; 23 copy-paste per-type one-value tests in `ftue_trigger_service_test.dart`; DS-token property assertions in `ai_settings_tab_bar_test.dart`; widespread raw `MaterialApp` boilerplate instead of `makeTestableWidget`; duplicated `registerFallbackValue` calls; orphan test file `ai_switch_field_test.dart` with no matching source.
- **3 Glados candidates**: `_maskApiKey` (pure `String → String`), `_findConfiguredKnownModel` (pure list search), URL/key normalisation logic in `connection_verifier_service.dart`.
- **7 test execution speed wins**: ~100 `pumpAndSettle` calls across 10 test files that could be replaced with `tester.pump()` or `tester.pump(kThemeAnimationDuration)`, eliminating 10-second hang risk per call. Biggest single win: `provider_filter_chips_row_test.dart` (22 calls) and `ai_settings_floating_action_button_test.dart` (9 calls on a stateless widget).
- **Biggest opportunity**: The 2033-line `provider_prompt_setup_service_test.dart` with 6 duplicated per-provider widget harnesses is the single file where consolidation (one generic helper + `registerAllFallbackValues`) would cut 300+ lines, remove centralised-mock violations, and reduce per-test widget-tree construction overhead simultaneously.
