## Generative (Glados) testing opportunities

- [x] **[HIGH] `ProfilePinningSelectorState._filterEligible` and `._resolveReferencedProviderTypes`** — both are pure functions with structured inputs: **RESOLVED:** both helpers exposed as `@visibleForTesting` statics (`filterEligible`, `resolveReferencedProviderTypes`); Glados properties pin the capability-superset filter (incl. cloud-only → all-eligible) and the id→provider-type resolution incl. legacy-id lookup, unknown ids, and the loading-state empty set (160 runs each).
  - `_filterEligible(nodes, requiredCapabilities)` returns a filtered `Iterable<SyncNodeProfile>`. The algebraic invariant: if `requiredCapabilities` is empty, all nodes pass; if a node's `capabilities` is a superset of `requiredCapabilities`, it passes; otherwise it is excluded. A Glados property with generators for `List<SyncNodeProfile>` (variable capabilities sets) and `Set<NodeCapability>` (random subsets of `NodeCapability.values`) could cover this exhaustively.
  - `_resolveReferencedProviderTypes(modelIds, models, providers)` maps model IDs to provider types. The invariant: for any model ID in the input list that resolves to a local-runtime provider type, the type appears in the output; cloud-only provider types are excluded. Both functions satisfy the Glados criteria (pure logic, structured input, algebraic invariants, not UI). Tag `tags: 'glados'` required.

- [ ] **[LOW] `InferenceModelPickerModal` — model list ordering**: the modal sorts models so the default appears first. For any list of `AiConfigModel` objects and any `defaultModelId` that appears in the list, the rendered order must put the default row at position 0. This is a pure-ish ordering property, though it is implicitly tested by the existing structural test. A Glados property with randomly ordered model lists would be more exhaustive but the current concrete test may be sufficient.

## Test execution speed opportunities

- [ ] **[MED] `ai_error_display_test.dart` — 30 `pumpAndSettle()` calls** across 17+ tests: for tests that have no animations (pure data-display widgets) a single `pump()` after the initial `pumpWidget` is sufficient. The `suggestions` group in particular resolves synchronously — replacing `pumpAndSettle()` with `tester.pump()` throughout would shave ~30 animation-settle timeouts from the test run.

- [ ] **[MED] `profile_pinning_selector_test.dart` — 14 `pumpAndSettle()` calls**, many after `DropdownButtonFormField` taps: the dropdown open/close animation is deterministic; `tester.pump(const Duration(milliseconds: 200))` captures the animation without the 10 s pumpAndSettle timeout risk.

- [ ] **[LOW] `ai_core_functionality_test.dart` — can be fully deleted after merging** into `settings/ai_config_card_test.dart`, eliminating 302 lines of redundant test execution.
