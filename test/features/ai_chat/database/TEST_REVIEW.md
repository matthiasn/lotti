## Test execution speed opportunities

- [x] **[HIGH]** `bulk_linked_entities_test.dart:484–575` — the 50-task performance test is the slowest unit test in this scope. It inserts 50 tasks + up to 150 linked entities into an in-memory Drift DB with 200+ `upsert` calls. On a CI runner this loop can take 200–400 ms. Reducing the entity count from 50 to 15 would still meaningfully exercise the bulk query without the insert overhead, and the correctness assertion (result length and per-task linked count) would remain valid. **RESOLVED:** the loop builds 15 tasks instead of 50; the per-task linked-count assertions are unchanged.

- [x] **[MED]** `bulk_linked_entities_test.dart:578–678` — the performance comparison test inserts 10 tasks + 10 linked entities and then runs both bulk and individual queries. The individual query loop makes 10 separate DB calls. This test is effectively measuring test infrastructure overhead, not production correctness. If the Stopwatch assertions are removed (as recommended above), the individual-call loop can also be removed, saving ~10 DB round-trips per test run.
  **RESOLVED:** stale — the Stopwatch assertions are already gone; the per-id loop now serves as a correctness oracle (bulk result must equal the union of individual `getLinkedEntities` calls), which is exactly the kind of equivalence check worth keeping.

- [x] **[LOW]** `bulk_linked_entities_test.dart` `setUp` creates a fresh `JournalDb(inMemoryDatabase: true)` per test and `tearDown` closes it. For 11 tests this means 11 DB open+close cycles. A `setUpAll` / `tearDownAll` with a single shared DB (and a `DELETE` sweep or `setUp` that resets modified tables) would reduce the open/close overhead, though for Drift in-memory databases this is typically fast.
  **RESOLVED:** confirmed but intentionally kept per-test, no change — the item's own conclusion ("for Drift in-memory databases this is typically fast") holds. The tests reuse overlapping IDs (`task-1`, `task-2`, `ai-response-1`, `link-1`, …) and rely on a clean DB per test (e.g. `returns empty map`/`returns empty lists`/`handles broken links` would observe leftover rows from prior tests under a shared DB). A shared `setUpAll` DB would therefore require a `DELETE` sweep between tests — exactly the cross-test resource-cleanup risk the repo's CI-cleanup convention (`very_good test` same-thread) warns against — for a negligible open/close saving. The fresh-per-test in-memory DB is the safer established pattern.

## Summary

- 0 oversized impl files (database logic lives in the core `database/` package, not in this feature directory).
- 1 oversized test file (`bulk_linked_entities_test.dart`, 909 L) driven by verbose entity construction and a performance test group that should be removed.
- 2 weak test patterns: wall-clock `Stopwatch` assertion (flaky on CI), and dead performance instrumentation with no assertions.
- 2 Glados candidates: `ChatMessage` and `ChatSession` JSON roundtrips (pure serialization).
- Biggest opportunity: remove the `Stopwatch` performance assertions and extract `_makeTask` / `_makeAiResponse` / `_makeLink` factory helpers — these two changes together reduce the file from 909 to ~380 lines.
- 1 significant speed win: reduce the 50-entity performance test to 15 entities, saving ~200–400 ms of CI DB insert time.
