## File size / split opportunities

- [x] **[HIGH]** `test/…/database/bulk_linked_entities_test.dart` — 909 lines for 11 tests on a single DB method. The per-test entity construction is very verbose: each test manually creates `JournalEntity.task(...)`, `JournalEntity.aiResponse(...)`, and `EntryLink.basic(...)` objects in full. A file-level `_makeTask(id)`, `_makeAiResponse(id)`, and `_makeLink(from, to)` factory trio would collapse ~30 lines of construction code per test to 3 lines. With that extraction the file would drop to ~400 lines. **RESOLVED:** file-level `_makeTask`/`_makeAiResponse`/`_makeLink` factories replace the standard-shape constructions (7 tasks, 7 responses, 15 links converted; 909 → 680 lines). Variant-status/date constructions stay explicit.

  Additionally, the `performance comparison` group (lines 578–678) is a separate concern (benchmarking, not correctness) and should be extracted to its own file or removed — see execution speed section below.

- [ ] **[MED]** `test/…/models/` — the four model test files are correctly slim (37–189 lines each). No splitting needed.

## Test quality improvements

- [x] **[HIGH]** `bulk_linked_entities_test.dart:484–575` (`handles large number of parent IDs efficiently`) — the assertion at line 567 is `expect(stopwatch.elapsedMilliseconds, lessThan(1000))`. Wall-clock timing assertions in unit tests are environment-dependent and flaky on loaded CI runners. This test should be replaced by a functional assertion only: verify the result map length and per-task link counts. Performance guarantees belong in a dedicated benchmark, not a unit test. **RESOLVED:** the wall-clock assertion is gone; the test asserts only the result-map length and per-task linked counts.

- [x] **[HIGH]** `bulk_linked_entities_test.dart:578–678` (`performance comparison`) — this entire group measures `Stopwatch` values for bulk vs. individual calls but the concluding assertion explicitly states it is not enforcing performance: "the functional equivalence assertion above is the important verification." The Stopwatch variables (`bulkStopwatch`, `individualStopwatch`) are created and stopped but never asserted. The test body only asserts result equality, making the performance instrumentation dead code. Remove the `Stopwatch` machinery and the group comment about flaky benchmarks — the functional assertion is correct on its own. **RESOLVED:** the dead Stopwatch instrumentation and the flaky-benchmark commentary are removed; the test keeps the functional bulk≡individual equivalence assertion.

- [ ] **[MED]** `bulk_linked_entities_test.dart` — the `basic functionality` tests at lines 122–362 each construct two tasks with full `TaskData`, `TaskStatus`, and `AiResponseData` payloads in-line. The test at line 259 (`handles multiple linked entities per parent ID`) is 103 lines of setup for a 7-line assertion. Extract `_makeTask(String id, {DateTime? date})` and `_makeAiResponse(String id, {String response = ''})` helpers to a file-level scope.

- [ ] **[MED]** `chat_session_model_test.dart:17–36` — the JSON roundtrip test manually does `json['messages'] = s.messages.map((m) => m.toJson()).toList()` because `ChatSession.toJson()` does not deep-serialize messages. This is testing the workaround rather than the model. The comment explains the workaround (`// ChatSession.toJson() doesn't deep-serialize messages`) but this is a model deficiency worth flagging — if the model's own `toJson` cannot serialize itself for a roundtrip, the test should explicitly document this as a known limitation or the model should be fixed.

- [ ] **[LOW]** `chat_exceptions_test.dart:38–46` — the two tests `is a ChatException` and `is an Exception` each instantiate `ChatRepositoryException` and do a single `isA<>` check. These are constructor smoke tests with zero behavioral value (the type hierarchy is guaranteed by the `extends` keyword). Both should be removed.

- [ ] **[LOW]** `task_summary_tool_model_test.dart:81–94` (`TaskSummaryResult JSON roundtrip`) — only checks `taskId` and `metadata`. Should also assert `taskTitle`, `summary`, `status`, and `taskDate` to be a meaningful roundtrip test.

## Coverage / missing-behavior gaps

- [x] **[HIGH]** `bulk_linked_entities_test.dart` does not cover the case where a `fromId` appears in a link but the linked entity (`toId`) exists as a DB row yet is of a type that fails deserialization (corrupt `serialized` JSON). The `getLinkedEntities` path deserializes each row; a corrupted or unknown `type` value would surface an exception that should be caught and reported rather than crashing the whole bulk fetch. **RESOLVED (assessed, no change):** the prescribed catch-and-report lives in `lib/database` (`getJournalEntitiesForIdsUnordered` → `fromDbEntity` has no per-row guard), which is excluded from this run by explicit instruction — lib/database and test/database are being handled separately. Today a corrupt `serialized` row would propagate a deserialization error out of the bulk fetch.

- [ ] **[MED]** `ChatMessage.copyWith` — the model exposes `copyWith` (via Freezed), but no test verifies that `copyWith(isStreaming: true)` produces a streaming message while preserving all other fields. The `chat_message_model_test.dart` does not exercise `copyWith` at all.

- [ ] **[MED]** `ChatSession.copyWith` — similarly untested. The `chat_repository_test.dart` uses `session.copyWith(messages: [...])` but this is exercised implicitly through the repository, not as a direct model test.

## Test execution speed opportunities

- [x] **[HIGH]** `bulk_linked_entities_test.dart:484–575` — the 50-task performance test is the slowest unit test in this scope. It inserts 50 tasks + up to 150 linked entities into an in-memory Drift DB with 200+ `upsert` calls. On a CI runner this loop can take 200–400 ms. Reducing the entity count from 50 to 15 would still meaningfully exercise the bulk query without the insert overhead, and the correctness assertion (result length and per-task linked count) would remain valid. **RESOLVED:** the loop builds 15 tasks instead of 50; the per-task linked-count assertions are unchanged.

- [ ] **[MED]** `bulk_linked_entities_test.dart:578–678` — the performance comparison test inserts 10 tasks + 10 linked entities and then runs both bulk and individual queries. The individual query loop makes 10 separate DB calls. This test is effectively measuring test infrastructure overhead, not production correctness. If the Stopwatch assertions are removed (as recommended above), the individual-call loop can also be removed, saving ~10 DB round-trips per test run.

- [ ] **[LOW]** `bulk_linked_entities_test.dart` `setUp` creates a fresh `JournalDb(inMemoryDatabase: true)` per test and `tearDown` closes it. For 11 tests this means 11 DB open+close cycles. A `setUpAll` / `tearDownAll` with a single shared DB (and a `DELETE` sweep or `setUp` that resets modified tables) would reduce the open/close overhead, though for Drift in-memory databases this is typically fast.

## Summary

- 0 oversized impl files (database logic lives in the core `database/` package, not in this feature directory).
- 1 oversized test file (`bulk_linked_entities_test.dart`, 909 L) driven by verbose entity construction and a performance test group that should be removed.
- 2 weak test patterns: wall-clock `Stopwatch` assertion (flaky on CI), and dead performance instrumentation with no assertions.
- 2 Glados candidates: `ChatMessage` and `ChatSession` JSON roundtrips (pure serialization).
- Biggest opportunity: remove the `Stopwatch` performance assertions and extract `_makeTask` / `_makeAiResponse` / `_makeLink` factory helpers — these two changes together reduce the file from 909 to ~380 lines.
- 1 significant speed win: reduce the 50-entity performance test to 15 entities, saving ~200–400 ms of CI DB insert time.
