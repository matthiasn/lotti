## Test quality improvements

- [ ] **[HIGH]** Inline `MockSelectable<T>` at `task_summary_repository_test.dart:28` — the central `test/mocks/mocks.dart` already has a `MockSelectable<T>` (line 126). The inline version differs only in not requiring pre-seeded values (it stubs `get()` per-test instead). The comment says this intentionally, but the central version also supports this pattern because `MockSelectable` is a `Mock`: any test can call `when(selectable.get).thenAnswer(...)` after constructing it with an empty list. Migrate to the central mock to remove the local definition and the comment.

- [ ] **[HIGH]** Inline GetIt boilerplate in both `task_summary_repository_test.dart` (setUp, lines 243–254) and `chat_repository_test.dart` (setUp/tearDown, lines 1247–1280) manually calls `getIt.isRegistered / unregister / registerSingleton` instead of the project-standard `setUpTestGetIt()` / `tearDownTestGetIt()`. This is the pattern the AGENTS guidelines explicitly prohibit. These should delegate to the helpers or at minimum be extracted into a shared setUp.

- [ ] **[HIGH]** `chat_repository_test.dart` — the system message test (lines 452–483) calls `repository.toString()` and asserts `isA<String>()` — this is a zero-value smoke assertion. The actual content check happens three tests later (lines 1191–1231) via `capturedSystemMessage`. The earlier test should be deleted or merged into the meaningful one.

- [ ] **[MED]** `task_summary_repository_test.dart:981-1133` — the `maps all task status values correctly` test iterates 7 status variants inside a single `test()` via a `for` loop. This is the right DRY approach, but each loop iteration re-stubs the mock from scratch (35+ lines per iteration) with an identical selectable response. Extract a `_stubFullPipeline(mock, task)` helper shared by the loop body.

- [ ] **[MED]** `chat_message_processor_test.dart` — the `processTaskSummaryTool` group contains two nearly-identical tests (`handles invalid date format or calendar date in arguments` at line 284, and `handles invalid calendar date in start_date specifically` at line 337) that differ only in the date string fed to the tool call arguments. Both mock the repository to throw `FormatException('Invalid calendar date')` and assert the same two things. These should be merged into a single parameterized test or the second removed.

- [ ] **[MED]** `chat_repository_test.dart:571-604` (`properly delegates to ChatMessageProcessor methods`) — the assertion `expect(exceptionCaught, isTrue)` guards against a test that never throws, but the preceding `try/catch` swallows the exception and re-uses the error string. This pattern makes failures silent: if the wrong exception is thrown the test still passes. Replace with `expectLater(..., throwsA(...))`.

- [ ] **[LOW]** `chat_message_processor_test.dart:452–485` (`buildFinalPromptFromMessages`) — the test verifies `contains('User:')` and `contains('Hello')` but not the Tool-response line format beyond a loose string match. Since this is pure formatting logic, stronger structural assertions (line ordering, exact prefix) would be valuable.

## Generative (Glados) testing opportunities

- [ ] **[MED]** `ChatMessageProcessor.buildPromptFromMessages` and `buildFinalPromptFromMessages` are pure deterministic string-assembly functions. A Glados property could assert that for any sequence of user/assistant/tool messages the output always contains each message's content exactly once and in the correct role prefix order, and that concatenation of chunks never produces partial tokens. This would replace the current 5 hand-picked fixture tests and catch edge cases like empty strings, newlines in content, or Unicode.

- [ ] **[MED]** `ChatMessageProcessor.accumulateToolCalls` is pure accumulation logic (no I/O). A Glados property over sequences of `(index, id?, arguments-chunk)` could assert: for any sequence, the final tool call list has exactly one entry per unique effective id, and the accumulated arguments equal the concatenation of all chunks for that id in order. The existing 4 static tests in `_accumulate_tool_calls_test.dart` cover corner cases but not random interleaving.

- [ ] **[LOW]** `TaskSummaryRepository.parseLocalStartOfDay` / `parseLocalEndOfDay` are pure date-parsing functions. `task_summary_tool_model_test.dart` already has a Glados round-trip test for `TaskSummaryRequest` serialization (correct). There is no property test for the repository's *validation* logic (regex + calendar overflow). A small Glados property could generate random date strings and assert that valid `YYYY-MM-DD` dates parse without throwing, while strings with wrong lengths or impossible days always throw `FormatException`.

## Test execution speed opportunities

- [ ] **[MED]** `task_summary_repository_test.dart` — each of the 23 tests calls `getIt.reset` in `tearDown` (line 254 and 1499), which is a full GetIt reset including any global state. Consolidating into fewer larger test flows (or using `setUpAll` for the mock wiring shared by the group) would reduce GetIt churn.
