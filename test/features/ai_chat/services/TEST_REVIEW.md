## Test quality improvements

- [x] **[HIGH]** `realtime_transcription_service_test.dart:415` — `_TestBench.scheduleDone()` uses `Future<void>.delayed(const Duration(milliseconds: 50))` as a real timer to schedule the `transcription.done` event. This is a fake-time policy violation: test suite delays the whole CI shard by 50 ms per call, and the delay is environment-dependent. The correct approach is to call `bench.simulateDone()` (which already exists in `_TestBench`) directly after `fakeAsync` elapses past the delay, or to schedule it through the fake-async clock.

- [x] **[HIGH]** `realtime_transcription_service_test.dart:950` and `:991` — two MLX error tests use `Future<void>.delayed(const Duration(milliseconds: 20))` to schedule the error event after `stop()` is called. These should use `fakeAsync`/`async.elapse(Duration(milliseconds: 20))` to avoid real-wall-clock waits. The existing `fakeAsync` wrappers elsewhere in the file (lines 686, 1013, 1423) demonstrate the pattern is understood; it just was not applied here.

- [x] **[HIGH]** `realtime_transcription_service_test.dart` — the `_FakeDomainLogger` inline class (lines 28–52) is a file-local fake that duplicates capability already provided by the central `MockDomainLogger` in `test/mocks/mocks.dart`. The only functional difference is that `_FakeDomainLogger` collects `events` and `exceptions` into lists for later assertion, while `MockDomainLogger` uses Mocktail `verify`. The inline fake is acceptable for the assertion style used here, but the `_FakeDomainLogger` class is 25 lines and is referenced in 5+ test groups via `fakeLogging`. It should be documented with a comment explaining why it is preferred over `MockDomainLogger` for this file, or extracted to a test utility file.

- [x] **[MED]** `audio_transcription_service_test.dart` — each of the 6 Gemini-config tests (lines 65–481) rebuilds the same `AiConfigInferenceProvider(id: 'p1', ...)` + `AiConfigModel(id: 'm1', name: 'gemini-2.5-flash', ...)` pair from scratch using `aiRepo.saveConfig(...)`. The `sharedDb` was introduced to amortize DB creation cost, but each test still saves its own config entries. Because Drift in-memory DBs do not auto-reset between tests, successive tests accumulate rows. A `setUpAll` that pre-populates the shared Gemini config once would reduce the per-test DB writes from 2 to 0 for these tests.
  - **RESOLVED:** done — `setUpAll` now seeds the shared DB once with the standard Gemini p1/m1 pair; tests that need it use `sharedRepo` with zero per-test writes. Tests whose intent depends on the *absence* of rows (flash-fallback, no-audio-model, provider-missing, keeps-Mistral-batch) were silently broken by shared-DB accumulation — they now use isolated in-memory DBs and verify the captured model id, restoring their actual intent.

- [x] **[MED]** `realtime_transcription_service_test.dart:649–656` (`connects WebSocket and sets isActive`) — the test asserts `bench.service.isActive == true` after starting, then closes the PCM controller. It does not verify that any WebSocket message was sent (e.g. a session setup payload). The neighboring test at line 658 does check the `input_audio.append` message is sent. These two tests could be merged into one richer assertion covering both state and wire output.
  - **RESOLVED:** done — merged into `'connects WebSocket, sets isActive, and forwards PCM as base64'`: one test asserting both the state flip and the `input_audio.append` wire output.

- [ ] **[LOW]** `audio_transcription_service_test.dart:483–607` — the `batch guard: excludes realtime models` group contains 3 tests that each create a fresh `AiConfigRepository`, register configs, and assert which model is selected. All three share the same structure. A parameterized helper `_batchGuardCase(realtime: ..., batch: ..., expectModel: ...)` would eliminate ~120 lines of duplication.

- [ ] **[LOW]** `realtime_transcription_service_test.dart:541–632` — the `resolveRealtimeConfig` group contains 5 tests for filter cases (`skips non-audio models`, `skips non-Mistral providers`) that each create a new `AiConfigDb`, `AiConfigRepository`, `MistralRealtimeTranscriptionRepository`, and `ProviderContainer` from scratch (35 lines each). These should use `_TestBench.create()` with config overrides, matching the existing bench pattern.

## Generative (Glados) testing opportunities

- [x] **[MED]** `RealtimeTranscriptionService` MLX confirmed-text deduplication — `realtime_transcription_service.dart` implements an incremental delta extraction algorithm: given a previous accumulated text and a new confirmed text, it extracts only the new suffix. This is non-trivial pure logic (string prefix stripping, rewrite handling). A Glados property could generate arbitrary `(previous, incoming)` string pairs and assert: (a) if `incoming` starts with `previous`, the delta equals the remainder; (b) if `incoming` is a prefix of `previous`, the delta is empty or a rewrite; (c) delta + previous never introduces duplicate characters. The current 3 static tests (`deduplicates`, `handles truncation and rewrites`) only cover hand-crafted cases.
  - **RESOLVED:** done — added a `@visibleForTesting debugConfirmedTextDelta` seam and a Glados2 property (overlap-rich 'ab ' alphabet, numRuns 150, `tags: 'glados'`) covering: empty-previous identity, append-remainder, prefix-truncation→empty, and the universal suffix law (the delta is always a suffix of `next`, so appending never duplicates confirmed text).

- [ ] **[LOW]** `AudioTranscriptionService.transcribe` model-selection logic — the selection algorithm (prefer flash model, fall back to first audio-capable non-realtime model) is pure filtering over a list. A Glados property over lists of `(modelId, providerModelId, providerType)` triples could assert that the selected model is always: non-realtime, audio-capable, and the flash model when present.

## Test execution speed opportunities

- [x] **[HIGH]** `realtime_transcription_service_test.dart` — 4 tests use real-wall-clock delays: `scheduleDone` (~50 ms, lines 415, 919, 1054, 1131, 1147, 1165, 1178), and two MLX error tests (~20 ms each, lines 950, 991). The PCM buffer cap test at line 1575 also loops 61 iterations with `await Future<void>.delayed(Duration.zero)` 61 times. Switching `scheduleDone` calls to `simulateDone` within `fakeAsync` blocks and using `async.elapse()` for the 20 ms delays would eliminate all real waits from the test — potentially saving 200–500 ms of CI wall time per run.

- [x] **[MED]** `audio_transcription_service_test.dart` — 10 of the 15 tests create real filesystem temp directories and write files, then delete them. These real I/O operations are necessary (the service reads from disk), but the `setUpAll` / `tearDownAll` pattern could be extended to create one shared temp directory per test group rather than per test, reducing directory creation overhead.
  - **RESOLVED:** done — one `Directory.systemTemp.createTemp` in `setUpAll` with indexed fixture files via the `audioFile()` helper; per-test temp-dir creation is gone (deleted once in `tearDownAll`).

- [ ] **[LOW]** `realtime_transcription_service_test.dart:_TestBench.create` — each call creates a real in-memory `AiConfigDb`, saves 2 rows, and creates a `ProviderContainer`. For the 30+ tests that call `_TestBench.create()`, this adds up to significant DB init overhead. A class-level `setUpAll`-style bench factory that reuses a pre-populated DB (read-only after creation) would be faster, though it would require ensuring tests do not mutate the shared DB.

## Summary

- 1 oversized impl (`realtime_transcription_service.dart`, 659 L); 2 oversized tests (1 626, 1 345 L).
- 3 fake-time policy violations in `realtime_transcription_service_test.dart` (lines 415, 950, 991) using real millisecond delays that should use `fakeAsync`.
- 2 Glados candidates: MLX delta-deduplication algorithm (pure string logic), audio model selection filter (pure list logic).
- Biggest opportunity: migrate `scheduleDone()` helper to `fakeAsync`-compatible `simulateDone()` + elapse, removing all real-timer waits from the largest test file and speeding up CI.
- Key coverage gap: second `startRealtimeTranscription` call while active (no guard test); `transcribeStream` mid-stream error path; `returns null audioFilePath` test is missing its core assertion.
