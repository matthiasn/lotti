## Test quality improvements

- [x] **[HIGH]** `correction_capture_service_test.dart:341–355` — **Real-time usage violation.** `remainingTime` test creates `createdAt = DateTime.now().subtract(...)` and asserts with `inInclusiveRange(2, 3)`. This uses a real-wall-clock delta and can flake on slow CI runners. The fix is to mock `createdAt` to an absolute past date and compute the expected remaining deterministically, e.g.:
  ```dart
  // Use a fixed date far in the past to test the non-zero branch:
  final now = DateTime(2025, 1, 1, 12, 0, 5); // 5s after createdAt
  final createdAt = DateTime(2025, 1, 1, 12, 0, 0);
  ```
  Since `PendingCorrection.remainingTime` calls `DateTime.now()` internally, truly deterministic testing requires either extracting a clock abstraction or accepting that this particular assertion covers only the trivial "expired → zero" branch. The "expired" test (line 357) is fine; the "approximately correct" test is the flaky one.
- [x] **[HIGH]** `correction_capture_service_test.dart:638–651` (`correctionCaptureServiceProvider` group) — **Constructor smoke test.** The only assertion is `expect(service, isA<CorrectionCaptureService>())`. This verifies the provider resolves without throwing, but tests zero behavior. Either remove it or replace with a meaningful check such as verifying that the returned service shares the same `categoryRepository` by calling `captureCorrection` and checking it routes to the mock.
- [x] **[MED]** `correction_capture_service_test.dart` — **Repeated `ProviderContainer` setup.** At least 8 tests manually create a `ProviderContainer` with the same `categoryRepositoryProvider.overrideWithValue(mockCategoryRepository)` override. This is non-DRY boilerplate. Extract a helper:
  ```dart
  ProviderContainer _makeContainer() => ProviderContainer(
    overrides: [
      categoryRepositoryProvider.overrideWithValue(mockCategoryRepository),
    ],
  );
  ```
  and call `addTearDown(container.dispose)` immediately after creation.
  - **RESOLVED:** done — extracted `makeRepoContainer()` (repo override + auto-dispose) and replaced all 9 inline `ProviderContainer` constructions.
- [x] **[MED]** `correction_capture_service_test.dart:547–581` (`setPending starts timer and cancel prevents onSave`) — uses real async (`async {}`-less test body). The test does not advance fake time, so the save timer is never checked — the assertion `expect(saveCalled, isFalse)` passes only because the timer has not fired yet in wall-clock time. Wrap in `fakeAsync` and elapse past the delay to confirm the cancel truly suppresses the save even after the delay would have elapsed.
  - **RESOLVED:** done — the test now runs under `fakeAsync`, cancels, then elapses past `kCorrectionSaveDelay`: the cancelled timer never fires, so `saveCalled` stays false by suppression rather than by outrunning the wall clock.
- [ ] **[LOW]** `correction_capture_service_test.dart:306–320` (`CorrectionCaptureResult` group) — pure enum-existence check. While not harmful, it adds no behavioral coverage; the values are exercised implicitly by every `captureCorrection` test. Consider removing this group and noting coverage is sufficient through the behavioral tests.

## Generative (Glados) testing opportunities

- [x] **[MED]** `_isMeaningfulCorrection(before, after)` is a pure function with two string inputs and a boolean output governed by two rules: (a) length < 3 AND case-insensitive equality → false; (b) all other cases → true. A Glados property could exhaustively verify both invariants across arbitrary string pairs, especially edge cases around the `< 3` length boundary. Example property: for any string pair where `before.length >= 3 || before.toLowerCase() != after.toLowerCase()`, `_isMeaningfulCorrection` returns `true`. This is exposed indirectly via `captureCorrection`, but the logic is non-trivial enough to warrant a direct property.
  - **RESOLVED:** done — added a `@visibleForTesting debugIsMeaningfulCorrection` seam and a Glados2 property over mixed-case string pairs (numRuns 150, `tags: 'glados'`) pinning the rule `meaningful ⇔ length ≥ 3 || case-insensitive difference`, plus explicit boundary cases at lengths 2 and 3.
- [ ] **[LOW]** `_isDuplicate(existing, before, after)` is also pure. A Glados property verifying that a round-trip (add an example, then check for duplicate) always returns `true` would guard against future refactors.

## Coverage / missing-behavior gaps

- [x] **[MED]** `_saveCorrection` is only reachable via the timer path. All timer-path tests drive it through `correctionCaptureServiceProvider` (full provider chain). There is no test that verifies `_saveCorrection` correctly builds the `ChecklistCorrectionExample` with the exact `capturedAt` field set to approximately `DateTime.now()` (not a fixed value). Since `capturedAt` uses `DateTime.now()` inside `_saveCorrection`, this is another real-time dependency that should be documented as a known gap or addressed with a clock abstraction.
  - **RESOLVED:** done — root-cause fix: both `DateTime.now()` calls in the service now read `clock.now()`, and a new test pins the saved example's `capturedAt` to `pinnedBase + kCorrectionSaveDelay` under `withClock` + `fakeAsync` (the stamp lands exactly when the save timer fires).
- [x] **[MED]** `updateActivity` and `lastActivity` in `UserActivityService` (a different feature, but referenced from the checklist test infrastructure indirectly) — see user_activity review. For checklist itself: there is no test for the case where `captureCorrection` is called concurrently (two rapid calls with different data). The `_nextId` static counter on `PendingCorrection` is not reset between tests, which means test ordering affects `id` values. While equality is `id`-based, this is not a flakiness risk but is worth noting.
  - **RESOLVED:** done — added `'a second capture replaces the first; only the second saves'`: two rapid `captureCorrection` calls inside one save window, elapse past the delay, and exactly one repository update carrying the second pair (setPending cancelled the first timer).
- [ ] **[LOW]** The `CorrectionCaptureService` constructor with `notifier: null` is tested (`does not set pending when no notifier provided`), but there is no test verifying that `notifier?.setPending(...)` is the **only** side effect path — i.e., if `notifier` is null the correction is still considered `pending` (correct), but no timer is started. This is covered implicitly by the "no notifier" test, but a comment in the test would make this invariant explicit.
