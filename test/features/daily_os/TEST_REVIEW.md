## Recurring patterns

- **pumpAndSettle overuse** — the feature has >120 `pumpAndSettle` calls across widget tests; the majority follow provider state changes or simple taps with no running animations. Every widget sub-dir is affected; `add_budget_sheet_test` (38 calls) is the worst offender.
- **Fake-time policy violations** — `Future.delayed(Duration.zero)` and real `timeout(Duration(seconds: 5))` patterns appear in `state/`, `repository/`, and `ui/pages/`; together they create flakiness risk and up to 20 s real-timer exposure per CI run.
- **Inline GetIt boilerplate** — raw `getIt.isRegistered`/`unregister`/`registerSingleton` patterns appear in at least 4 test files instead of `setUpTestGetIt()` from `widget_test_utils.dart`.
- **Smoke-only assertions** — `findsOneWidget(WidgetType)` as the sole assertion appears in ~15 tests across `ui/widgets/` and `ui/pages/`; these only prove the widget tree builds without crashing.
- **Non-DRY mock setup** — the pattern of repeating full `when(...)` stub chains in every test body (most severely in `unified_daily_os_data_controller_test.dart` with 93 repetitions) appears throughout the feature; extracting shared `_stubDefaultMocks()` helpers is the universal fix.
- **Copy-paste test permutations** — `TaskStatus` icon tests (×7 in `time_budget_card_test.dart`), `DailyOsState` property tests (×35 in `daily_os_controller_test.dart`), DST boundary tests (×6 in `time_history_header_controller_test.dart`), `progressFraction` cases (×7 in `time_budget_progress_controller_test.dart`) all repeat nearly-identical test bodies differing by a single parameter — all should be parameterized loops.
- **Missing mirror test files** — `set_time_blocks_mock_data.dart`, `zigzag_fold_indicator.dart`, `day_label_chip.dart`, `today_button.dart` all lack dedicated test files in violation of the one-test-per-source-file rule.
