## Generative (Glados) testing opportunities

- [ ] **[MED]** `unified_daily_os_data_controller.dart` — `_calculateStatus(Duration planned, Duration recorded)` (lines 769–784). Pure function: given two `Duration` values, returns a `BudgetProgressStatus`. Has 4 output categories separated by thresholds (negative → `overBudget`; zero → `exhausted`; ≤15 min remaining → `nearLimit`; else → `underBudget`). A Glados property can assert:
  1. `recorded > planned ↔ overBudget`
  2. `recorded == planned ↔ exhausted`
  3. `0 < planned − recorded ≤ 15min ↔ nearLimit`
  4. `planned − recorded > 15min ↔ underBudget`
  with generators for arbitrary non-negative durations. This is a pure function entirely isolated from I/O.

- [ ] **[MED]** `time_history_header_controller.dart` — `_computeStackedHeights(List<DayTimeSummary> days, List<String> categoryOrder, Duration maxTotal)` (lines 489–528). Pure: given a list of days and a max, produces per-day cumulative height fractions. Algebraic invariants suitable for Glados:
  1. For any day and category, `heights[cat] ∈ [0.0, 1.0]`.
  2. The final cumulative value after all categories ≤ 1.0 (doesn't overflow).
  3. Category ordering is preserved (heights are non-decreasing along the stack).
  4. When `maxTotal == Duration.zero`, result is `{}`.
  This function currently has only two concrete integration tests and one edge case; a property test would substantially improve confidence.

- [ ] **[LOW]** `unified_daily_os_data_controller.dart` — `_calculateDayStartHour` and `_calculateDayEndHour` (lines 786–838). Pure functions on sorted `List<PlannedTimeSlot>` + `List<ActualTimeSlot>`. Simple invariants:
  - `result ∈ [0, 24]`
  - `_calculateDayStartHour` never exceeds the earliest `.startTime.hour`
  - `_calculateDayEndHour` is at least the latest `.endTime.hour + 1` (clamped to 24)
  Currently tested only through integration scenarios; a Glados property over random slot times would be lightweight.

## Test execution speed opportunities

- [ ] **[HIGH]** `daily_os_controller_test.dart` lines 904, 989, 1 093, 1 150 — **Four `activeFocusCategoryId` tests use real async wait via `.timeout(const Duration(seconds: 5))`**. Each test opens a stream, creates a `Completer`, adds a real 5-second timeout, and awaits the completer. When the stream emits immediately (as in all four tests), the wait resolves quickly, but the 5-second timeout still creates a real `Timer` in the test zone, which can interact badly with CI shard timing and causes test flakiness if the provider is slow to emit. These should be rewritten using `fakeAsync`/`withClock` + `flushMicrotasks()`, exactly as done for the `does not throw when disposed during polling delay` test at line 1 161.

  Estimated impact: 4 tests × up to 5-second wall-clock exposure per run = **up to 20 seconds saved per CI shard** in the worst case; eliminates potential flakiness.

- [ ] **[HIGH]** `daily_os_controller_test.dart` line 850 — `await Future<void>.delayed(Duration.zero)` in the `RunningTimerCategoryId / updates when timer stream emits` test. This is an explicit fake-time policy violation. Replace with `await Future<void>.value()` or restructure with `fakeAsync` + `flushMicrotasks()`.

- [ ] **[HIGH]** `time_history_header_controller_test.dart` line 446 — `await Future<void>.delayed(Duration.zero)` in `does not load more when already loading` test. Same violation; replace with `fakeAsync` + `flushMicrotasks()` or an await-the-futures pattern.

- [ ] **[MED]** `unified_daily_os_data_controller_test.dart` — All 93 tests spin up a `ProviderContainer` with `getIt` singleton registration. Because each test repeats the full `setUp`/`tearDown` with `getIt.reset()`, the per-test overhead for container creation + GetIt reset is higher than necessary. Grouping related tests that share the same mock configuration into sub-`group`s with a shared `setUp` would reduce redundant work. The Mutation group already demonstrates this pattern with `setupBasicMocks(plan)`.

- [ ] **[MED]** `time_history_header_controller_test.dart` — DST group (lines 864–1 233): 6 tests each call `container.read(timeHistoryHeaderControllerProvider.future)` and build 60 `DayTimeSummary` objects with zero entries. These are heavyweight provider builds for what is essentially a calendar-math check. Extracting the day-generation logic into a testable pure function and testing it directly (without a full provider container) would be ~10× faster per test and more focused.

- [ ] **[LOW]** `time_history_header_controller_test.dart` — DST tests (lines 864–1 233) are structurally near-identical (same mock setup, same assertions, only the `dstDay` and `expectedMonthDay` differ). This is both a quality issue (copy-paste) and a speed issue (6 full provider builds). A parameterized loop over `(testDateString, expectedDstDay, description)` tuples would run the same coverage in a single provider build cycle.

## Summary

- **1 oversized impl file** (`unified_daily_os_data_controller.dart`, 1 022 lines): clear split into lifecycle, data-aggregation, and mutation layers.
- **3 oversized test files**: `unified_daily_os_data_controller_test.dart` (4 821 lines), `time_history_header_controller_test.dart` (1 799 lines), `daily_os_controller_test.dart` (1 691 lines); each has concrete split seams documented above.
- **6+ weak or smoke-only test patterns**: 5 timer-subscription tests (`verify(getStream).called(greaterThan(0))`), 2 tests asserting only `hasValue`, and 1 midnight-crossing test with an under-specified assertion. All require real state-change assertions.
- **3 Glados candidates**: `_calculateStatus` (pure 4-branch threshold function), `_computeStackedHeights` (pure ratio computation with stackable invariants), `_calculateDayStart/EndHour` (pure clamping logic).
- **4 speed wins** (fake-time policy violations): 4 `activeFocusCategoryId` tests using real 5-second `timeout`, 1 `Future.delayed(Duration.zero)` in `RunningTimerCategoryId` test, 1 `Future.delayed(Duration.zero)` in `time_history_header_controller_test`; together eliminate up to ~20 s real-timer exposure per CI run and remove flakiness risk.
- **Biggest single opportunity**: Extract the data-aggregation logic from `UnifiedDailyOsDataController` into a plain-Dart `DailyOsDataAggregator` class — this would make all budget-progress, timeline-building, and task-progress tests run without a `ProviderContainer`, slashing test boilerplate and execution time for the majority of the 93-test file.
