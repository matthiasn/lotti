## Generative (Glados) testing opportunities

- [x] **[HIGH]** `_CaptureLayoutMetrics.resolve` in `capture_page.dart` (lines 167-200) is a pure static factory. It has three deterministic invariants for any valid `(viewportHeight, phase)` input: (1) `stateSlotHeight` is always within `[minimumSlotHeight, maximumSlotHeight]`; (2) `liveTranscriptLineCount` is in `[3, 7]`; (3) `reviewTranscriptLineCount` is at least `minimumReviewTranscriptLineCount`. Currently only tested implicitly via widget tests at specific viewport sizes. Extracting `_CaptureLayoutMetrics` to its own file would enable a proper Glados property test tagged `glados`. **RESOLVED:** exposed as `CaptureLayoutMetrics` via `@visibleForTesting` (no file split needed); Glados property test pins slot-height clamp + line-count ranges over (viewport, phase).
- [ ] **[MED]** `dailyOsNextRoutePath` / `parseDailyOsNextRouteDate` in `daily_os_next_routes.dart` — the round-trip property `parseDailyOsNextRouteDate(dailyOsNextRouteDate(d)) == DateTime(d.year, d.month, d.day)` is a clean Glados candidate using `any.dateTime`. The current test covers only concrete examples.

Exercising arbitrary `(date, clock)` pairs via Glados would cover the edge cases at month/year boundaries.

## Test execution speed opportunities

- [x] **[HIGH]** `capture_page_test.dart` — 8 `tester.runAsync(() async {})` zero-delay calls are used to flush real async I/O in four tests that drive the full `CaptureController` recording chain (mic permission, `startRecording`, `transcribe`). These tests touch `Directory.systemTemp.createTempSync` and mock audio services. The page-level tests for these code paths could instead use `_StubCaptureController` (already defined at line 1281) and inject a pre-baked `CaptureState.captured` without any real I/O, cutting test time materially. The real recording chain is better exercised in the `CaptureController` unit test. **RESOLVED:** all 8 zero-delay `runAsync` calls removed — the three submission-path tests now use `_StubCaptureController` with pre-baked `CaptureState.captured`; the recording chain stays covered by the controller unit tests.
- [ ] **[MED]** `shutdown_page_test.dart` — 11 tests each call `tester.pump(const Duration(milliseconds: 200))` to let the mock agent resolve. The `MockDayAgent` used in this test has `parseLatency: Duration.zero` already, so the `200 ms` pump is conservative. Verify whether `tester.pump()` alone (single frame) suffices; if so, each test saves one real-clock wait.
- [ ] **[MED]** `refine_page_test.dart` (1158 lines) — the heavy test file results in longer parse+compile time in CI. Splitting the boilerplate `_RecordingAgent` into a shared helper (see Test quality section) reduces the per-file overhead.
- [ ] **[LOW]** `day_page_test.dart` (806 lines) — each test independently constructs both `_RecordingAgent` and provider overrides. A `setUp`/`setUpAll` that pre-constructs the agent where the diff is the same across tests would eliminate repeated construction.
