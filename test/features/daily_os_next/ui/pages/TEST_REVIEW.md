## Test execution speed opportunities

- [x] **[HIGH]** `capture_page_test.dart` — 8 `tester.runAsync(() async {})` zero-delay calls are used to flush real async I/O in four tests that drive the full `CaptureController` recording chain (mic permission, `startRecording`, `transcribe`). These tests touch `Directory.systemTemp.createTempSync` and mock audio services. The page-level tests for these code paths could instead use `_StubCaptureController` (already defined at line 1281) and inject a pre-baked `CaptureState.captured` without any real I/O, cutting test time materially. The real recording chain is better exercised in the `CaptureController` unit test. **RESOLVED:** all 8 zero-delay `runAsync` calls removed — the three submission-path tests now use `_StubCaptureController` with pre-baked `CaptureState.captured`; the recording chain stays covered by the controller unit tests.
- [x] **[MED]** `shutdown_page_test.dart` — 11 tests each call `tester.pump(const Duration(milliseconds: 200))` to let the mock agent resolve. The `MockDayAgent` used in this test has `parseLatency: Duration.zero` already, so the `200 ms` pump is conservative. Verify whether `tester.pump()` alone (single frame) suffices; if so, each test saves one real-clock wait.
  **RESOLVED (assessed, no change):** `tester.pump(Duration)` advances FAKE time — there is no real-clock wait to save. The 200 ms pump robustly drains the zero-latency mock's timer chain; trimming it to bare pumps risks under-pumping for zero benefit.
- [x] **[MED]** `refine_page_test.dart` (1158 lines) — the heavy test file results in longer parse+compile time in CI. Splitting the boilerplate `_RecordingAgent` into a shared helper (see Test quality section) reduces the per-file overhead.
  **RESOLVED (stale):** the shared `RecordingDayAgent` extraction already happened; `refine_page_test.dart` is down to 998 lines.
- [x] **[LOW]** `day_page_test.dart` (806 lines) — each test independently constructs both `_RecordingAgent` and provider overrides. A `setUp`/`setUpAll` that pre-constructs the agent where the diff is the same across tests would eliminate repeated construction.
  **RESOLVED (adapted):** a shared `setUp`-level agent is unsafe here — several tests need per-test parameterised agents (`renameError`, `diff`/`acceptedPlan`) and assert on per-test counters, so a single mutable agent would leak state. Instead extracted `_pumpDayPage(tester, {draft, agent, extraOverrides})` (set surface + wrap DayPage with the `dayAgentProvider` override + first frame), folding the repeated `_wrap(... overrides: [dayAgentProvider.overrideWithValue(agent)] ...)` + `pump` boilerplate across 7 tests. Tests with bespoke hosting (Builder route, `agent_providers.dayAgentProvider`/NavService, multi-pump rename closure) keep their own setup.
