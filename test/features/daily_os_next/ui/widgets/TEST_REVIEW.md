## Generative (Glados) testing opportunities

- [ ] **[HIGH]** `_TimelineFoldingState.positionForHourValue` (`day_timeline.dart:665`) is pure math with a clear algebraic invariant: for any two `hourValue` inputs `a < b` (both in `[startHour, endHour]`), `positionForHourValue(a, px) <= positionForHourValue(b, px)` (monotonicity). Additionally, `positionForHourValue(startHour, px) == 0` and `positionForHourValue(endHour, px) == totalHeight(px)` (boundary exactness). After extracting to `day_timeline_folding.dart`, these become straightforward Glados properties requiring `any.double` generators bounded to `[0, 24]` and `any.positiveDouble` for `pxPerMinute`.

- [ ] **[HIGH]** `_TimelineFoldingState.fromBlocks` cluster-merge algorithm (`day_timeline.dart:506–621`): the invariant is "every hour in the `occupiedHours` set must fall inside at least one `_TimelineVisibleRegion` in the output `segments` list". A Glados test generates random sets of hour-range pairs and asserts the invariant. A secondary property: the sum of all segment heights via `totalHeight(px)` must equal `(endHour - startHour) * 60 * px` only for an all-visible state; for folded states it must be strictly less (compressibility invariant).

- [ ] **[MED]** `_TimeSpentSummary._formatMinutes` / `_ActualCategoryBars._formatMinutes` (`day_timeline.dart:841–847` and `914–920`) — both are identical pure functions. A Glados property: `_formatMinutes(60 * h + m)` round-trips correctly: if `h == 0` result ends with `'m'`; if `m == 0` result ends with `'h'`; otherwise result contains `'h '` and `'m'`. Use `any.nonNegativeInt` bounded to e.g. `0–1439`. Note: extracting the duplicate into a shared top-level function is also a worthwhile DRY fix.

- [ ] **[LOW]** `_FoldRegionToggle._formatFoldRange` (`day_timeline.dart:1317–1325`) — pure formatter with the special case `hour == 24`. A Glados property using `any.int` in `[0, 24]` verifies the formatted output always matches `HH:00-HH:00` and that `24` formats as `24:00` (not `00:00`).

## Test execution speed opportunities

- [x] **[HIGH]** `processing_category_filter_button_test.dart` — 5 occurrences of `await tester.pump(const Duration(milliseconds: 600))` (lines 55, 80, 114, 144, 169). This pump likely waits for a Wolt bottom-sheet close animation. If the animation can be short-circuited in tests (e.g., by reducing the animation duration via `timeDilation` or replacing the sheet with a `ProviderScope` override that skips animation), each test would save ~600 ms of virtual time. At 5 tests, that is ~3 seconds of virtual time per full test run. Impact: moderate. **RESOLVED (assessed, no change):** `tester.pump(Duration)` advances *virtual* time instantly inside the test binding — these pumps add no wall-clock cost, so there is nothing to save.

- [ ] **[MED]** `day_timeline_test.dart` — 29 tests each invoke a full widget build of `DayTimeline` (1832-line class with multiple nested `LayoutBuilder`s, a `BackdropFilter`, and a `SingleChildScrollView`). Once the file is split and the layout-math classes are isolated, a subset of tests can target `_TimelineFoldingState` directly as unit tests (no widget pump at all), dramatically reducing execution overhead for the coverage-critical math paths.
