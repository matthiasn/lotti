## Generative (Glados) testing opportunities

- [x] **[HIGH]** `lib/features/dashboards/state/health_data.dart` — `aggregateDailySum` and `aggregateDailyMax` are pure functions on lists of `QuantitativeEntry`. Glados can verify invariants like: sum of all daily buckets equals sum of all input values (for `aggregateDailySum`); each bucket value ≥ any single input for the same day (for `aggregateDailyMax`); output length ≤ number of distinct calendar days in input. No I/O, no UI. (`lib/features/dashboards/state/health_data.dart:60-127`)

- [x] **[HIGH]** `lib/features/dashboards/state/health_data.dart` — `colorByValueAndType` threshold selection is non-trivial pure logic: for any observation value and sorted threshold map, the returned color must be the CSS hex mapped to the highest threshold that is ≤ the value. A Glados property can verify this invariant across arbitrary `{threshold: color}` maps and arbitrary values. (`lib/features/dashboards/state/health_data.dart:11-34`)

- [ ] **[MED]** `lib/features/dashboards/state/workout_data.dart` — `aggregateWorkoutDailySum` fills days in range with zeros even when no workout entries exist, then accumulates. A generative test can assert: (a) output length always equals `rangeEnd.difference(rangeStart).inDays`; (b) sum of all output values equals sum of matching input entries; (c) entries outside the range do not appear in output. (`lib/features/dashboards/state/workout_data.dart:8-62`)

- [ ] **[MED]** `lib/features/dashboards/state/survey_data.dart` — `aggregateSurvey` extracts a single score key from heterogeneous entity lists. Glados can verify that for any list of mixed entity types, only `SurveyEntry` items contribute to output, and only those with the specified key. (`lib/features/dashboards/state/survey_data.dart:41-66`)

- [ ] **[LOW]** `lib/features/dashboards/state/health_data.dart` — `transformToHours` is a pure map (divide by 60). Glados round-trip: `transformToHours(obs).map((o) => o.value * 60)` ≈ original values for any non-empty list. Trivial to implement and documents the contract. (`lib/features/dashboards/state/health_data.dart:100-107`)

## Test execution speed opportunities

- [x] **[HIGH]** `test/features/dashboards/ui/pages/dashboards_list_page_test.dart` makes **14 `pumpAndSettle()` calls** across 9 test cases, including after every `tap`. Each call runs the full animation pipeline with a 10-second timeout. Replace most with `tester.pump()` (for immediate state changes) or `tester.pump(const Duration(milliseconds: 50))` (for short animations). (`test/features/dashboards/ui/pages/dashboards_list_page_test.dart:94,107,128,200,250...`) **RESOLVED:** 13 of 14 settles replaced with bounded pumps (100ms after page loads, 600ms around bottom-sheet open/close); the one kept settle drains flutter_animate timers after disposing the detail page.
