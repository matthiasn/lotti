## Test execution speed opportunities

- [x] **[HIGH]** `test/…/lists/design_system_list_item_test.dart` lines 370, 372, 403, 405 — four `pumpAndSettle()` calls in the `onHoverChanged` hover tests. Mouse-enter/exit events are synchronous state changes (no animations); each `pumpAndSettle()` carries a 10-second default timeout. Replace all four with `await tester.pump()`. Impact: each `pumpAndSettle` can stall up to 10 s if an unexpected animation is present; the standard CI shard this falls in is timing-sensitive. **RESOLVED:** the four hover settles are plain pump() now.
- [ ] **[MED]** `test/…/calendar_pickers/design_system_time_calendar_picker_test.dart` lines 201–203, 210–213, 228–229, 356–357, 362–363 — uses `await tester.pump(const Duration(milliseconds: 250))` twice per test to let the overlay animation settle after tapping the month picker. This is correct behaviour for an animated overlay, but the 250 ms is a magic constant. Document it (e.g. `// wait for overlay slide-in animation`) and consider whether `tester.pumpAndSettle()` would be cleaner now that it is known to settle.
- [ ] **[LOW]** `test/…/calendar_pickers/design_system_calendar_picker_test.dart` — several tests rebuild a large `DesignSystemCalendarPicker` with a full month's worth of `DesignSystemCalendarDayCellData` objects for every individual cell variant. A shared `_buildMinimalPicker({required Widget cell})` helper could reduce boilerplate and keep per-test build sizes small.
- [ ] **[LOW]** `test/…/buttons/design_system_floating_action_button_test.dart` — the `pumpFab` helper is `public` (no leading `_`). It should be private since it is file-local. No runtime impact but signals it was perhaps intended for reuse.
