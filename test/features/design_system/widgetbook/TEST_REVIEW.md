## Test execution speed opportunities

- [x] **[HIGH]** `design_system_calendar_picker_widgetbook_test.dart` — 6 `pumpAndSettle()` calls across 6 test cases (lines 68, 80, 107, 116, 131, 158), each with the default 10-second timeout. If any animation never settles (e.g., a month-rail scroll that loops), this can add minutes to the suite. Replace each with `await tester.pump(const Duration(milliseconds: 300))` scoped to the actual animation duration. Estimated worst-case impact: up to 60 seconds of extra wait time if an animation timer fires repeatedly. **RESOLVED:** all six settles are bounded pumps (pump + 300ms).

- [x] **[MED]** `design_system_task_filter_widgetbook_test.dart` — two `pumpAndSettle()` calls (lines 130, 149) inside the modal-open test. These are used to wait for the dialog open/close animation. Replace with `tester.pump(const Duration(milliseconds: 250))` matching the animation duration actually used by the modal.
  **RESOLVED:** done — both settles replaced with `pump()` + `pump(300ms)` bounded pairs; all 5 tests pass.

- [ ] **[LOW]** `design_system_toast_widgetbook_test.dart` — the cleanup pump is `await tester.pump(const Duration(seconds: 10))` which is correct but heavy. If the countdown bar duration is shorter than 10 seconds, this can be reduced to the actual animation duration (e.g., 3 seconds) to save real clock time.
