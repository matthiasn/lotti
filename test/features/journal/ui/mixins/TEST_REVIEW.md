## Test execution speed opportunities

- [x] **[HIGH]** `highlight_scroll_mixin_test.dart:402-410` — 'retry logic handles missing context and logs warning' runs a manual loop: `for (var i = 0; i < 35; i++) await tester.pump(const Duration(milliseconds: 120))` = **35 pump calls × 120 ms = 4,200 ms of virtual time**. In test mode `maxScrollRetries=5` and `scrollRetryDelay=50 ms`, so the actual time needed is only 5 attempts × (50 ms delay + one post-frame) ≈ **250-300 ms total**. The loop overshoots by ~14×. Replace with `for (var i = 0; i < 7; i++) await tester.pump(const Duration(milliseconds: 60))` (7 × 60 ms = 420 ms, safely past the 250 ms needed) plus one final `tester.pump()` to drain post-frame callbacks. Estimated saving: **~3.8 s per run** of this test.

This is correct and necessary (the highlight timer is 4800 ms), but it accounts for 4.8 s of virtual time per test run. The duration is hardcoded in the impl as `const Duration(milliseconds: 4800)`. Consider whether the `highlightDuration` getter should be overridable in test harness classes so tests can use a much shorter value (e.g., 50 ms) to achieve the same behavioral assertion without the wall-clock cost.

- [ ] **[MED]** `highlight_scroll_mixin_test.dart` has **9 `pumpAndSettle()` calls**. Several are in tests where the scroll completes in one post-frame cycle (context is immediately available, no retry). Replace these with `tester.pump()` + `tester.pump()` (two frames: one for the Timer, one for the post-frame callback). Affected tests: 'scrollToEntry calls onScrolled callback' (line 247), 'scrollToEntry prevents duplicate concurrent scrolls' (line 282), 'scrollToEntry sets highlighted entry' (line 308), 'scrollToEntry accepts different alignment values' (line 377), 'concurrent scroll to different entry cancels previous' (line 558).
