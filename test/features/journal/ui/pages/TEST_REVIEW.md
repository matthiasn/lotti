## Test execution speed opportunities

- [x] **[HIGH]** `infinite_journal_page_test.dart:162-172` — `pumpWithDelay` pumps `pump()` + `pump(500ms)` + 10×`pump(100ms)` = **1,600 ms of virtual time per call**. It is called 9 times = **~14.4 s of virtual-time overhead** per test run of this file. The comment says "replace pumpAndSettle" but the chosen approach over-pumps heavily. Each `pumpWithDelay` call should be replaced with a targeted `tester.pump()` + `tester.pump(const Duration(milliseconds: 16))` pair (one event loop + one frame), since all data comes from synchronous mock stubs. Estimated saving: **≥10 s per full test run** of this file.

- [x] **[HIGH]** `entry_details_page_edge_cases_test.dart` — **14 `pumpAndSettle()` calls** (the highest count of any file in scope). For tests whose mocks deliver synchronous streams (`Stream.fromIterable`), each `pumpAndSettle()` can run for the full 10-second default if any pending animation or timer doesn't settle. Many of these are in scroll-interaction tests where `tester.pump(const Duration(milliseconds: 100))` after the drag would suffice. Estimated saving: significant per-test-run, especially in CI. **RESOLVED (file gone):** `entry_details_page_edge_cases_test.dart` was merged away; the consolidated `entry_details_page_test.dart` runs its 19 tests in ~1s, and its remaining settles drain the page's real fadeIn/route animations.

- [x] **[MED]** `entry_details_page_test.dart` — **6 `pumpAndSettle()` calls**. Same root cause: synchronous mocks don't need full animation settlement. The drag-based scroll test (lines 408-434) and the DropTarget tests are the most justified uses; the simple rendering tests (lines 149-208) could use `tester.pump()` × 2.
  **RESOLVED:** done — all 26 `pumpAndSettle` calls replaced with bounded `pump()` + `pump(300ms)` pairs; full file passes.

- [x] **[MED]** `entry_details_page_auto_scroll_test.dart` — **5 `pumpAndSettle()` calls** plus a manual retry loop at lines 166-168 (`for (var i = 0; i < 10 ...) await tester.pump(60ms)`). This loop pumps up to 600 ms of virtual time and terminates early if the intent clears. The `pumpAndSettle()` at line 162 immediately after should be folded into the loop or replaced with targeted pumps driven by the retry count documented in the mixin (`maxScrollRetries=5`, `scrollRetryDelay=50ms` → 250 ms total).
  **RESOLVED:** done — subsumed by the bulk conversion: the settle preceding the retry loop is now bounded pumps and the loop (up to 600 ms virtual, early-exit on intent clear) remains the documented driver for the mixin's 5×50 ms retry schedule.

- [x] **[LOW]** `entry_details_page_timer_highlight_test.dart` — 7× `await Future.delayed(Duration.zero)` in unit `test(...)` blocks. These are real-time yields, violating the fake-time policy. ## Summary

- **2 source files** covered by **6 test files** — both are "one source → multiple test files" violations (entry_details_page: 4 files; infinite_journal_page: 2 files). These must be consolidated.
- **0 oversized impl files** (189 and 118 lines); `infinite_journal_page_test.dart` at 585 lines is above the soft threshold and should be reduced during consolidation.
- **~12 weak tests** across edge_cases and auto_scroll: sole assertion is `findsOneWidget` on the page type or `expect(true, isTrue)`.
- **1 misscoped test file** (`entry_details_page_timer_highlight_test.dart`): all five `test()` cases test `FakeTimeService` stream behavior, not the page widget.
- **0 Glados candidates** at the pages level (UI widgets; logic is in providers/mixin).
- **Biggest speed win**: replace `pumpWithDelay` in `infinite_journal_page_test.dart` (9 calls × 1.6 s each = ~14 s saving) and reduce the 14 `pumpAndSettle` calls in `entry_details_page_edge_cases_test.dart`.
- **Biggest quality win**: consolidate the four `entry_details_page_*_test.dart` files into one, eliminating the triplicated 80-line `setUp` blocks and converting smoke-only tests into behavioral assertions; add a `Checklist` branch test to close the only meaningful coverage gap in the impl.
