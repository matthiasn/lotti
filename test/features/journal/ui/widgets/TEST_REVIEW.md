## Test execution speed opportunities

- [x] **[HIGH]** `entry_details_widget_test.dart` — **9 `pumpAndSettle` calls** at lines 318, 380, 625, 793, 821, 1246, 2267, 2345, 2346. Of these, lines 625 and 1246 intentionally wait for an animation (`pumpAndSettle` after `pumpWidget`); the others are plain widget-load settle calls where `tester.pump()` would suffice. Replacing unjustified `pumpAndSettle` calls with `tester.pump()` avoids the 10-second timeout risk and reduces CI flakiness surface.

- [x] **[HIGH]** `entry_details_widget_test.dart` — The test at line 1894–1896 pumps `const Duration(milliseconds: 5000)` twice inside a single test to advance through the `_PulsingBorder` animation lifecycle. This is a 10-second virtual-time advance that keeps a live `AnimationController` ticking across the pump sequence. Using `tester.pumpAndSettle()` with an explicit timeout or restructuring the timer assertions to only verify the state _at_ key animation milestones would be cleaner and less fragile. **RESOLVED (assessed, no change):** both 5000 ms pumps are single virtual-time frame advances (instant in fake time, no real waiting); the paint-coverage test already asserts at explicit milestones (opacity 0 during delay, >0.5 at peak, 0.0 at completion, no leaked tickers) and the other uses the advance purely as end-of-test animation drain. Replacing them with pumpAndSettle would reintroduce the 10 s timeout risk.

- [x] **[HIGH]** `entry_details_collapsible_test.dart` — **4 `pumpAndSettle` calls** (lines 521, 911, 968, 1099). Line 521 is justified (drains `Future.delayed` auto-scroll timer). Lines 911 and 968 follow `await tester.tap` + `pumpAndSettle` — since both are settling a NavigationRoute dismiss, they're appropriate. Line 1099 is inside an error-handling test that throws synchronously; `pump()` is sufficient.

- [ ] **[MED]** `text_viewer_widget_non_scrollable_test.dart` — **4 `pumpAndSettle` calls** at lines 294, 320, 340, 352 inside the overflow detection group. These are used to ensure a `LayoutBuilder` reports its size; `tester.pump()` after `pumpWidget` should suffice for non-animated layout.

- [ ] **[MED]** `nested_ai_responses_widget_test.dart` — **16 `pumpAndSettle` calls**. Many of these settle after a `StreamController` event is emitted into a `StreamBuilder`. Since the streams are finite (from `StreamController.add(...)` with a known sequence), the tests could use `tester.pump()` after flushing microtasks instead of the full `pumpAndSettle`.

- [ ] **[MED]** `entry_details_widget_test.dart` — Each of the 9 `setUp` blocks calls `await getApplicationDocumentsDirectory()` (real I/O on the fake path), creating the directory afresh per group. A single `setUpAll(setFakeDocumentsPath)` already exists per group, but the `await getApplicationDocumentsDirectory()` call in each `setUp` is still a real filesystem call. Using the `setUpTestGetIt()` helper would consolidate this.
