The preferred pattern per `test/README.md` is a finite stream or explicit subscription flush; the `Duration.zero` delays are fake-time-policy violations (no `fakeAsync` wrapper). **Affects** stream reliability; impact: intermittent flakiness in CI shards.

Candidate properties:
  - Any ID in `_overviewNotificationTokens` always triggers refresh.
  - A `projectEntityUpdateNotification(id)` triggers refresh iff `id` is already in the snapshot.
  - Unrecognised IDs never trigger refresh.
  - Adding the `propagatedNotification` wrapper of a token already in the set does not change the outcome.
  This function has no I/O and has structured inputs; adding 80–120 Glados runs would catch any prefix-stripping regression.
  **RESOLVED:** done — `_projectsOverviewNeedsRefresh` got a `@visibleForTesting debugProjectsOverviewNeedsRefresh` seam; a Glados property (numRuns 150) asserts all four candidate invariants per seed-built snapshot: every broad token refreshes, member project ids refresh identically bare or `PROJECT_ENTITY_UPDATE:`-prefixed, the category id refreshes, and unrecognised ids (even prefixed) never do.
