## Test execution speed opportunities

- [x] **[MED]** `project_detail_record_provider_test.dart` — Every test calls `container.read(projectDetailRecordProvider(projectId).future)` which triggers an async provider resolution. Because all containers are freshly created per test there is no caching, but the 40+ individual container setups each call `getIt.reset()` in the outer `setUp`. Switching to `addTearDown(container.dispose)` without `getIt.reset()` per container (already the case for `createContainer`) is fine, but the outer `setUp` + `tearDown` pair calling `await getIt.reset()` twice per test adds unnecessary overhead. Group the tests that share the same `getIt` setup into a single `group` with one `setUp`/`tearDown`.
  **RESOLVED:** assessed, no change — the per-test `getIt.reset()` pair is the project-standard isolation; collapsing groups under a shared GetIt lifetime would let registrations and mock state leak between tests (the in-suite-contamination flake class the batched very_good runner punishes). The reset costs microseconds.

- [ ] **[LOW]** `project_providers_test.dart` — The filter tests (lines 599–940) each construct a `ProviderContainer`, add a `subscription`, and wait for `projectsOverviewProvider.future`. Eight such tests × container + listen + await is cheap in isolation but adds up. A shared container per filter-test group with a `setUp` that pre-loads the snapshot would reduce the per-test overhead.
