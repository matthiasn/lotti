## Test execution speed opportunities

- [ ] **[MED]** `project_detail_record_provider_test.dart` вЂ” Every test calls `container.read(projectDetailRecordProvider(projectId).future)` which triggers an async provider resolution. Because all containers are freshly created per test there is no caching, but the 40+ individual container setups each call `getIt.reset()` in the outer `setUp`. Switching to `addTearDown(container.dispose)` without `getIt.reset()` per container (already the case for `createContainer`) is fine, but the outer `setUp` + `tearDown` pair calling `await getIt.reset()` twice per test adds unnecessary overhead. Group the tests that share the same `getIt` setup into a single `group` with one `setUp`/`tearDown`.

- [ ] **[LOW]** `project_providers_test.dart` вЂ” The filter tests (lines 599вЂ“940) each construct a `ProviderContainer`, add a `subscription`, and wait for `projectsOverviewProvider.future`. Eight such tests Г— container + listen + await is cheap in isolation but adds up. A shared container per filter-test group with a `setUp` that pre-loads the snapshot would reduce the per-test overhead.
