## Generative (Glados) testing opportunities

- [ ] **[MED]** `lib/features/ratings/ui/rating_utils.dart` — `findOptionLabel` already has two Glados properties in `rating_utils_test.dart`. The fallback chain in `_resolveInputType` and `_resolveLabel` in `rating_summary.dart` involves a three-way precedence rule (stored → catalog → key). A Glados property over `(String? storedQuestion, bool inCatalog)` combinations would confirm the priority chain holds across all possible inputs without the current 7 manually written permutation tests. **Genuine candidate.**
- [ ] **[LOW]** `RatingSegmentedInput._selectedIndex` is a pure function over `(List<double> segmentValues, double? value)` with a tolerance rule identical to `findOptionLabel`. A separate Glados test verifying the round-trip `_selectedIndex(segments[i].value) == i` for all valid `i` would give confidence it stays in sync if the tolerance is ever adjusted. This is a narrow but pure-logic target.

## Summary

- 2 oversized test files: `rating_repository_test.dart` (833 lines) and `session_rating_modal_test.dart` (737 lines); both inflate due to repeated mock-stub blocks that should be extracted into shared helpers.
- 0 oversized impl files (all under 500 lines).
- 2 weak/smoke tests: `'renders drag handle'` in `session_rating_modal_test.dart:172` asserts only that the modal type exists; the `rating_catalogs_test.dart` locale loop covers only 2 of 6 locales.
- 2 Glados candidates: `_resolveLabel`/`_resolveInputType` priority chain in `rating_summary.dart`, and `RatingSegmentedInput._selectedIndex` round-trip.
- 3 coverage gaps: non-default `catalogId` in `createOrUpdateRating`, `_updateRating` catalogId preservation, and the empty-catalog `_canSubmit` edge-case.
- Biggest opportunity: replace `pumpAndSettle` with `pump()` across `rating_summary_test.dart` (31 calls) and `rating_input_widgets_test.dart` (11 calls) — no animations exist on these widgets, and the current pattern risks hang if an animation is accidentally introduced.
