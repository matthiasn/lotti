## Test execution speed opportunities

- [x] **[HIGH]** `attachment_ingestor_test.dart:920, 972, 1025, 1098` — **RESOLVED:** the four 5-second caps lowered to 2 seconds (the timeout is a hang-failure bound, not a wait — it costs nothing on the happy path).
  `await ingestor.whenIdle().timeout(const Duration(seconds: 5))` uses **real wall
  clock time** (5-second OS timeout). The fake-time policy requires avoiding real
  `Duration` waits. These tests exercise the async download queue which uses real
  `dart:io` file I/O, so a complete fake-time rewrite is not straightforward;
  however, the 5-second hard cap can be lowered to 2 seconds for the cases where
  the download is expected to complete quickly. The `_runDownload` re-queue test
  (line 1098) is the heaviest because it chains two real file writes behind a
  `Completer`. Estimated impact: ~20 seconds of potential wall-clock headroom per
  CI run if these tests hit the timeout under load.

- [ ] **[MED]** `catch_up_strategy_test.dart:2402` — `await Future<void>.delayed(const Duration(milliseconds: 40))`
  inside `requestFuture` mock is used to make the `overallTimeout` test
  work (the real `collectForwardForBootstrap` calls `DateTime.now()` to check elapsed
  time, which cannot be faked). This is a genuine real-I/O boundary where the
  production function uses a wall-clock timeout. The simplest fix is to inject a
  `DateTime Function()` clock into `collectForwardForBootstrap` (matching the pattern
  already used in `AgentVcDominanceCheck`) so the test can use `fakeAsync` without
  the real-time sleep.

- [ ] **[MED]** `attachment_ingestor_test.dart:720, 856, 862` —
  `await ingestor.whenIdle().timeout(const Duration(seconds: 1))`. These lifecycle
  tests (`whenIdle resolves immediately`, `dispose completes waiter`, `dispose safe
  multiple times`) should complete synchronously or in one microtask pump. The 1-second
  hard timeout is the only guard — if `whenIdle()` never resolves these tests silently
  hang for 1 second instead of failing fast. Replace with
  `expectLater(ingestor.whenIdle(), completes)` or add an explicit assertion that
  the future is already complete before awaiting.

- [ ] **[LOW]** Glados tests in `catch_up_strategy_test.dart` (three properties) run
  at default `numRuns` (no `ExploreConfig` specified), which defaults to 100 runs
  each. Given the scenario combinatorial space is already well-covered by the static
  tests in the same file, consider adding `ExploreConfig(numRuns: 80)` to cap the
  property runs. Impact: modest CPU saving on each CI Glados shard run.
