## Test execution speed opportunities

- [x] **[MED]** `attachment_decoding_test.dart` — the "large gzipped payload" test
  (line 128) generates a 32 KB random byte slice, compresses it with `gzip.encode`,
  and then decompresses it via `compute` (a real isolate spawn). Spawning an isolate
  in a unit test is slow (typically 100–400 ms on CI). This is a real I/O boundary
  (isolate spawn is not fakeable) so the test is justified, but it should carry a
  comment explaining why `fakeAsync` is inapplicable here, and the random seed should
  remain fixed (it already is: `Random(0x5EED)`).
  **RESOLVED:** done — the large-gzip test now documents why fakeAsync is inapplicable (compute() spawns a real isolate that a fake clock cannot drive); the seed was already fixed at `Random(0x5EED)`.

- [ ] **[LOW]** `atomic_write_test.dart` (707 lines) creates a real `tempDir` per
  test via `setUp` and cleans it up in `tearDown`. Because `Directory.systemTemp.createTempSync`
  is synchronous the cost is a real OS syscall per test. With 13+ test cases this is
  a modest but real overhead. The cleanup `tempDir.deleteSync(recursive: true)` is
  also a real recursive directory delete. Given the impl requires real I/O this is
  unavoidable, but tests that only exercise error branches (e.g. rename-always-fails)
  do not need a real temp file on disk — they could use an in-memory `IOOverrides`
  scope throughout to avoid the disk round-trips.
  **DEFERRED:** A fully in-memory `IOOverrides` scope would have to back the whole
  `atomicWriteBytes` path (`tmpFile.parent.create`, `writeAsBytes(flush:)`,
  `File.exists`, `rename`, `delete`, plus `pid`/`DateTime.now`-derived tmp/bak
  names). No in-memory-filesystem package is on the dependency list (only
  `cross_file`), so this means hand-rolling a complete fake `Directory`/`File`/
  `IOOverrides` — far larger and more fragile than the per-branch rename fakes
  already in `atomic_write_test_helpers.dart`. A subtly wrong fake could silently
  pass, hang, or mask a real regression, and per the task's HARD RULES I cannot run
  the suite to validate it. Deferring is safer than shipping an unverified fake FS;
  the per-test temp-dir syscall is a known, modest cost the item itself calls
  "unavoidable" given the real-I/O impl.

- [x] **[LOW]** `timeline_utils_test.dart` — the Glados backoff property
  (`numRuns` not specified, defaults to 100) and the timeline-ids property (also
  default 100) both run at the default rate. The functions are trivially fast so
  impact is negligible; no change needed.
  **RESOLVED:** verified, no change — both `glados.Glados(...)` invocations omit
  `ExploreConfig`, so they run at the default rate. The covered functions (pure
  list scans / arithmetic clamp) are trivially fast; tuning `numRuns` would add
  churn with no measurable benefit.
