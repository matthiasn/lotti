## Generative (Glados) testing opportunities

- [x] **[HIGH]** `inbound_event_queue.dart` — `waitForDrainAtMostTo` completion semantics under arbitrary interleaving of `enqueueLive`, `commitApplied`, and `markSkipped` calls. The invariant: if a `waitForDrainAtMostTo(n)` call is pending and a sequence of commits/skips causes `stats().total` to reach `n`, the future must complete. A Glados property could generate arbitrary multi-step sequences of enqueue/commit/skip operations and assert the waiter always completes once the depth condition is met and never completes before it is met. This is non-trivial pure logic with a real ordering invariant.

- [x] **[HIGH]** `inbound_event_queue.dart` — `_emitDepth` coalescing logic. The `_inBatchMode` / `_batchDirty` state machine that suppresses in-transaction depth emissions and fires exactly one post-commit emission has no dedicated property test. The invariant: across any sequence of `runInTransaction` calls (including nested ones) and concurrent `_scheduleDepthEmit` triggers, the `depthChanges` stream must emit at least once after each outermost transaction completes if any mutation occurred. A Glados property over arbitrary nesting depths (1–5) and mutation sequences would catch regression in the dirty-flag / re-run path.

- [ ] **[MED]** `bridge_coordinator.dart` — the incomplete-retry counter state machine. The Glados test at `bridge_coordinator_test.dart:523` covers single-flight coalescing but not the `_consecutiveIncomplete` / `_maxIncompleteRetries` state machine. An invariant: after `_maxIncompleteRetries` consecutive incomplete walks, `_consecutiveIncomplete` is reset to 0 and no further timer is scheduled, regardless of the interleaving of `bridgeNow` calls, limited-sync triggers, and stop/start cycles. This is pure state-machine logic with a clear counting invariant.

- [ ] **[MED]** `queue_pipeline_coordinator.dart` — `_runBootstrap` dispatch invariant. The forward/backward selection logic at `_runBootstrap` (line 944) must obey: if `marker.lastAppliedEventId != null`, the forward walk always runs first; the backward walk runs only if the forward walk returns `errorNoProgress`. A Glados property over `(_GeneratedBridgeAnchorKind, _GeneratedBootstrapOutcome)` pairs would exhaust the dispatch matrix. The `runBootstrapForTest` entry point already exists, making this straightforward to wire.

- [ ] **[LOW]** `inbound_event_queue.dart` — `resurrectByPaths` chunking invariant at `_resurrectByPathsChunkSize`. The property: for any path list of length 1–2 000, the total rows resurrected equals the sum of individual `resurrectByPath` calls, and the number of DB round-trips equals `ceil(uniquePaths / 900)`. The chunk-boundary arithmetic is pure and the existing `_withFreshInboundQueue` helper already supports multi-event setups. This is a targeted correctness check for the chunking logic (currently only tested with fixed sizes at `inbound_event_queue_test.dart:1629`).

## Coverage / missing-behavior gaps

- [x] **[HIGH]** `_countSuppressedSelfEcho` suppression-window reset path (`queue_pipeline_coordinator.dart:727–741`) — the branch where `_suppressionLogInterval` has elapsed and the counter resets is entirely untested. Because the method uses `DateTime.now()` (not `clock.now()`), this branch cannot be exercised deterministically today. Fix the production code first (see Test quality item above), then add a fakeAsync test covering: (1) counter accumulates without logging during the interval, (2) first call after the interval logs with the accumulated count and resets.

- [x] **[HIGH]** `_runForwardBootstrap` fallback chain in `queue_pipeline_coordinator.dart` — the path where the forward walk returns `_BootstrapOutcome.errorNoProgress` and execution falls through to `_runBackwardBootstrap` (lines 955–965) is not directly tested. - [x] **[HIGH]** `QueuePipelineCoordinator.stop()` error-unwind path when `start()` throws after `_liveSub` is set but before `_started` is flipped (`queue_pipeline_coordinator.dart:390–432`) — the explicit cleanup of `_attachmentPathFlushInFlight` on the error path (line 410–418) is not covered. A test that injects a throw from `_worker.start()` after `_attachmentIndex` has already emitted a path (so `_attachmentPathFlushInFlight` is non-null) would exercise the shutdown-race guard.

- [ ] **[MED]** `inbound_event_queue.dart:earliestReadyAt` — the `MAX(nextDueAt, leaseUntil)` semantics are tested for the lease case (after `peekBatchReady`) but not for the `retrying` case where `nextDueAt > leaseUntil`. - [ ] **[MED]** `bridge_coordinator.dart:_handleIncompleteFollowUp` — the `_maxIncompleteRetries` give-up path (lines 318–326) logs `queue.bridge.incomplete.giveUp` and resets the counter. The bridge test covers `_consecutiveIncomplete` up to the retry limit but never past it. ## Test execution speed opportunities

- [x] **[HIGH]** `queue_pipeline_coordinator_test.dart` — the Glados test `generated drainUntilEmpty follows queue, pen, ready-at and timeout model` (line 1 459, `numRuns: 120`) uses real `Future<void>.delayed` inside `drainUntilEmpty` for the retry-sleep path (`queue_pipeline_coordinator.dart:519`). Even though the per-run delay is short (`Duration(milliseconds: 20)` timeout in the timeout scenario), 120 runs × real async yields accumulate. The test already wraps the body in `fakeAsync` + `withClock`, but `drainUntilEmpty`'s internal `Future<void>.delayed(capped)` at line 519 is a real timer that `fakeAsync` controls correctly only when `async.elapse(…)` is called. Verify the fakeAsync harness is actually advancing the clock for the sleep (it appears to be via `async.elapse(scenario.advanceBy)`), then confirm the `_GeneratedDrainReadyAtKind.soon` branch (15 ms delay) resolves deterministically under fake time. **Impact: ~30 s per full Glados run if any real waits slip through.**

- [x] **[HIGH]** `queue_pipeline_integration_test.dart` — `_waitForQueueStats` (lines 73–101) has no timeout. If `depthChanges` stops emitting (e.g., due to a regression in `_emitDepth`), the four integration tests will hang indefinitely in CI rather than failing fast. Add a `timeout: const Duration(seconds: 5)` to the `completer.future` await. **Impact: prevents silent CI hangs that exhaust the job timeout.**

- [ ] **[MED]** `queue_pipeline_coordinator_test.dart:632` — Glados `generated live ingress filters room/status/self-echo` (`numRuns: 120`) creates a fresh `SyncDatabase` + `JournalDb` (two SQLite in-memory databases) for every generated run and closes them in a `finally` block. Creating/closing Drift in-memory DBs has measurable overhead (~5–10 ms each). At 120 runs that is ~1 200–2 400 ms of pure DB construction with no actual DB writes. The test mocks `localQueue`, so the databases are never used — the `localSyncDb` and `localJournalDb` are needed only because `QueuePipelineCoordinator` takes them as required constructor args. Introducing a shared no-op placeholder for these two args (or a static in-memory DB opened once per Glados property) would cut this overhead to near zero. **Impact: ~1–2 s per Glados run.**

- [ ] **[MED]** `queue_pipeline_coordinator_test.dart:1464` — same pattern as above: the drain scenario Glados test creates fresh databases per run. Same fix applies. **Impact: ~1–2 s per Glados run.**

- [ ] **[MED]** `bridge_coordinator_test.dart` — `_drainBridgeCoordinatorMicrotasks()` (lines 197–201) uses a `for (var i = 0; i < 6; i++) await Future<void>.value()` loop in place of `pumpEventQueue()`. `pumpEventQueue()` is idiomatic and avoids a magic constant that must match the coordinator's async depth. Replace all call sites with `await pumpEventQueue()`. **Impact: minor readability win and removes a brittle constant that will break silently if the coordinator adds an async hop.**

- [ ] **[LOW]** `inbound_event_queue_test.dart:934` — the batch-accounting Glados test (`numRuns: 140`) seeds the DB with up to 14 events per run. The 140 × 14 × SQLite inserts are legitimate since this test exercises real DB behavior, but the `maxBatch: scenario.rows.length` `peekBatchReady` call (line 826 in the marker-clamp property, `numRuns` not specified = default) re-reads every row on every run. Consider capping at `numRuns: 100` — 140 is above the "ask before going above 200" guideline and saves ~30% CPU on a saturated CI runner without materially reducing coverage. **Impact: ~15 s off the Glados CI lane.**

## Summary

- **2 oversized implementation files** exceed the 1 000-line threshold: `queue_pipeline_coordinator.dart` (1 295) and `inbound_event_queue.dart` (1 164). Both have clear, concrete split seams documented above.
- **2 oversized test files**: `queue_pipeline_coordinator_test.dart` (3 874 lines) and `inbound_event_queue_test.dart` (1 748 lines); both will resolve naturally once the impl splits.
- **6 notable test-quality issues**: inline mock duplication in `queue_pipeline_integration_test.dart`, duplicated Glados harness setup in the coordinator test (~140 lines of waste), `DateTime.now()` in `_countSuppressedSelfEcho` blocking deterministic testing, fragile `Future.delayed(Duration.zero)` yields, near-zero-value smoke test, and a duplicated `_buildSyncEvent` helper across 3 files.
- **5 Glados candidates** identified: `waitForDrainAtMostTo` ordering invariant, `_emitDepth` coalescing invariant, bridge incomplete-retry state machine, `_runBootstrap` dispatch invariant, and `resurrectByPaths` chunking correctness. All are pure logic with clear algebraic invariants over non-trivial input sequences.
- **5 execution-speed wins**: hanging `_waitForQueueStats` in integration tests, DB-creation overhead in two Glados Coordinator tests (~1–2 s each per run), over-specified `numRuns: 140` in the batch-accounting Glados test, and the magic-constant `_drainBridgeCoordinatorMicrotasks` loop.
- **Biggest single opportunity**: splitting `queue_pipeline_coordinator.dart` into 3 focused files (lifecycle, live-ingress, gap-recovery) would halve the file size, eliminate the two duplicated Glados harnesses, and make the `DateTime.now()` / coverage gaps trivially easy to address in their natural homes.
