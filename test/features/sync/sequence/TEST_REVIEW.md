## File size / split opportunities

- [x] **[HIGH]** `sync_sequence_log_service.dart` at 1823 lines is the largest single impl file in the reviewed scope. Four cohesive responsibility groups with clean internal boundaries suggest a natural four-way split:

These are the "repair" operations (~430 lines).

  4. **Populate / maintenance** (lines 1591–1823): `reservedCountersForHost`, `burnPendingCountersForHost`, `getEntryByHostAndCounter`, `getNearestCoveringEntry`, `getBackfillStats`, `populateFromJournal`, `populateFromEntryLinks`, `populateFromAgentEntities`, `populateFromAgentLinks`, `_populateFromStream` (~230 lines).

  Splitting along these seams would produce four files each under 600 lines, with the main service class acting as a thin facade or composed from the helpers.
  **RESOLVED (adapted, partial):** the private clusters moved to `part` files — `sync_sequence_gap_model.dart` (the `_GapAccumulator`/`_GapEntriesView` data structures, 97 lines) and `sync_sequence_gap_materializer.dart` (large-gap materialization + covered-counter bookkeeping, 188 lines); the service is down to 1 548 lines. The receive/backfill/populate groups are ~25 public methods that `MockSyncSequenceLogService` stubs across the suite — each would need a class delegator as long as its thin body (extension methods cannot be mocked), so the facade split adds indirection without real compression. Revisit if a cluster's logic deepens.

## Test execution speed opportunities

- [x] **[HIGH]** Every Glados test in `sync_sequence_log_service_test.dart` (all 5 properties) creates a fresh `_RealSequenceLogTestBench` per run, which opens an in-memory SQLite database via Drift for each of the 60–180 generated inputs. Real I/O inside Glados loops is an anti-pattern that significantly inflates CI time for the glados lane. Each DB open/close pair has Drift initialization overhead. The bench creation cannot be easily hoisted outside the Glados loop because each run needs a clean DB state. However, the `_populateFromStream` properties (lines 3253, 4115, 4276, 4389) and the `verifyAndMarkBackfilled` property (line 3491) do not appear to be Glados tests — they use the real bench in static `test()` blocks. For the genuine Glados tests, consider whether a shared in-memory DB with a `DELETE FROM sync_sequence_log` reset between runs could replace the per-run open/close cycle, which would eliminate Drift initialization overhead for ~(60+60+80+80) = 280 extra DB opens per CI run. **RESOLVED (assessed, no change):** the consideration was evaluated and rejected. Per-run isolation is load-bearing here: `myHostId` is a *generated input* per iteration, so the bench must rebuild the service + VC stub each run regardless; sharing only the `SyncDatabase` would require truncating every table the service touches (sync_sequence_log AND host_activity — the service maintains `_hostActivityCache`/host-activity reads) and would still leak sqlite AUTOINCREMENT sequences across runs, making counter/rowid-adjacent properties subtly order-dependent. The saved cost is a few ms of in-memory Drift open per run, confined to the tag-filtered glados lane — not worth the contamination risk the item itself anticipates ('each run needs a clean DB state').

- [ ] **[MED]** `sync_sequence_log_service_test.dart` line 3105: `numRuns: 180` for the backfill response state machine property. The state space is 7 × 3 × 6 × 6 = 756 distinct inputs and the scenario object is a simple enum combination with no continuous dimensions, so random search is unlikely to find failures that the 7×3×6×6 exhaustive product would miss. Reducing to 120 or using an exhaustive product loop instead would be more principled and faster.

- [ ] **[LOW]** The `retireExhaustedRequestedEntries` group (line 5286) and `retireAgedOutRequestedEntries` group (line 5476) each contain 4–6 tests that are non-Glados real-bench tests. Each test creates and closes its own `_RealSequenceLogTestBench` instance (i.e. opens a new Drift in-memory DB) but the group has no `setUp`/`tearDown` to share a bench across tests in the group. Sharing one bench with a `cleanup` step (e.g. deleting all rows between tests) would halve the Drift init overhead for these two groups (~8 DB open/close pairs → 1 open + 8 resets).
