## Recurring patterns

- **Inline `getIt` boilerplate** bypassing `setUpTestGetIt()` / `tearDownTestGetIt()` is the single most common anti-pattern across this feature — found in state, ui root, checklists, filtering, labels, and header sub-directories.
- **`pumpAndSettle` overuse**: 313+ calls in the `ui/` group alone; most fire on widgets with no pending animations (static `ConsumerWidget` or synchronous provider overrides) and risk 10-second hangs.
- **One-test-file-per-source rule violations**: multiple test files for `task_filter_modal.dart`, `title_text_field.dart`, and orphan timer-text test files — a pattern that also causes duplicate `setUp` boilerplate.
- **Inline mock class definitions**: `MockTask`, `MockNavService`, `MockEntitiesCacheService`, `MockTimeService`, `_MockSelectable` defined repeatedly in test files instead of centralizing in `test/mocks/mocks.dart`.
- **Missing test files for small but non-trivial production files**: pure-function utilities (`utils.dart`, `task_navigation.dart`, `label_ui_utils.dart`) and small widgets with branching logic (`time_recording_icon.dart`, `task_detail_back_leading.dart`, `task_app_bar_controller.dart`) consistently lack any test coverage.
- **Oversized test files mirroring oversized impl files**: the 3-class `checklist_controller.dart` (530 ln) produces a 1 866-line test; the 8-class `desktop_task_header.dart` (1 127 ln) produces a 1 246-line test. Splitting the impl along class boundaries naturally resolves both.
- **Weak/smoke assertions**: `findsOneWidget` alone, `tester.takeException() == null`, and `expect(true, isTrue)` appearing in tests that claim to test meaningful behavior — concentrated in `task_focus_controller_test.dart`, `tasks_root_page_test.dart`, and checklist widget tests.
