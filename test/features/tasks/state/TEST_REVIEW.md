## Generative (Glados) testing opportunities

- [ ] **[MED]** `ChecklistController._insertItemAt` (impl line 218–238) is a pure function with structured inputs: an existing string list, an item id, an optional `targetIndex`, and an optional `targetItemId`. Its invariants are amenable to property testing:
  - The output list always has exactly `max(existing.length, existing.length + 1)` items (never grows by more than 1 when itemId is already present, always grows by 1 when it is not).
  - The result always contains `itemId` exactly once (idempotency — inserting an item already in the list moves it, not duplicates it).
  - When `targetIndex` is provided and `itemId` is not already present, `result.indexOf(itemId) == targetIndex.clamp(0, existing.length)`.
  This logic currently has only 4–5 hand-crafted concrete tests; a Glados property with a generator over `(List<String>, String, int?, String?)` would cover the boundary arithmetic far more thoroughly.

- [ ] **[MED]** `SavedTaskFilter` JSON round-trip (test file `saved_task_filter_test.dart`): two near-identical concrete tests at lines 7–44 both encode then decode a `SavedTaskFilter`. A Glados property over generated `TasksFilter` values (all its enum and `Set<String>` fields) would cover the full encode/decode surface with a single `round-trips through fromJson/toJson` property. The `TasksFilter` type already has a limited set of enum values (`TaskSortOption`, `AgentAssignmentFilter`) that can be enumerated with `AnyUtils(any).choose(...)`.

- [ ] **[LOW]** `ChecklistCompletionController._computeState` (impl lines 446–480) is a pure aggregation function (count checked vs. total active items from a list of `ChecklistItem`s). A Glados property could verify:
  - `completedCount <= totalCount` for any combination of items.
  - `totalCount == 0` when all items are archived or deleted.
  - `completedCount / totalCount` stays in `[0.0, 1.0]` (as consumed by `ChecklistCompletionRateController`).
  The logic is currently tested with 3 hand-crafted fixtures. This is lower priority because the logic is short and the existing tests cover the main branches.

## Test execution speed opportunities

- [x] **[HIGH]** `test/…/checklist_controller_test.dart` lines 910–911 use two consecutive `await Future<void>.delayed(Duration.zero)` to let the stream listener's `async` callback complete. The Fake Time Policy (`test/README.md`) prohibits `Future.delayed` in tests; replace with `await pumpEventQueue()` (as already used correctly in `checklist_item_controller_test.dart`) or `async.flushMicrotasks()` inside a `fakeAsync` block. Same pattern at line 952. Estimated impact: minor wall-clock reduction per run, but these are policy violations that can mask flakiness.

- [x] **[HIGH]** `test/…/task_progress_controller_test.dart` lines 213 and 418 use `.timeout(const Duration(seconds: 1))` on `Completer.future` — real clock waits that can make the test hang for a full second on CI if the state never transitions. Replace with a `fakeAsync`-based approach: drive the stream event synchronously and use `async.flushMicrotasks()` rather than waiting on a real `Completer`. Both tests are structurally ready for this: they build the stream controller outside `fakeAsync` and could be refactored to use `fakeAsync`.

- [ ] **[MED]** `test/…/saved_filters/saved_task_filter_count_provider_test.dart` lines 147 and 168 (`controller.add(...); await Future<void>.delayed(Duration.zero);`) use real-time async to let the `updateStream` listener invalidate the provider. Replace with `fakeAsync` + `async.flushMicrotasks()` as the `saved_task_filters_persistence_test.dart` in the same directory already does.

- [ ] **[MED]** `test/…/checklist_controller_test.dart` and `test/…/checklist_item_controller_test.dart` both call `getIt.isRegistered` / `getIt.unregister` / `getIt.registerSingleton` in every `setUp`/`tearDown`. This repeated GetIt surgery is slower and more fragile than using the centralized `setUpTestGetIt()` helper (which calls `getIt.reset()` internally). Fixing the HIGH item in "Test quality improvements" above also resolves this speed issue.

- [ ] **[LOW]** `test/…/task_progress_controller_test.dart` line 94 calls `getIt.reset()` directly in `setUp`, then manually registers two singletons. Using `setUpTestGetIt()` would achieve the same result with less boilerplate and avoid accidentally leaving un-reset state if a test throws before teardown.

## Summary

- 1 impl file (`task_app_bar_controller.dart`) has no test file at all.
- 1 test file is severely oversized (`checklist_controller_test.dart` at 1866 lines, 3.5× the 500-line soft limit); it maps to an impl with 3 natural class boundaries that should each have their own file.
- 1 additional test file is notably oversized (`checklist_item_controller_test.dart` at 845 lines).
- 3 test files contain inline getIt registration boilerplate instead of the mandated `setUpTestGetIt()` pattern (`checklist_controller_test.dart`, `checklist_item_controller_test.dart`, `task_progress_controller_test.dart`).
- 3 speed / policy violations exist: `Future.delayed(Duration.zero)` in two files and real `.timeout(Duration(seconds:1))` waits in `task_progress_controller_test.dart`.
- 2 genuine Glados candidates exist for pure logic with algebraic invariants: `_insertItemAt` (list manipulation with move-not-duplicate semantics) and `SavedTaskFilter` JSON round-trips across all enum/set field combinations.
- Biggest single opportunity: split `checklist_controller_test.dart` (1866 lines) along the 3-class boundary and migrate all three resulting files to `setUpTestGetIt()` — this would simultaneously fix the oversized-file, inline-boilerplate, and `Future.delayed` findings in one PR.
