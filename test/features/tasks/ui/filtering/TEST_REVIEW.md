## Test execution speed opportunities

- [ ] **[HIGH]** `task_category_filter_test.dart` — 18 `pumpAndSettle` calls, many of which fire immediately after simple state changes that do not involve animations. Most can be replaced with `tester.pump()` or `tester.pump(const Duration(milliseconds: N))`. Each unnecessary `pumpAndSettle` carries a 10-second hang risk and adds ~50-200 ms in CI.

- [ ] **[HIGH]** `task_filter_modal_test.dart` — 50+ `pumpAndSettle` calls, largely justified by the Wolt bottom-sheet animation (which genuinely needs settling). However, the `buildSubject()` chain duplicates full provider boilerplate three times; shared setup would reduce tree-rebuild overhead across tests.

- [ ] **[MED]** `task_filter_icon_test.dart` — every test opens the modal with `pumpAndSettle` then immediately pumps again. Modal open/close patterns could use bounded `tester.pump(const Duration(milliseconds: 400))` if the animation is bounded, instead of open-ended `pumpAndSettle`.

- [ ] **[LOW]** `task_project_selection_modal_test.dart:8` — 8 `pumpAndSettle` calls; acceptable given the modal animation, but consider bounded pumps for determinism.
