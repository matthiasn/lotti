## Test quality improvements

- [x] **[HIGH]** `theming_controller_test.dart:382–401` (`handles stream errors`) — **Real async timeout violation.** Uses a real `Completer<void>` with `.timeout(const Duration(milliseconds: 100))`. This waits for real wall-clock time. Rewrite using `fakeAsync` and a synchronous stream error emission:
  ```dart
  fakeAsync((async) {
    container.listen(enableTooltipsProvider, (_, _) {});
    tooltipController.addError(error);
    async.flushMicrotasks();
    final state = container.read(enableTooltipsProvider);
    expect(state.hasError, isTrue);
  });
  ```
- [ ] **[MED]** `theming_controller_test.dart:53–108` — **Inline GetIt setup / teardown.** The `setUp` manually calls `GetIt.I.allowReassignment = true` and registers five singletons; `tearDown` calls `GetIt.I.reset()`. The project provides `setUpTestGetIt()` / `tearDownTestGetIt()` (from `test/widget_test_utils.dart`) for this purpose. Migrate to those helpers, passing the extra service registrations as additional setup after calling `setUpTestGetIt()`.
- [ ] **[MED]** `theming_controller_test.dart:121–128` (`initial state has default Grey Law theme`) — includes `expect(state.darkTheme, isNotNull)` and `expect(state.lightTheme, isNotNull)`. These bare existence checks add no coverage beyond what the non-null typed field guarantees. Replace with structural assertions, e.g. `expect(state.darkTheme!.brightness, Brightness.dark)` and `expect(state.lightTheme!.brightness, Brightness.light)`.
- [ ] **[MED]** `theming_controller_test.dart:405–454` (`gracefully handles OutboxService not registered`) — the test body comment says "Test passes if no exception is thrown" and makes no `expect` call. This is a pure smoke test. Add at least one assertion, e.g. that the state's `lightThemeName` is unchanged after the call, or that `saveSettingsItem` was still called.
- [ ] **[LOW]** `theme_definitions_test.dart:44–48` (`has non-null FlexScheme value`) — `expect(themes[defaultThemeName], isNotNull)` is a bare existence check. The surrounding group already checks all themes for `isA<FlexScheme>()`, making this assertion redundant. Remove or replace with `expect(themes[defaultThemeName], FlexScheme.greyLaw)` to pin the specific value.
- [ ] **[LOW]** `theme_definitions_test.dart:51–60` (`contains expected number of standard themes`) — hardcodes `equals(21)`. This will silently fail when a new theme is added. Replace with a size check that auto-derives from the map, or add a comment noting that this test is intentionally a pinned count and must be updated when adding themes.

## Coverage / missing-behavior gaps

- [ ] **[MED]** `ThemingController.setDarkTheme` is tested, but there is no test that sets both light and dark themes and then verifies that `_buildTheme` is called with the correct `isDark` flag for each. The test only checks the name stored in state, not whether `FlexThemeData.dark` vs `FlexThemeData.light` was actually used.
- [ ] **[MED]** `_getEmojiFontFallback()` returns `null` on web (`kIsWeb`) and the fallback list otherwise. There is no test exercising the web branch. Since `kIsWeb` is compile-time, this is hard to test without a platform stub; document as a known untestable branch.
- [ ] **[LOW]** `ThemingState.copyWith` with all-null arguments is tested (`copyWith preserves all values`), but `copyWith` where only `themeMode` is updated (and themes are null in original state) is not. Low risk given the simple null-coalescing pattern.
- [ ] **[LOW]** `onThemeSelectionChanged` with an empty set (`modes.first` would throw). This is a potential crash path if the `SegmentedButton` ever emits an empty set. There is no test or guard for `modes.isEmpty`. Consider adding a defensive check in the impl and a test for the empty-set path.

## Test execution speed opportunities

- [x] **[HIGH]** `theming_controller_test.dart:382–401` — real 100 ms timeout via `completer.future.timeout(...)`. Even though 100 ms is short, it still involves real wall-clock time on every CI run. Switching to `fakeAsync` removes the real-time wait entirely. **Estimated impact:** −100 ms per run (low absolute impact but violates the fake-time policy).
- [ ] **[MED]** Multiple tests call `async.elapse(const Duration(milliseconds: 100))` to wait for the initial `_loadSelectedSchemes` async call. This is correct under fakeAsync but adds virtual-time overhead. Consider extracting a `_waitForInit(async)` helper to centralize the `elapse(100ms) + flushMicrotasks()` pattern used across ~10 tests.
- [ ] **[LOW]** All other timer and debounce tests correctly use `fakeAsync` with appropriate `elapse` + `flushMicrotasks` sequences. No real-timer or `Future.delayed` usage found beyond the one Completer case above.
