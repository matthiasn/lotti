## Test quality improvements

- [ ] **[MED]** `user_activity_gate_test.dart` — the three existing tests are behavioral and use `fakeAsync` correctly. However, all three tests call `gate.dispose()` and `service.dispose()` at the end **inside** `fakeAsync`, which means the async `Future`-returning `dispose()` calls are not awaited (fakeAsync does not support `await` at the top level). Change to:
  ```dart
  addTearDown(() async {
    await gate.dispose();
    await service.dispose();
  });
  ```
  or call dispose after exiting the `fakeAsync` block. Without `await`, any pending stream close operations may leak into subsequent tests.
- [ ] **[LOW]** `user_activity_gate_test.dart:19` — `expect(gate.canProcess, isTrue)` alone is a reasonable assertion but the test never verifies the initial stream event. After construction, `canProcessStream` emits one value immediately; this is not tested.

## Coverage / missing-behavior gaps

- [ ] **[HIGH]** `lib/features/user_activity/state/user_activity_service.dart` has **no dedicated test file**. The file contains `updateActivity()`, `lastActivity`, `activityStream`, and `dispose()`. While the service is used as a test fixture in other feature tests (e.g., sync), there is no `test/features/user_activity/user_activity_service_test.dart`. Add one covering:
  - Initial `lastActivity` is `DateTime.fromMillisecondsSinceEpoch(0)`.
  - `updateActivity()` sets `lastActivity` to approximately `DateTime.now()` and emits on `activityStream`.
  - `dispose()` closes the stream (subsequent `updateActivity()` calls are silently ignored, not thrown).
  - `activityStream` broadcasts to multiple subscribers.
- [ ] **[MED]** `UserActivityGate.canProcessStream` applies `.distinct()`. There is no test verifying that rapid identical-state transitions do not produce duplicate stream events. Add a test that calls `service.updateActivity()` twice rapidly and asserts the stream emits only one `false` event per state change.
- [ ] **[MED]** `UserActivityGate._scheduleTimer` has a zero/negative duration branch (`if (duration.isNegative || duration == Duration.zero)`) that immediately calls `_setCanProcess(true)`. This path is triggered when `idleThreshold - elapsed <= 0` at construction time. None of the three tests explicitly exercise the zero-duration branch (though "already idle at construction" comes close). Add a test where `lastActivity` is set far enough in the past that the remaining duration is exactly zero.
- [ ] **[MED]** `UserActivityGate` disposal behavior is not tested: after `dispose()` is called, any subsequent `service.updateActivity()` call should not cause a "add to closed controller" error. Add a test:
  ```dart
  await gate.dispose();
  expect(() => service.updateActivity(), returnsNormally);
  ```
- [ ] **[LOW]** `UserActivityGate.waitUntilIdle()` is tested for the idle and non-idle cases, but not for the case where the gate is disposed while `waitUntilIdle()` is waiting. This could cause a `StateError` on the stream. The implementation guards with `_controller.isClosed` before adds, but does not close the pending `firstWhere` future. Documenting this as a known limitation, or testing for graceful cancellation, would be valuable.
