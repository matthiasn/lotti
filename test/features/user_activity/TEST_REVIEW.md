## Test quality improvements

- [x] **[MED]** `user_activity_gate_test.dart` — the three existing tests are behavioral and use `fakeAsync` correctly. However, all three tests call `gate.dispose()` and `service.dispose()` at the end **inside** `fakeAsync`, which means the async `Future`-returning `dispose()` calls are not awaited (fakeAsync does not support `await` at the top level). Change to:
  ```dart
  addTearDown(() async {
    await gate.dispose();
    await service.dispose();
  });
  ```
  or call dispose after exiting the `fakeAsync` block. Without `await`, any pending stream close operations may leak into subsequent tests.
  - **RESOLVED:** done — all three tests now kick the async `dispose()` futures off with `unawaited(...)` and drain `flushMicrotasks()` before the fakeAsync zone closes (with a comment on why awaiting isn't possible there).
- [x] **[LOW]** `user_activity_gate_test.dart:19` — `expect(gate.canProcess, isTrue)` alone is a reasonable assertion but the test never verifies the initial stream event. After construction, `canProcessStream` emits one value immediately; this is not tested.
  **RESOLVED:** non-actionable as stated. The constructor's `_controller.add(_canProcess)` runs before any external subscriber exists, and Dart's broadcast `StreamController._sendData` drops events when `_isEmpty` (no listeners) — verified in the SDK source. So the "initial emit" is never delivered to a subscriber attached after construction and cannot be asserted via the stream. The initial state is fully observable via the synchronous `gate.canProcess` getter, which every scenario test already asserts. I prototyped a stream-subscription test and confirmed it cannot observe the initial value, so no test was added.

## Coverage / missing-behavior gaps

- [x] **[HIGH]** `lib/features/user_activity/state/user_activity_service.dart` has **no dedicated test file**. The file contains `updateActivity()`, `lastActivity`, `activityStream`, and `dispose()`. While the service is used as a test fixture in other feature tests (e.g., sync), there is no `test/features/user_activity/user_activity_service_test.dart`. Add one covering: **RESOLVED:** new `state/user_activity_service_test.dart` covers the epoch sentinel initial value, `updateActivity` advancing `lastActivity` (bounded by surrounding `now()` reads) and emitting it, broadcast delivery to two subscribers, and silently-ignored updates after `dispose()`.
  - Initial `lastActivity` is `DateTime.fromMillisecondsSinceEpoch(0)`.
  - `updateActivity()` sets `lastActivity` to approximately `DateTime.now()` and emits on `activityStream`.
  - `dispose()` closes the stream (subsequent `updateActivity()` calls are silently ignored, not thrown).
  - `activityStream` broadcasts to multiple subscribers.
- [x] **[MED]** `UserActivityGate.canProcessStream` applies `.distinct()`. There is no test verifying that rapid identical-state transitions do not produce duplicate stream events. Add a test that calls `service.updateActivity()` twice rapidly and asserts the stream emits only one `false` event per state change.
  - **RESOLVED:** done — added a distinct-stream test: rapid repeated `updateActivity()` calls emit at most one `false`, and going idle emits exactly one `true`.
- [x] **[MED]** `UserActivityGate._scheduleTimer` has a zero/negative duration branch (`if (duration.isNegative || duration == Duration.zero)`) that immediately calls `_setCanProcess(true)`. This path is triggered when `idleThreshold - elapsed <= 0` at construction time. None of the three tests explicitly exercise the zero-duration branch (though "already idle at construction" comes close). Add a test where `lastActivity` is set far enough in the past that the remaining duration is exactly zero.
  - **RESOLVED:** done — added an exactly-elapsed-threshold test for both entry points (constructor and timer path). This required a fake-time fix in the impls: gate and service now read `clock.now()` instead of `DateTime.now()`, so fakeAsync actually controls the idle computation.
- [x] **[MED]** `UserActivityGate` disposal behavior is not tested: after `dispose()` is called, any subsequent `service.updateActivity()` call should not cause a "add to closed controller" error. Add a test:
  ```dart
  await gate.dispose();
  expect(() => service.updateActivity(), returnsNormally);
  ```
  - **RESOLVED:** done — added `'activity after dispose does not throw on the closed controller'`: dispose, then `updateActivity()` plus timer elapse complete without errors (the subscription is cancelled before the controller closes).
- [x] **[LOW]** `UserActivityGate.waitUntilIdle()` is tested for the idle and non-idle cases, but not for the case where the gate is disposed while `waitUntilIdle()` is waiting. This could cause a `StateError` on the stream. The implementation guards with `_controller.isClosed` before adds, but does not close the pending `firstWhere` future. Documenting this as a known limitation, or testing for graceful cancellation, would be valuable.
  **RESOLVED:** added `'disposing while waiting never resolves the wait as idle'`. While non-idle, it starts `waitUntilIdle()` (swallowing any terminal stream error), disposes the gate, then elapses well past the idle threshold and asserts the waiter never completes as idle. This pins the safety-relevant contract — disposal must not make a pending waiter believe the system went idle — without asserting the exact terminal-error type, which is timing-fragile under `fakeAsync` (the `dispose()` chain's `subscription.cancel()` / broadcast-close propagation does not settle deterministically via `flushMicrotasks` alone). The no-graceful-cancel limitation is documented in the test comment.
