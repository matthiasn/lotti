## Test quality improvements

- [ ] **[HIGH]** Large portions of `flatpak_security_test.dart` are documentation disguised as tests. Examples:
  - `'uses Freedesktop runtime 24.08'` (lines 162-175) — only asserts on two `const` strings defined locally inside the test body. No app code is exercised.
  - `'network access is properly declared'` (lines 212-222), `'IPC is shared for display server communication'` (lines 224-233), `'audio access uses PulseAudio socket'` (lines 235-243) — each asserts `expect(true, isTrue)` on a `const bool = true` literal defined in the test body. These have zero value and should be removed.
  - `'principle of least privilege is followed'` (lines 289-319) — iterates a `const` map declared inside the test and validates its own keys. Nothing in the app is exercised.
  - `'validates restricted filesystem access paths'` (lines 47-69) — same pattern; a locally defined `const` map is iterated and validated against itself.
- [x] **[HIGH]** `'portal handle tokens are unique and secure'` (lines 141-158) uses `await Future<void>.delayed(const Duration(milliseconds: 2))` — a real timer delay inside a test. This violates the project fake-time policy. Since `PortalService.createHandleToken` uses `DateTime.now().millisecondsSinceEpoch`, the test relies on wall-clock progress to assert token uniqueness, which is inherently flaky at microsecond resolution. The correct approach: mock the timestamp source or assert on token prefix only, not timestamp suffix difference.
- [ ] **[MED]** `'ensures app data is properly isolated'` (lines 72-84) and `'verifies system directories are not directly accessible'` (lines 86-104) — when outside Flatpak (the common test environment) the bodies are empty (`if (PortalService.isRunningInFlatpak)` gates them). These tests pass vacuously and provide no signal outside the sandbox. Document this limitation or convert to unconditional assertions on the constants being tested.
