## Generative (Glados) testing opportunities

All existing Glados tests are tagged `tags: 'glados'` — full compliance observed.

- [x] **[HIGH]** **`health_import.dart` — `sumNumericHealthValues` and `getDays` have no Glados tests**. Both are pure functions with rich invariants:
  - `sumNumericHealthValues`: for any list of `HealthDataPoint` containing N `NumericHealthValue` entries, the result equals the sum of their `numericValue`. A Glados test with a generated list of mixed `HealthDataPoint` types could check this in O(1) with a reference implementation.
  - `getDays(dateFrom, dateTo)`: for any two dates `d1 ≤ d2`, the result should have `d2.difference(d1).inDays + 1` elements; every element should have midnight time; elements should be monotonically increasing by exactly one day. These are algebraic invariants perfectly suited to Glados.

- [x] **[HIGH]** **`health_import.dart` — type-list membership invariant**: no test verifies that `sleepTypes`, `bpTypes`, `heartRateTypes`, `bodyMeasurementTypes`, and `activityTypes` are pairwise disjoint, or that any `HealthDataType` that appears in a specific list produces the correct duplication behavior in `fetchHealthData`. A Glados generator over `sleepTypes` values could assert the duplication invariant (sleep subtypes get an additional `SLEEP_ASLEEP` entry) without hand-coding each case.

- [ ] **[MED]** **`habits/habit_completion_resolution.dart` — `compareHabitCompletionWriteRecency`**: the comparator has four ordered tie-break levels (`updatedAt`, `createdAt`, `dateTo`, `id`). A Glados test could generate two entries, assert `sign(compare(a,b)) == -sign(compare(b,a))` (antisymmetry), and assert transitivity across triples. The existing test (`habit_completion_resolution_test.dart`) covers only happy-path cases with specific tie-breakers via static examples; it does not verify the transitivity invariant.

- [ ] **[MED]** **`create/create_entry.dart`** — `createTextEntry`, `createTimerEntry` are pure-enough coordinator functions. A Glados test over generated `EntryText` + `categoryId` could assert that the persisted entry always has a non-null `id`, that `createdAt ≤ dateTo`, and that the `linkedId` relationship is written when provided. These are currently tested only with fixed fixtures.

- [ ] **[LOW]** **`health_import.dart` — `addActivityEntries` day-boundary calculation**: the `dayEnd = dayStart.add(Duration(days: 1)).subtract(Duration(milliseconds: 1))` expression is a time boundary that could be fuzz-tested. A Glados test over generated `DateTime` values could assert `dayEnd < dayStart.add(Duration(days: 1))` and `dayEnd ≥ dayStart`, and that when `dayEnd.isAfter(now)`, the cap to `now` never exceeds `now`.

## Test execution speed opportunities

- [x] **[HIGH]** **Glados `numRuns: 180` × 3 in `exif_data_extractor_test.dart`** (540 total Glados runs from one file) and **`numRuns: 180` × 3 in `audio_metadata_extractor_test.dart`** (540 runs): these are parsing-function tests on deterministic, stateless pure functions. For such functions, 100–120 runs is the repo convention and is sufficient to expose regressions. Reducing all three in each file to 100 saves ~480 runs per CI build across just these two files with no meaningful coverage loss.

- [x] **[HIGH]** **`create/create_entry_test.dart` — real in-memory SQLite DB shared across all tests**: a single `setUpAll` spins up a full `JournalDb`, `SettingsDb`, `VectorClockService`, real `MetadataService` etc. for 20+ test cases. Because state accumulates, later tests in the group observe the results of earlier entries. This increases execution time and creates ordering dependencies. Per-test `setUp`/`tearDown` using `setUpTestGetIt()` would give each test a clean slate at a modest DB-creation overhead. **RESOLVED:** the `setUpAll` is converted to a per-test `setUp` using `setUpTestGetIt(additionalSetup: ...)` — fresh in-memory `JournalDb`/`SettingsDb` (closed in tearDown) plus fresh registrations per test, so no state accumulates across tests. All 21 tests pass with full isolation (no ordering dependencies surfaced).

- [ ] **[MED]** **`persistence_logic_test.dart` — shared real SQLite DB across 60+ tests**: the `setUpAll`/`tearDownAll` pattern wrapping the main `Database Tests` group means DB state accumulates across all 60 test cases. Several tests call `upsertEntityDefinition(habitFlossing)` redundantly (lines 580, 1037, 1384). Each redundant upsert is a real I/O write whose only purpose is test isolation workaround. Resetting to in-memory DB per test group would eliminate hidden ordering dependencies.

- [ ] **[MED]** **`entry_creation_service_test.dart` — 12+ sequential `pumpAndSettle()` calls**: each has a 10-second timeout. Replacing them with `pump(const Duration(milliseconds: 100))` or the appropriate fixed interval eliminates the hang risk and speeds up each test by eliminating unnecessary wait cycles.

- [ ] **[LOW]** **`media_parsing_edge_cases_test.dart` — 464-line file that largely duplicates already-running tests**: all test cases in this file are re-executions of logic covered in `media/audio_metadata_extractor_test.dart` and `media/exif_data_extractor_test.dart`. Deleting this file eliminates ~60 duplicate test invocations per CI run with zero coverage regression.

- [ ] **[LOW]** **`media_import_integration_test.dart` — 615-line file that re-tests `AudioMetadataExtractor`**: the bulk of this file retests `selectReader`, `bypassMediaKitInTests`, and duration extraction already covered in `media/audio_metadata_extractor_test.dart`. Consolidating into `media_import_test.dart` and removing duplicates reduces the test count without any coverage loss.
