## Test execution speed opportunities

- [x] **[HIGH]** Both test files use `pumpAndSettle()` heavily (17 calls in `create_measurement_dialog_test.dart`, 11 in `complete_habit_dialog_test.dart`). For dialogs that render synchronously after `pump()`, most of these could be replaced with `tester.pump()` to cut run time and reduce the risk of hanging on an unsettled animation. Use `pumpAndSettle` only where animation completion is load-bearing. **RESOLVED:** all 28 settles (17 + 11) replaced with bounded `pump()` + `pump(300ms)` pairs; the one animation-coupled site (HabitDialog close-button route pop) keeps its unbounded settle, verified by failure when bounded.
- [x] **[LOW]** No `Future.delayed` or real timers found. Hotkey registration in `initState` may not play well with `pumpAndSettle` in the desktop path — worth investigating if tests flake on CI.
  - **RESOLVED:** investigated, confirmed not a flake risk (no change). Both dialogs register `HotKey(keyS, meta, scope: HotKeyScope.inapp)`. In `hotkey_manager-0.2.3`, `HotKeyManager.register`/`unregister` only `await _platform.register/unregister` for `HotKeyScope.system`; for `inapp` they perform pure synchronous in-memory map/list mutations and return an already-completed `Future`. So `initState` registration schedules no platform-channel call, timer, or pending microtask for `pumpAndSettle` to hang on. On Linux/macOS CI `isDesktop` defaults true, so the regular pump helpers already exercise this path every run without flaking, and the desktop-lifecycle test asserts the registration/unregistration round-trip directly. The shared-isolate concern is also handled: `unregister` uses `removeWhere(identifier)` (clears all entries for the Cmd+S identifier) and each subsequent `pumpWidget` disposes the prior `HabitDialog`, so no leaked registration survives into the `isEmpty` assertion.
